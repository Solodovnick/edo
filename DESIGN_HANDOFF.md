# Документация ЭДО Банка — Передача аналитику и дизайнеру

**Версия:** 1.0
**Дата:** 2026-04-30
**Стек:** React + TypeScript + Tailwind CSS v4
**Хранение:** localStorage (mock backend)

---

## 1. Цель системы

Система электронного документооборота (ЭДО) банка для управления обращениями клиентов. Цель — повысить SLA compliance с **94% → 99%** через прозрачный BPMN-процесс, мониторинг очередей и аналитику.

---

## 2. Дизайн-система

### 2.1. Цветовая палитра

| Назначение | HEX | Применение |
|---|---|---|
| Primary (фиолетовый) | `#673AB7` | CTA, активные вкладки, статусы "в работе" |
| Primary hover | `#5E35B1` | Состояние hover для primary |
| Primary light | `#EDE7F6` | Фон выделенных блоков |
| Success | `#4CAF50` | Успешные SLA, "Готово", "Закрыто" |
| Warning | `#FF9800` | Приближение к дедлайну (<24ч) |
| Danger | `#F44336` | Просрочка SLA, ошибки |
| Neutral | `#9E9E9E` | Вспомогательный текст |
| Background | `#FAFAFA` | Фон страниц |
| Surface | `#FFFFFF` | Карточки, модалки |

> Стиль — Google Forms (фиолетовый акцент, минимализм, скруглённые карточки `rounded-lg`, мягкие тени).

### 2.2. Типографика

Базовые стили заданы в `/src/styles/theme.css` (НЕ переопределять inline-классами Tailwind типа `text-2xl`, `font-bold`).

- H1 — заголовки страниц
- H2 — секции
- H3 — подсекции карточек
- body — основной текст
- small — метаданные, таймстампы

### 2.3. Сетка и отступы

- Контейнер: `max-w-7xl mx-auto px-6`
- Карточки: `rounded-lg p-6 shadow-sm`
- Расстояния между блоками: `gap-4` / `gap-6`
- Header: фиксированный, высота 64px

## 3. Архитектура страниц

### 3.1. Глобальные элементы

- **Header** (`/src/app/components/Header.tsx`) — навигация с подсветкой активной вкладки.

### 3.3. Жизненный цикл обращения (`/src/app/components/lifecycle/`)

Унифицированный фиолетовый стиль на всех страницах кабинетов:

1. **RegistrationPageEnhanced.tsx** — регистрация обращения секретарём.
2. **SecretaryOfficePage.tsx** + `SecretaryCabinet.tsx` + `SecretaryCard.tsx` — кабинет секретаря.
3. **ManagerCabinetPageUnified.tsx** + `ManagerCardDetailed.tsx` — кабинет руководителя (распределение).
4. **ProcessingPageUnified.tsx** + `ProcessingCardNew.tsx` — кабинет ответственного (исполнение).
5. **AuditPageUnified.tsx** + `AuditCardDetailed.tsx` — аудит и контроль качества.
6. **ArchivePage.tsx** — архив.
7. **CategoryKnowledgeBase.tsx** — база знаний по категориям.

## 4. BPMN-процесс и обязательные элементы

На **каждой** странице кабинета должны присутствовать:

1. **Видимый статус обращения** — бейдж в шапке карточки.
2. **Блок "Следующее действие"** — что должен сделать пользователь сейчас.
3. **Сайдбар "Протокол"** — хронология событий с фильтрацией по этапам.
4. **Сайдбар "Данные обращения"** — клиент, канал, категория, вложения.
5. **Валидация обязательных действий** — нельзя закрыть/передать без заполнения.

| # | Статус | Кабинет | Цвет |
|---|---|---|---|

| 2 | Зарегистрировано | Секретарь | primary-light |
| 3 | На распределении | Руководитель | primary |
| 4 | Назначено | Ответственный | primary |
| 5 | В работе | Ответственный | primary |
| 6 | Ожидает уточнения | Ответственный | warning |
| 7 | Готово к проверке | Аудит | primary-light |
| 8 | На аудите | Аудит | primary |
| 9 | Возвращено на доработку | Ответственный | danger |
| 10 | Согласовано | Аудит | success |
| 11 | Закрыто | — | success |
| 12 | Архив | — | neutral |

## 6. SLA-сроки

| Тип обращения | Срок |
|---|---|
| Устное | 7 дней |
| Письменное | 15 дней |
| Регуляторное | 3 дня |

**Индикация дедлайна:**
- `> 48ч` — зелёный
- `24–48ч` — жёлтый (warning)
- `< 24ч` — оранжевый
- Просрочка — красный (danger), бейдж "Просрочено"

## 10. Чек-лист для дизайнера

- [ ] Все экраны используют palette из §2.1 без отклонений.
- [ ] Бейджи статусов соответствуют §5.
- [ ] На каждом кабинете присутствуют 5 обязательных блоков (§4).
- [ ] Индикатор SLA отображается в карточке списка и в детальной карточке.
- [ ] Header подсвечивает активную вкладку фиолетовой нижней границей.
- [ ] Адаптив: desktop (≥1280), tablet (768–1279), mobile (<768).

## 11. Чек-лист для аналитика

- [ ] BPMN-схема соответствует 12 статусам (§5).
- [ ] Переходы между статусами валидируются (нет skip-ов).
- [ ] SLA-таймеры стартуют от регистрации, не от назначения.
- [ ] Метрики KPI: % SLA, среднее время обработки, нагрузка по сотрудникам.
- [ ] Roles & permissions матрица: секретарь / руководитель / ответственный / аудитор.
- [ ] Регуляторные обращения помечаются и приоритизируются автоматически.
