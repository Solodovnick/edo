### Часть 1. Сценарии использования (Use Cases) для Регистратора

#### UC-1. Автоматический поиск клиента (Проверка CRM)
*   **Предусловие:** Регистратор открыл форму создания обращения. Выбран тип заявителя "Физлицо".
*   **Действие пользователя:** Вводит номер телефона в поле `Номер телефона` (или ФИО + Дату рождения). Нажимает Enter или уходит с поля (OnBlur).
*   **Действие системы:**
    1.  Отправляет запрос в CRM (Integrations Mock).
    2.  Если клиент найден:
        *   Автозаполняет поля: `ФИО Заявителя`, `Дата рождения`, `Адрес`.
        *   Переключает сегмент `Клиент / Неклиент` в состояние **"Клиент"**.
        *   Проверяет наличие Персонального менеджера. Если есть — отображает поле `ФИО Персонального менеджера` (заполненное) и ставит метку VIP.
    3.  Если не найден: Оставляет поля пустыми для ручного ввода, переключает в **"Неклиент"**.
*   **Постусловие:** Данные заявителя валидированы и заполнены.

#### UC-2. Расчет SLA и Приоритета (Калькулятор сроков)
*   **Предусловие:** Форма открыта. Текущая дата подставлена в поле `Дата регистрации`.
*   **Действие пользователя:** Переключает сегмент `Выберите тип обращения` (Устное / Письменное / Регулятор).
*   **Действие системы:**
    1.  Мгновенно пересчитывает поле `Плановая дата` (SLA):
        *   **Устное:** +7 дней. `Приоритет` -> **Средний**.
        *   **Письменное:** +15 дней. `Приоритет` -> **Низкий**.
        *   **Регулятор:** +3 дня. `Приоритет` -> **Высокий** (блокирует изменение).
    2.  Подсвечивает поле SLA цветом (Зеленый/Желтый/Красный в зависимости от срочности).
*   **Постусловие:** Установлен корректный дедлайн выполнения заявки.

#### UC-3. Классификация через Базу Знаний
*   **Предусловие:** Поле `Тема обращения` заполнено (например, "Не работает карта").
*   **Действие пользователя:** Нажимает кнопку `База Знаний` (или иконку лупы в поле `Категория`).
*   **Действие системы:**
    1.  Открывает модальное окно/дропдаун со списком категорий, релевантных введенной теме.
    2.  Пользователь выбирает категорию (напр., "Эквайринг -> Сбой").
    3.  Система заполняет поле `Категория`.
    4.  Система может *переопределить* Приоритет и SLA, если для данной категории настроены особые критичные сроки (Proto 6).
*   **Постусловие:** Обращение классифицировано по справочнику (единый стандарт).

#### UC-4. Мгновенное решение (First Call Resolution)
*   **Предусловие:** Регистратор в ходе разговора понял, что может помочь сам (консультация).
*   **Действие пользователя:**
    1.  Заполняет поле `Содержание обращения` информацией о консультации.
    2.  Отмечает чек-бокс `☑ Решено на первой линии`.
*   **Действие системы:**
    1.  Скрывает поля маршрутизации (Пул, ПМ).
    2.  Меняет текст на кнопке `Зарегистрировать (ОК)` на **"Зарегистрировать и Закрыть"**.
    3.  При нажатии ОК: Создает обращение сразу в статусе **"Закрыто"**.
*   **Постусловие:** Обращение создано и закрыто, не попадает в пул распределения, но учтено в статистике.

#### UC-5. Валидация и Отправка
*   **Предусловие:** Регистратор заполнил форму.
*   **Действие пользователя:** Нажимает кнопку `Зарегистрировать (ОК)`.
*   **Действие системы:**
    1.  Проверяет заполненность обязательных полей (* отмечены: ФИО, Телефон, Категория, Содержание).
    2.  Если есть ошибки: Подсвечивает поля красным, показывает тост "Заполните обязательные поля".
    3.  Если успешно:
        *   Генерирует номер обращения (напр., "REQ-2026-001").
        *   Показывает всплывающее окно (Overlay/Modal): **"Обращение №REQ-2026-001 создано. Озвучьте номер клиенту!"**.
        *   Очищает форму для нового ввода.
*   **Постусловие:** Новая карточка сохранена в БД (Proto 1).

### Часть 2. Инструкции для Figma (Figma Make Format)

Ниже описана логика компонентов для создания **интерактивного прототипа**.

```yaml
---
# ==========================================
# 🧩 FIGMA COMPONENT SETS & INTERACTIONS
# ==========================================

COMPONENT: Input_Field_Search_CRM
VARIANTS: Default, Focused, Filled, Loading, Success
INTERACTION:
  - On_Click (Default) -> Change to Focused.
  - On_Key_Press (Enter) -> Change to Loading (Delay 800ms) -> Change to Success.
  - Smart_Animate:
      When [Success]:
      Target_Frame: Include "Auto-filled Group" (ФИО, Адрес, Дата рожд).
      Variable: Set Is_Client = True.

COMPONENT: Segmented_Control_Appeal_Type
VARIANTS: Oral_Selected, Written_Selected, Regulator_Selected
PROPERTIES:
  - Oral_Selected:
      Set Variable [SLA_Days] = 7.
      Set Component [Priority_Segment] = "Medium".
  - Written_Selected:
      Set Variable [SLA_Days] = 15.
      Set Component [Priority_Segment] = "Low".
  - Regulator_Selected:
      Set Variable [SLA_Days] = 3.
      Set Component [Priority_Segment] = "High".
      Lock Component [Priority_Segment] (State = Disabled).

COMPONENT: Button_Knowledge_Base
VARIANTS: Default, Hover, Active
INTERACTION:
  - On_Click -> Open Overlay "Modal_Knowledge_Base".
  - Overlay Behavior: Center, Dim Background (30%).
  - Inside Overlay: List Item Click -> Close Overlay -> Set Variable [Category_Text] to Selected Item.

COMPONENT: Checkbox_First_Line_Resolution
VARIANTS: Unchecked, Checked
INTERACTION:
  - On_Click -> Toggle Variant.
  - Logic:
      If Checked -> Change Button_Submit Text to "Зарегистрировать и Закрыть".
      If Unchecked -> Change Button_Submit Text to "Зарегистрировать (ОК)".

COMPONENT: Button_Submit_Main
VARIANTS: Disabled, Enabled, Loading
INTERACTION:
  - Logic (Conditional Visibility):
      If [Phone_Input] is empty OR [Category_Input] is empty -> State = Disabled.
      Else -> State = Enabled.
  - On_Click (Enabled) -> Change to Loading (Delay 1000ms) -> Navigate to "Success_Overlay".

FRAME: Success_Overlay
DESCRIPTION: "Финальное сообщение"
ELEMENTS:
  - Icon: Large Green Checkmark.
  - Title: "Обращение создано!".
  - Subtitle: "Номер: REQ-2026-885".
  - Text: "Плановая дата: 08/03/2026".
  - Button: "Создать еще одно" (Reset Flow).

# ==========================================
# 🎨 ВИЗУАЛЬНЫЕ ДОРАБОТКИ (UI TWEAKS)
# ==========================================
1. Поле "Приоритет":
   - Сделайте его не просто текстом, а Segmented Control (переключателем), который блокируется (Disabled), если система расчитала его жестко (например, для Регулятора).
   
2. Поле "Плановая дата":
   - Добавьте иконку "Календарь" справа.
   - Сделайте фон поля светло-серым (Read-only), чтобы показать, что оно расчетное.
   - Цвет текста: Зеленый (если дата далеко), Красный (если дата < 2 дней).

3. Поле "ФИО Регистратора":
   - Должно быть предзаполнено (User Profile) и заблокировано, или вообще убрано в "шапку" интерфейса как "Профиль пользователя".

4. "Клиент / Неклиент":
   - Это не просто текст, это переключатель (Tabs/Segment). При переключении на "Неклиент" поля поиска CRM должны очищаться.
```
