### Промпт для Figma Make: сгенерировать Use Case «Просмотреть обращение»


**Роль:** Ты — системный аналитик уровня Senior. Сгенерируй Use Case строго по методологии Коберна/Вигерса. Сосредоточься на поведенческой логике, а не на деталях интерфейса.


**Цель промпта:** Получить готовый к передаче разработчикам Use Case, описывающий сценарий просмотра карточки обращения ответственным сотрудником. Результат должен подходить для включения в ТЗ или спецификацию.


**Формат ответа:** Markdown с обязательными разделами Use Case (см. ниже).



**Что нужно сделать:**
1. Описать **основной поток** как последовательность шагов «Пользователь → Система».
2. Чётко выделить: **предусловия**, **постусловия**, **альтернативные потоки**, **исключительные ситуации**.
3. Включить **бизнес‑правила** и **специальные требования** (безопасность, производительность, доступность).
4. Использовать терминологию банковской CRM (ДУКО, обращение, SLA, протокол, статус и т. п.).
5. **Не описывать** цвета, шрифты, отступы, иконки, расположение кнопок — только логика и данные.



**Контекст для генерации:**
- **Актор:** Ответственный сотрудник ДУКО (не супервизор).
- **Цель:** Изучить полную информацию по обращению (детали, историю, статусы, данные смежных подразделений), чтобы принять решение о дальнейших действиях.
- **Триггер:** Пользователь перешёл к карточке обращения из реестра (списка обращений) или по прямой ссылке.
- **Частота:** 50–150 раз в день на сотрудника.
- **Приоритет:** Высокий (критично для корректной обработки обращений и соблюдения SLA).


**Структура Use Case (обязательные разделы):**
1. **Паспорт Use Case** (ID, название, актор, цель, триггер, частота, приоритет).
2. **Предусловия** (что должно быть истинным до начала сценария).
3. **Основной поток** (нумерованный список шагов диалога «Пользователь → Система»).
4. **Постусловия** (состояние системы и данные после успешного завершения).
5. **Альтернативные потоки** (A1, A2, … — отклонения от основного сценария).
6. **Исключительные ситуации** (E1, E2, … — ошибки и сбои).
7. **Специальные требования** (производительность, безопасность, доступность).
8. **Бизнес‑правила** (BR‑01, BR‑02, … — логика работы с данными).
9. **Связь с другими Use Cases** (перечислить UC‑XXX, которые вызываются или зависят от этого сценария).



**Детали логики для включения:**
- Пользователь переходит к карточке обращения (из реестра или по ссылке).
- Система проверяет права доступа пользователя к данному обращению.
- Система загружает:
  - основные данные обращения (номер, дата регистрации, канал, SLA);
  - данные заявителя (ФИО, контакты, тип клиента);
  - текущий статус и ответственного;
  - содержание обращения;
  - историю обработки (протокол действий);
  - статусы и ответы смежных подразделений (HelpDesk, CRM и др.).
- Система отображает карточку с секциями:
  - «Заявитель»;
  - «Данные обращения»;
  - «Содержание обращения»;
  - «История обработки смежными подразделениями»;
  - «Следующее действие» (с полями для решения и ответа, если статус позволяет).
- Пользователь просматривает данные, может разворачивать/сворачивать секции.
- При необходимости пользователь переходит к другим действиям (обработка, отказ, запрос в смежное подразделение).


**Альтернативные потоки:**
- A1: Нет прав доступа → система показывает уведомление, перенаправляет в реестр.
- A2: Обращение не найдено → система отображает ошибку, предлагает вернуться в реестр.
- A3: Ошибка загрузки данных из смежной системы (например, HelpDesk) → система показывает заглушку и предупреждение, продолжает работу с остальными данными.
- A4: Обращение в архиве → система отображает карточку в режиме «только для чтения», блокирует редактируемые поля.

**Исключительные ситуации:**
- E1: Ошибка БД при загрузке → система показывает экран ошибки, логирует инцидент, предлагает повторить.
- E2: Таймаут запроса к API смежной системы → система уведомляет о задержке, даёт возможность продолжить работу с основной информацией.
**Специальные требования:**
- Производительность: полная загрузка карточки — ≤2 сек при нормальной нагрузке.
- Безопасность: проверка прав доступа на уровне записи (row‑level security); логирование просмотров (для аудита).
- Доступность: поддержка навигации с клавиатуры, ARIA‑метки для секций и полей.
- UI: поля, обязательные для заполнения на следующих шагах, отмечены красной звёздочкой (*).
**Бизнес‑правила:**
- BR‑01: Карточка отображается только для пользователей с правами доступа.
- BR‑02: В режиме «Архив» все поля доступны только для чтения.
- BR‑03: При ошибке загрузки части данных система продолжает работу с доступными данными (с уведомлениями).
- BR‑04: Протокол обработки загружается полностью, с фильтрацией по этапам (если предусмотрено).
- BR‑05: Поля для решения и ответа видны только в статусах «В работе», «На доведении».
**Связь с другими UC:**
- UC‑LIST‑01: Просмотр реестра обращений (предшествует, из него переходят к карточке).
- UC‑DECIDE‑01: Принять решение, создать ответ, довести ответ (может следовать после просмотра).
- UC‑QUERY‑01: Запросить информацию в смежном подразделении (может быть вызван из карточки).

**Пример начала ответа:**
```markdown

- **Идентификатор:** UC‑VIEW‑01  
- **Название:** Просмотреть обращение  
- **Актор:** Ответственный сотрудник ДУКО  
- **Цель:** ...
```
**Ожидаемый результат:** Полный Use Case в формате Markdown, готовый для включения в ТЗ или спецификацию.
