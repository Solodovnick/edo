### Промпт для Figma Make: сгенерировать Use Case «Поиск и навигация по реестру обращений ответственного» (готовый к разработке вариант)


**Роль:** Ты — системный аналитик уровня Senior. Сгенерируй Use Case строго по методологии Коберна/Вигерса. Результат должен быть пригоден для передачи разработчикам и включения в ТЗ.


**Цель промпта:** Получить полный, детализированный Use Case, описывающий сценарий просмотра, поиска, фильтрации и сортировки реестра обращений ответственным сотрудником (и супервизором).


**Формат ответа:** Markdown с чётко выделенными разделами Use Case (см. ниже). Используй нумерованные списки для шагов основного потока, маркированные — для перечней и альтернатив.


**Что нужно сделать:**
1. Сохранить и уточнить существующую структуру Use Case (11 разделов).
2. Дополнить **детализацию шагов** в основном потоке (добавить валидацию, промежуточные реакции системы, граничные условия).
3. Расширить **альтернативные потоки** — включить все вероятные отклонения (не только перечисленные).
4. Уточнить **исключительные ситуации** — добавить коды ошибок, действия системы, логирование.
5. Доработать **специальные требования** — конкретизировать метрики, механизмы безопасности, доступность.
6. В разделе **бизнес‑правила** явно прописать логику фильтров, поиска, пагинации, прав доступа.
7. В **Data Dictionary** дать полное описание каждой колонки (тип данных, обязательность, формат, ограничения).
8. В **связи с другими UC** перечислить все смежные сценарии (включая те, что могут быть вызваны из реестра).
9. Исключить UI‑детали (цвета, шрифты, отступы, иконки). Описывать только *поведение* и *логику данных*.



**Контекст для генерации:**
- **Актор:**  
  - Ответственный сотрудник ДУКО  
  - Супервизор (расширенные права)  
- **Цель:**  
  - Найти обращение по номеру, ФИО, содержанию  
  - Фильтровать по статусу, типу заявителя, ответственности  
  - Сортировать по дате, дедлайну для приоритизации  
  - Перейти к детализации обращения  
  - Начать регистрацию нового обращения  
- **Триггер:** Переход в раздел «Кабинет ответственного»  
- **Частота:** 20–50 раз/день на сотрудника  
- **Приоритет:** Критический  



**Требования к структуре (обязательные разделы):**
1. **Паспорт Use Case** (ID, название, актор, цель, триггер, частота, приоритет)  
2. **Предусловия** (условия до начала сценария)  
3. **Основной поток** (нумерованный список шагов «Пользователь → Система»)  
4. **Постусловия** (состояние системы после успешного завершения)  
5. **Альтернативные потоки** (A1, A2, … — отклонения от основного сценария)  
6. **Исключительные ситуации** (E1, E2, … — ошибки и сбои)  
7. **Специальные требования** (производительность, безопасность, доступность)  
8. **Бизнес‑правила** (BR‑01, BR‑02, … — логика работы с данными)  
9. **Описание данных (Data Dictionary)** (список колонок таблицы с деталями)  
10. **Связь с другими Use Cases** (перечень UC‑XXX)  
11. **Дополнительная информация** (версия, дата, автор, статус, связанные документы)  



**Детали логики для включения:**

- **Загрузка реестра:**  
  - Проверка роли → выбор набора данных (свои/подразделения/все)  
  - Параметры по умолчанию: фильтр «Мои обращения», сортировка «Дедлайн ASC», лимит 50  
  - Отображение таблицы, поиска, фильтров, пагинации  
- **Поиск:**  
  - Debounce 300 мс  
  - Поиск по: номер, ФИО заявителя, первые 80 симв. содержания  
  - Регистронезависимый (ILIKE)  
  - Индикация найденных фрагментов (подсветка)  
- **Фильтры:**  
  - Логика AND между группами, OR внутри группы (если есть множественный выбор)  
  - Обновление счётчиков в табах в реальном времени  
  - Сохранение комбинации фильтров  
- **Сортировка:**  
  - ASC/DESC/сброс по клику на заголовок  
  - Визуальная индикация направления (↑/↓)  
- **Переход к детализации:**  
  - Сохранение текущего состояния (фильтры, сортировка, страница) в `sessionStorage`  
  - Проверка прав доступа перед открытием карточки  
- **Создание нового обращения:**  
  - Сохранение контекста реестра  
  - Возврат к реестру с восстановлением состояния  


**Альтернативные потоки (дополнить):**
- A6: Пользователь меняет лимит пагинации (10/25/50/100) → система перезагружает данные с новым лимитом.
- A7: Пользователь переходит на другую страницу пагинации → система загружает следующую порцию записей.
- A8: Пользователь пытается отсортировать колонку без права сортировки → система игнорирует клик.
- A9: Поиск возвращает >1000 записей → система ограничивает выдачу, уведомляет о необходимости уточнить запрос.

- A10: Пользователь снимает все фильтры → система возвращается к параметрам по умолчанию.


**Исключительные ситуации (дополнить):**
- E4: Ошибка парсинга даты в фильтре → система сбрасывает фильтр, уведомляет пользователя.
- E5: Превышение квоты запросов (rate limiting) → система показывает «Слишком много запросов. Попробуйте позже», блокирует действия на 1 мин.
- E6: Невалидные данные от БД (null в обязательных полях) → система пропускает запись, логирует, продолжает с остальными.
- E7: Сессия истекла → система просит авторизоваться, сохраняет черновик поиска/фильтров (если возможно).


**Специальные требования (уточнить):**
- **Производительность:**  
  - Время загрузки реестра (50 записей): ≤1.5 сек  
  - Ответ поиска (debounce): 300 мс  
  - Применение фильтра: ≤500 мс  
  - Сортировка: ≤200 мс  
  - Таймаут запроса к БД: 10 сек → уведомление  
- **Безопасность:**  
  - JWT-аутентификация (`Authorization` header)  
  - Row-level security (RDS):  
    - Ответственный: `responsible_user_id = current_id` + подразделение (read-only)  
    - Супервизор: всё подразделение (read-write)  
  - Логирование: все просмотры, фильтры, поиски (user_id, timestamp, query)  
  - Маскирование: телефон, адрес в таблице (только для супервизора)  
- **Доступность (WCAG 2.1 AA):**  
  - Клавиатурная навигация: Tab, Enter, Escape, стрелки  
  - ARIA-labels для всех интерактивных элементов  
  - `role="table"`, `role="row"`, `role="cell"`  
  - Контраст текста: ≥4.5:1  
- **UI/UX:**  
  - Sticky header таблицы  
  - Анимация фильтров/поиска: fade-in 150 мс  
  - Сохранение состояния: фильтры, сортировка, страница в `sessionStorage`  


**Бизнес‑правила (расширить):**
- BR‑LIST‑09: Поиск поддерживает частичное совпадение (LIKE) и полнотекстовый поиск (при наличии).
- BR‑LIST‑10: Фильтры сохраняются между сеансами (если пользователь не очистил `sessionStorage`).
- BR‑LIST‑11: При смене роли (ответственный ↔ супервизор) реестр перезагружается с новыми правами.
- BR‑LIST‑12: Пагинация включается автоматически при >100 записях в выборке.
- BR‑LIST‑13: Сортировка по «Дедлайн» всегда учитывает часовой пояс пользователя.
- BR‑LIST‑14: Поле «Содержание (краткое)» обрезается до 80 символов + «...», tooltip показывает полный текст.
- BR‑LIST‑15: Кнопка «Создать обращение» доступна всегда, даже при 0 записей.


**Data Dictionary (детализировать):**
Для каждой колонки указать:  
- Название (в интерфейсе)  
- Тип данных (string, date, enum и т. п.)  
- Формат (например, «ДД.ММ.ГГГГ ЧЧ:ММ»)  
- Обязательность (да/нет)  
- Ограничения (длина, допустимые значения)  
- Источник данных (таблица/поле БД)  
- Примечания (например, tooltip при переполнении)


**Связь с другими UC (дополнить):**
- UC‑00: Регистрация нового обращения (из кнопки «Создать»)  
- UC‑01: Просмотр детализации обращения (переход по номеру)
