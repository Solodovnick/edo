**Роль:** Ты — системный аналитик уровня Senior. Сгенерируй Use Case строго по методологии Коберна/Вигерса (с акцентом на поведенческую логику, а не UI‑детали).
**Цель промпта:** Получить готовый к передаче разработчикам Use Case, описывающий полный цикл:
- принятия решения по обращению;
- формирования ответа клиенту (на основе шаблонов или вручную);
- отправки ответа («доведения») и фиксации результата.


**Формат ответа:** Markdown с обязательными разделами Use Case (см. ниже).


**Что нужно сделать:**
1. Описать **основной поток** как последовательность шагов «Пользователь → Система».
2. Чётко выделить **предусловия**, **постусловия**, **альтернативные потоки**, **исключительные ситуации**.
3. Включить **бизнес‑правила** и **специальные требования** (безопасность, производительность).
4. Использовать терминологию банковской CRM (ДУКО, обращение, SLA, протокол, статус, шаблон ответа и т. п.).
5. **Не описывать** цвета, шрифты, отступы, иконки, расположение кнопок.



**Контекст для генерации:**
- **Актор:** Ответственный сотрудник ДУКО (не супервизор).
- **Цель:**  
  - принять обоснованное решение по обращению;  
  - сформировать корректный ответ клиенту (с использованием шаблонов или без);  
  - довести ответ до клиента и зафиксировать результат в системе.
- **Триггер:** Обращение находится в статусе «В работе», все необходимые данные собраны.
- **Частота:** 30–100 раз в день на сотрудника.
- **Приоритет:** Критический (напрямую влияет на SLA и клиентский опыт).

**Структура Use Case (обязательные разделы):**

1. **Паспорт Use Case** (ID, название, актор, цель, триггер, частота, приоритет).
2. **Предусловия** (что должно быть истинным до начала сценария).
3. **Основной поток** (нумерованный список шагов диалога «Пользователь → Система»).
4. **Постусловия** (состояние системы и данные после успешного завершения).
5. **Альтернативные потоки** (A1, A2, … — отклонения от основного сценария).
6. **Исключительные ситуации** (E1, E2, … — ошибки и сбои).
7. **Специальные требования** (производительность, безопасность, доступность).
8. **Бизнес‑правила** (BR‑01, BR‑02, … — логика работы с данными).
9. **Связь с другими Use Cases** (перечислить UC‑XXX, которые вызываются или зависят от этого сценария).



**Детали логики для включения:**

- Пользователь открывает карточку обращения в статусе «В работе».
- Система отображает блоки:  
  - «Внутреннее решение» (обязательное поле);  
  - «Шаблон ответа клиенту» (выбор из списка + предпросмотр/редактирование);  
  - кнопки «Сохранить черновик», «Довести ответ».
- Пользователь:  
  - заполняет поле «Внутреннее решение»;  
  - выбирает шаблон ответа, редактирует текст;  
  - нажимает «Довести ответ».
- Система:  
  - валидирует заполнение обязательных полей;  
  - обновляет статус обращения (например, «На доведении» / «Закрыто»);  
  - фиксирует в протоколе: «Ответ доведён», дата/время, ID пользователя;  
  - отправляет ответ клиенту (через канал, указанный в обращении);  
  - сохраняет историю изменений;  
  - уведомляет заинтересованных (супервизора, смежные подразделения).
- Система возвращает пользователя в реестр обращений с обновлённым статусом.



**Альтернативные потоки:**
- A1: Не заполнено «Внутреннее решение» → система требует заполнения.
- A2: Выбранный шаблон недоступен/устарел → система показывает ошибку, предлагает выбрать другой.
- A3: Ошибка отправки ответа клиенту → система сообщает об ошибке, сохраняет черновик.

- A4: Пользователь нажимает «Сохранить черновик» → система сохраняет данные без смены статуса, возвращает в реестр.


**Исключительные ситуации:**
- E1: Ошибка БД при сохранении → система показывает сообщение, логирует ошибку, предлагает повторить.
- E2: Таймаут API отправки ответа → система уведомляет о сбое, оставляет статус «В работе».

- E3: Потеря сессии → система восстанавливает черновик из `sessionStorage` (если возможно).


**Специальные требования:**
- Производительность: сохранение черновика — ≤500 мс; отправка ответа — ≤2 сек.
- Безопасность: все действия логируются (ID пользователя, timestamp, ID обращения); доступ только по ролям.
- Доступность: поддержка навигации с клавиатуры, ARIA‑метки для полей и кнопок.
- UI: поля с красной звёздочкой (*) — обязательные для заполнения.


**Бизнес‑правила:**
- BR‑01: Ответ можно довести только при заполненном «Внутреннем решении».
- BR‑02: Шаблоны ответов хранятся в справочнике и актуализируются регулярно.
- BR‑03: Статус «Закрыто» устанавливается только после успешной отправки ответа клиенту.
- BR‑04: История всех черновиков и отправленных ответов сохраняется в протоколе обращения.
- BR‑05: При ошибке отправки статус остаётся «В работе» или «На доведении» (в зависимости от этапа).

**Связь с другими UC:**
- UC‑01: Просмотр карточки обращения (предшествует).
- UC‑QUERY‑01: Запрос в смежное подразделение (может быть вызван внутри сценария).
- UC‑NOTIFY‑01: Отправка уведомлений (вызывается при отправке ответа).
