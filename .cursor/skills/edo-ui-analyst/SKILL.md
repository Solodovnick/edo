# EDO Bank — Аналитик требований

## Когда применять

- Добавление новой функциональности в систему обращений
- Анализ UI-артефактов из папки `.edo/` (*.md файлы с описанием кабинетов)
- Обновление `docs/functional-requirements.md` или `docs/business-requirements.md`
- Составление acceptance criteria для user stories

## Инструкции

1. **Источник правды:** `docs/functional-requirements.md`. Все новые требования добавляются туда по структуре FR-XX.X.
2. **Формат требования:** ID | Требование | Приоритет (P0/P1/P2). Ссылка на источник — MD-артефакт из `.edo/`.
3. **Приоритеты:**
   - P0 — критично для MVP (регистрация, жизненный цикл, кабинеты)
   - P1 — важно (CRM, аналитика, уведомления)
   - P2 — усиление (администрирование, экспорт)
4. **Роли пользователей:** Ответственный специалист / Аудитор / Администратор. Всегда указывать, для кого требование.
5. После изменений обновить `docs/VERSIONS.md` (PATCH или MINOR).

## Домен EDO Bank

- **Обращения:** физлицо / юрлицо / регулятор (ЦБ РФ)
- **Типы:** письменное / устное
- **SLA:** дедлайн + таймер обратного отсчёта
- **Кабинеты:** Ответственного, Аудитора
- **CRM:** поиск клиента, VIP-статус, история обращений
