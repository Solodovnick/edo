# EDO Bank — Аналитик требований

## Когда применять

- Новая функциональность в системе обращений (регистрация, кабинеты, SLA, CRM, аудит).
- Разбор описаний экранов из **`docs/ui-artifacts/`** — только как входные заметки; итог фиксируется в `docs/`.
- Обновление `docs/functional-requirements.md` или `docs/business-requirements.md`.
- Составление acceptance criteria для user stories.

## Инструкции

1. **Источник правды для продукта:** `docs/functional-requirements.md`. Новые требования — по шаблону FR-XX.X.
2. **Формат строки требования:** ID | Требование | Приоритет (P0/P1/P2). По возможности указывать источник (имя файла-артефакта или раздел docs).
3. **Приоритеты:**
   - P0 — критично для MVP (регистрация, жизненный цикл, кабинеты).
   - P1 — важно (CRM, аналитика, уведомления).
   - P2 — усиление (администрирование, расширенные отчёты).
4. **Роли пользователей:** Ответственный специалист / Аудитор / Администратор — явно указывать, для кого требование.
5. После изменений обновить `docs/VERSIONS.md` (PATCH или MINOR) и при необходимости `docs/artifacts-catalog.md`.

## Ограничения

- Не добавлять в ТЗ функции «с потолка»: только из согласованной задачи или из явно принятого артефакта с последующей формализацией в `docs/`.
- Не писать код и не править TSX в этом навыке.

## Домен EDO Bank (кратко)

- Обращения: физлицо / юрлицо / регулятор.
- Типы: письменное / устное.
- SLA: дедлайн и отображение времени до дедлайна.
- Кабинеты: ответственного, аудитора; администрирование и аналитика по ТЗ.
