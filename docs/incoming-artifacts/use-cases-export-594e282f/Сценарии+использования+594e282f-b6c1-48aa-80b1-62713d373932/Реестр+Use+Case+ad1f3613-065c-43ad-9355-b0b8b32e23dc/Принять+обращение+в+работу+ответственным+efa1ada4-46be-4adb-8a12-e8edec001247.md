---

## 1. Паспорт Use Case

- **Идентификатор:** UC-R03

- **Название:** Принять обращение в работу ответственным сотрудником ДУКО

- **Актор:** Ответственный сотрудник ДУКО (Группа обработки обращений клиентов)

- **Цель:** Официально взять обращение в работу, зафиксировав начало обработки, для изменения статуса на «В работе», запуска таймера SLA и получения прав на редактирование полей решения и ответа клиенту

- **Триггер:** Пользователь открыл карточку обращения в статусе «Назначено» и принял решение приступить к обработке

- **Частота использования:** 50–100 раз в день на одного сотрудника

- **Приоритет:** Высокий (критично для соблюдения SLA и контроля рабочей нагрузки)

- **Уровень:** Пользовательская задача (User Goal Level)

---

## 2. Предусловия

- Обращение существует в системе с уникальным идентификатором

- Текущий статус обращения — «Назначено»

- Пользователь аутентифицирован в системе

- Пользователь является ответственным сотрудником, назначенным на данное обращение (поле `responsible_user_id` содержит ID текущего пользователя)

- Пользователь имеет роль «Ответственный сотрудник ДУКО» с правами на изменение статуса обращения

- Карточка обращения открыта и отображена пользователю (выполнен UC-01 «Просмотр детализации обращения»)

- Поля «Внутреннее решение» и «Шаблон ответа клиенту» заблокированы для редактирования (read-only mode)

- База данных доступна и транзакции выполняются корректно

---

## 3. Основной поток (Main Flow)

### Фаза 1: Инициация принятия обращения

1. **Пользователь** активирует действие «Принять в работу».

2. **Система** проверяет текущий статус обращения в базе данных.

3. **Система** подтверждает, что статус обращения — «Назначено».

4. **Система** проверяет, что пользователь является назначенным ответственным (сравнивает `current_user_id` с `responsible_user_id` обращения).

### Фаза 2: Обновление данных обращения

1. **Система** начинает транзакцию базы данных.

2. **Система** обновляет запись обращения:

    - Устанавливает поле `status` = «В работе»

    - Устанавливает поле `work_started_at` = текущая дата и время (timestamp)

    - Устанавливает поле `last_modified_at` = текущая дата и время

    - Устанавливает поле `last_modified_by` = ID текущего пользователя

1. **Система** проверяет, запущен ли таймер SLA (поле `sla_started_at`):

    - Если `sla_started_at` = NULL, устанавливает `sla_started_at` = текущая дата и время

    - Если `sla_started_at` уже установлен, оставляет значение без изменений

1. **Система** создаёт запись в протоколе обработки (таблица `appeal_audit_log`):

    - `appeal_id` = ID текущего обращения

    - `action_type` = «work_accepted»

    - `action_description` = «Обращение взято в работу»

    - `performed_by_user_id` = ID текущего пользователя

    - `performed_at` = текущая дата и время

    - `stage` = «processing»

    - `previous_status` = «Назначено»

    - `new_status` = «В работе»

### Фаза 3: Завершение операции

1. **Система** фиксирует транзакцию в базе данных (COMMIT).

2. **Система** отправляет уведомление супервизору о начале обработки обращения (опционально, согласно настройкам уведомлений).

3. **Система** обновляет интерфейс карточки обращения:

    - Изменяет отображаемый статус на «В работе»

    - Разблокирует поля «Внутреннее решение» и «Шаблон ответа клиенту» для редактирования

    - Удаляет или скрывает кнопку «Принять в работу»

    - Отображает кнопки следующего этапа: «Сохранить черновик», «Отправить на доведение»

    - Обновляет блок «Следующее действие» с инструкциями для этапа обработки

1. **Система** отображает уведомление пользователю: «Обращение №[номер] принято в работу».

2. **Система** остаётся на странице карточки обращения.

---

## 4. Постусловия

### Успешное завершение:

- Статус обращения в базе данных изменён на «В работе»

- Поле `work_started_at` содержит дату и время начала обработки

- Таймер SLA запущен (поле `sla_started_at` заполнено)

- В протоколе обработки создана запись о принятии обращения в работу с фиксацией исполнителя, даты и времени

- Поля «Внутреннее решение» и «Шаблон ответа клиенту» доступны для редактирования

- Пользователь видит обновлённый статус и может продолжить работу с обращением

- Обращение отображается в реестре с новым статусом «В работе»

- Метрики SLA начали отсчёт времени обработки

- Супервизор получил уведомление о начале обработки (если включены уведомления)

### Состояние системы:

- Транзакция базы данных успешно завершена (committed)

- Целостность данных сохранена

- Аудиторский след зафиксирован

- Система готова к выполнению UC-02 «Обработка обращения ответственным»

## 5. Альтернативные потоки

### A1: Обращение уже принято в работу другим сотрудником

- **Точка ветвления:** Шаг 3

- **Условие:** Текущий статус обращения — «В работе» (изменён между открытием карточки и нажатием кнопки)

- **Действия:**

    1. Система отменяет операцию (не начинает транзакцию)

    2. Система отображает уведомление: «Обращение №[номер] уже находится в работе»

    3. Система автоматически обновляет данные карточки обращения из базы данных

    4. Система отображает актуальный статус «В работе» и дату начала обработки

    5. Система разблокирует поля для редактирования (если пользователь остаётся ответственным)

    6. Сценарий завершается, пользователь может продолжить обработку без повторного принятия

### A2: Обращение переназначено другому сотруднику

- **Точка ветвления:** Шаг 4

- **Условие:** Поле `responsible_user_id` не совпадает с `current_user_id` (обращение переназначено между открытием карточки и принятием в работу)

- **Действия:**

    1. Система прерывает выполнение операции

    2. Система отображает уведомление: «Обращение №[номер] переназначено другому сотруднику. У вас нет прав для принятия этого обращения в работу.»

    3. Система отображает информацию о текущем ответственном сотруднике (ФИО, должность)

    4. Система блокирует все интерактивные элементы карточки (режим read-only)

    5. Система предлагает действие: «Вернуться к списку обращений»

    6. Сценарий завершается

### A3: Статус обращения изменён на последующий этап

- **Точка ветвления:** Шаг 3

- **Условие:** Текущий статус обращения — «На доведении», «Закрыто» или «Архив» (обращение прошло дальше по процессу)

- **Действия:**

    1. Система отменяет операцию

    2. Система отображает уведомление: «Обращение №[номер] уже находится на этапе "[текущий статус]". Принятие в работу невозможно.»

    3. Система автоматически обновляет карточку обращения из базы данных

    4. Система отображает актуальный статус и историю обработки

    5. Система переводит все поля в режим read-only

    6. Сценарий завершается

### A4: Попытка принять обращение в статусе «Готово к регистрации» или «Зарегистрировано»

- **Точка ветвления:** Шаг 3

- **Условие:** Статус обращения — «Готово к регистрации», «Зарегистрировано» или «На аудите» (обращение находится на предшествующих этапах)

- **Действия:**

    1. Система отменяет операцию

    2. Система отображает уведомление: «Обращение №[номер] находится на этапе "[текущий статус]". Необходимо дождаться назначения ответственного.»

    3. Система отображает информацию о текущем этапе процесса

    4. Система блокирует поля решения и ответа

    5. Сценарий завершается

### A5: Пользователь отменяет принятие обращения

- **Точка ветвления:** Между шагом 1 и шагом 2

- **Условие:** Система отображает диалог подтверждения «Вы уверены, что готовы приступить к обработке обращения?», и пользователь выбирает «Отмена»

- **Действия:**

    1. Система отменяет операцию (не выполняет шаги 2–12)

    2. Система закрывает диалог подтверждения

    3. Карточка обращения остаётся в исходном состоянии (статус «Назначено», поля заблокированы)

    4. Сценарий завершается

## 6. Исключительные ситуации (Exception Flows)

### E1: Ошибка подключения к базе данных

- **Точка возникновения:** Шаг 2, 5, 6, 7 или 8

- **Условие:** Потеря соединения с базой данных или таймаут запроса

- **Действия:**

    1. Система прерывает выполнение операции

    2. Система выполняет ROLLBACK транзакции (если она была начата)

    3. Система отображает уведомление: «Не удалось принять обращение в работу. Ошибка подключения к базе данных. Попробуйте повторить операцию через несколько секунд.»

    4. Система логирует ошибку с уровнем ERROR:

        - Тип ошибки: DATABASE_CONNECTION_FAILED

        - Appeal ID

        - User ID

        - Timestamp

        - Текст ошибки БД

    1. Система отправляет алерт команде DevOps

    2. Система оставляет карточку обращения в исходном состоянии (статус «Назначено»)

    3. Кнопка «Принять в работу» остаётся доступной для повторной попытки

    4. Сценарий завершается

### E2: Нарушение целостности данных (конкурентное изменение)

- **Точка возникновения:** Шаг 9 (COMMIT транзакции)

- **Условие:** Другой процесс одновременно изменил запись обращения, возникла блокировка или конфликт версий

- **Действия:**

    1. Система получает ошибку CONCURRENT_MODIFICATION или LOCK_TIMEOUT

    2. Система выполняет ROLLBACK транзакции

    3. Система повторяет попытку выполнения транзакции (retry logic, максимум 3 попытки с интервалом 500ms)

    4. Если после 3 попыток ошибка сохраняется:

        - Система отображает уведомление: «Не удалось принять обращение в работу. Данные обращения были изменены другим пользователем. Обновите страницу и повторите попытку.»

        - Система логирует ошибку с уровнем WARNING

        - Система предлагает действие: «Обновить карточку обращения»

    1. Сценарий завершается

### E3: Недостаточно прав доступа

- **Точка возникновения:** Шаг 4

- **Условие:** Между открытием карточки и принятием в работу у пользователя были отозваны права на обработку обращений

- **Действия:**

    1. Система проверяет актуальные разрешения пользователя (permissions check)

    2. Система обнаруживает отсутствие роли «Ответственный сотрудник ДУКО» или прав на изменение статуса

    3. Система отменяет операцию

    4. Система отображает уведомление: «Недостаточно прав для принятия обращения в работу. Обратитесь к администратору системы.»

    5. Система логирует событие безопасности (SECURITY_AUDIT_LOG):

        - Event: UNAUTHORIZED_ACCESS_ATTEMPT

        - User ID

        - Appeal ID

        - Action: ACCEPT_APPEAL

        - Timestamp

    1. Система переводит карточку в режим read-only

    2. Сценарий завершается

### E4: Превышен лимит одновременно обрабатываемых обращений

- **Точка возникновения:** Шаг 5

- **Условие:** Бизнес-правило ограничивает количество обращений в статусе «В работе» на одного сотрудника (например, максимум 10), и лимит уже достигнут

- **Действия:**

    1. Система выполняет запрос к БД для подсчёта обращений в работе у текущего пользователя

    2. Система обнаруживает превышение лимита (COUNT >= max_concurrent_appeals)

    3. Система отменяет операцию (не начинает транзакцию)

    4. Система отображает уведомление: «Вы не можете принять в работу более [лимит] обращений одновременно. Завершите обработку текущих обращений, прежде чем принимать новые.»

    5. Система отображает список текущих обращений пользователя в статусе «В работе» (быстрый доступ к ним)

    6. Сценарий завершается

### E5: Таймаут операции

- **Точка возникновения:** Шаги 5–9

- **Условие:** Выполнение транзакции заняло более 30 секунд (настроенный таймаут)

- **Действия:**

    1. Система прерывает выполнение операции

    2. Система выполняет ROLLBACK транзакции

    3. Система отображает уведомление: «Превышено время ожидания операции. Попробуйте повторить попытку.»

    4. Система логирует ошибку с уровнем WARNING:

        - Тип: OPERATION_TIMEOUT

        - Appeal ID

        - User ID

        - Duration: 30+ секунд

    1. Система оставляет карточку в исходном состоянии

    2. Кнопка «Принять в работу» остаётся доступной

    3. Сценарий завершается

## 7. Специальные требования

### 7.1. Производительность

- **NFR-PERF-01:** Время выполнения операции принятия обращения в работу (шаги 2–11) не должно превышать 2 секунд при нормальной нагрузке

- **NFR-PERF-02:** Время обновления интерфейса после принятия (шаг 11) не должно превышать 500 миллисекунд

- **NFR-PERF-03:** Операция должна выполняться в рамках одной атомарной транзакции БД с уровнем изоляции READ COMMITTED

- **NFR-PERF-04:** Система должна поддерживать одновременное принятие обращений в работу 50+ пользователями без деградации производительности

### 7.2. Безопасность

- **NFR-SEC-01:** Все операции изменения статуса обращения должны логироваться в аудиторском журнале с фиксацией:

    - ID пользователя, выполнившего действие

    - Точной даты и времени (с точностью до миллисекунд)

    - Предыдущего и нового статуса

    - IP-адреса клиента

    - Session ID

- **NFR-SEC-02:** Операция принятия в работу доступна только пользователям с ролью «Ответственный сотрудник ДУКО» и правом `appeal.status.update`

- **NFR-SEC-03:** Проверка прав доступа должна выполняться на уровне API (backend), а не только на уровне UI

- **NFR-SEC-04:** Система должна блокировать повторное выполнение операции при дублировании запроса (idempotency check по уникальному request ID)

- **NFR-SEC-05:** Все изменения статуса должны выполняться в рамках зашифрованного HTTPS-соединения

### 7.3. Надёжность

- **NFR-REL-01:** Система должна гарантировать атомарность операции (все шаги 5–8 выполняются полностью или не выполняются вовсе)

- **NFR-REL-02:** При сбое на любом шаге транзакции все изменения должны быть откачены (ROLLBACK)

- **NFR-REL-03:** Система должна поддерживать механизм повторных попыток (retry) при временных ошибках БД (максимум 3 попытки)

- **NFR-REL-04:** Целостность данных должна быть защищена от конкурентных изменений (optimistic locking или row-level locking)

### 7.4. Доступность

- **NFR-AVAIL-01:** Операция принятия в работу должна быть доступна для выполнения с клавиатуры (горячая клавиша или последовательность Tab + Enter)

- **NFR-AVAIL-02:** Уведомления о результате операции должны быть доступны для screen readers (ARIA live regions)

- **NFR-AVAIL-03:** Интерфейс должен соответствовать стандарту WCAG 2.1 уровня AA

### 7.5. Отказоустойчивость

- **NFR-FAULT-01:** При недоступности службы уведомлений операция принятия в работу должна завершиться успешно (уведомление — некритичный компонент)

- **NFR-FAULT-02:** Ошибки при отправке уведомлений должны логироваться отдельно и не влиять на основной процесс

## 8. Бизнес-правила

### BR-ACCEPT-01: Переход статуса

**Правило:** Операция «Принять в работу» разрешена только для обращений в статусе «Назначено».

**Обоснование:** Обеспечивает последовательность этапов обработки согласно BPMN-процессу.

**Исключения:** Нет.

---

### BR-ACCEPT-02: Проверка ответственности

**Правило:** Принять обращение в работу может только тот сотрудник, который указан в поле `responsible_user_id` обращения.

**Обоснование:** Предотвращает несанкционированный доступ и конфликты при обработке.

**Исключения:** Супервизор с правом `appeal.override` может принять в работу любое обращение своего подразделения.

---

### BR-ACCEPT-03: Старт таймера SLA

**Правило:** Таймер SLA запускается в момент принятия обращения в работу, если ранее не был запущен на этапе регистрации.

**Расчёт дедлайна:**

- Устное обращение: `sla_started_at + 7 рабочих дней`

- Письменное обращение: `sla_started_at + 15 рабочих дней`

- Регуляторное обращение: `sla_started_at + 3 рабочих дня`

**Обоснование:** Фиксация точки отсчёта для контроля соблюдения сроков обработки.

**Исключения:** Если `sla_started_at` уже установлен на этапе регистрации или аудита, значение не изменяется.

---

### BR-ACCEPT-04: Аудиторский след

**Правило:** Каждое изменение статуса обращения должно фиксироваться в протоколе обработки с записью:

- Типа действия (`action_type` = «work_accepted»)

- Описания действия («Обращение взято в работу»)

- ID исполнителя

- Точной даты и времени

- Предыдущего и нового статуса

**Обоснование:** Обеспечение прозрачности процесса и возможности аудита.

**Исключения:** Нет.

---

### BR-ACCEPT-05: Разблокировка полей для редактирования

**Правило:** При переходе обращения в статус «В работе» поля «Внутреннее решение» и «Шаблон ответа клиенту» становятся доступными для редактирования.

**Обоснование:** Ответственный сотрудник получает права на заполнение решения только после официального принятия обращения в работу.

**Исключения:** Супервизор может редактировать поля на любом этапе.

---

### BR-ACCEPT-06: Лимит одновременно обрабатываемых обращений

**Правило:** Один сотрудник может одновременно иметь в работе не более 10 обращений (настраиваемый параметр `max_concurrent_appeals_per_user`).

**Обоснование:** Предотвращение перегрузки сотрудника и обеспечение качества обработки.

**Исключения:** Супервизор может иметь в работе до 20 обращений. Настройка может быть изменена администратором системы.

---

### BR-ACCEPT-07: Уникальность статуса «В работе» для обращения

**Правило:** Обращение может быть принято в работу только один раз. Повторное принятие в работу того же обращения невозможно.

**Обоснование:** Предотвращение дублирования записей в протоколе и ошибок при расчёте метрик.

**Исключения:** При возврате обращения на доработку (статус «Возвращено на доработку») возможно повторное принятие в работу с созданием новой записи в протоколе.

---

### BR-ACCEPT-08: Уведомление супервизора

**Правило:** При принятии обращения в работу система отправляет уведомление супервизору подразделения, если в настройках подразделения включён параметр `notify_supervisor_on_work_start = true`.

**Обоснование:** Информирование руководства о ходе работы для оперативного контроля.

**Исключения:** Уведомление не отправляется в нерабочее время (с 20:00 до 8:00, выходные и праздники) — уведомления накапливаются и отправляются единым дайджестом в начале рабочего дня.

## 9. Связь с другими Use Cases

### Предшествующие Use Cases:

- **UC-00:** Регистрация нового обращения

    - *Связь:* Обращение должно быть зарегистрировано до назначения ответственного

- **UC-AUDIT-01:** Аудит обращения

    - *Связь:* После прохождения аудита обращение попадает в очередь на обработку

- **UC-ASSIGN-01:** Назначение ответственного на обращение

    - *Связь:* Обращение должно быть назначено на конкретного сотрудника (статус «Назначено») перед принятием в работу

- **UC-01:** Просмотр детализации обращения

    - *Связь:* Пользователь просматривает карточку обращения перед принятием в работу

- **UC-LIST-01:** Поиск и навигация по реестру обращений

    - *Связь:* Пользователь выбирает обращение из реестра для принятия в работу

### Последующие Use Cases:

- **UC-R05:** Обработка обращения ответственным

    - *Связь:* После принятия в работу пользователь переходит к заполнению полей решения и ответа

- **UC-R06:** Отправка обращения на доведение

    - *Связь:* Завершение обработки и переход к следующему этапу

### Параллельные Use Cases:

- **UC-COMMENT-01:** Добавление комментария к обращению

    - *Связь:* После принятия в работу пользователь может добавлять комментарии в процессе обработки

- **UC-HISTORY-01:** Просмотр протокола обработки

    - *Связь:* Запись о принятии в работу отображается в протоколе обработки

- **UC-NOTIFY-01:** Просмотр уведомлений

    - *Связь:* Супервизор получает уведомление о принятии обращения в работу

### Альтернативные Use Cases:

- **UC-REJECT-01:** Отказ от обработки обращения

    - *Связь:* Вместо принятия в работу пользователь может отказаться от обращения (если предусмотрено процессом)

- **UC-REASSIGN-01:** Переназначение обращения на другого сотрудника

    - *Связь:* Вместо принятия в работу текущий ответственный может инициировать переназначение

|Атрибут|Предусловие|Постусловие|Тип изменения|
|-|-|-|-|
|`status`|«Назначено»|«В работе»|UPDATE|
|`work_started_at`|NULL|TIMESTAMP|INSERT|
|`sla_started_at`|NULL или TIMESTAMP|TIMESTAMP (если NULL)|INSERT/PRESERVE|

### 10.4. Метрики для мониторинга

- **Время от назначения до принятия в работу** (target: < 30 минут)

- **Количество отказов от принятия** (для анализа причин отказов)

- **Среднее количество обращений в работе на сотрудника** (для балансировки нагрузки)

- **Процент обращений, принятых в работу в тот же день** (target: > 95%)
