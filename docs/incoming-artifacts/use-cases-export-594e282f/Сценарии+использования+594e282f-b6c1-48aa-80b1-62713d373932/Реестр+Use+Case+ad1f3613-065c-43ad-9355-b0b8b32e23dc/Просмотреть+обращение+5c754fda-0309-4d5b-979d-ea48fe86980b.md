---

## 1. Паспорт Use Case

- **Идентификатор:** UC-VIEW-01

- **Название:** Просмотр детализированной карточки обращения клиента

- **Актор:** Ответственный сотрудник ДУКО (Группа обработки обращений клиентов)

- **Цель:** Изучить полную информацию по обращению (детали претензии, историю обработки, данные заявителя, статусы смежных подразделений, ответы на запросы) для принятия обоснованного решения о дальнейших действиях и обеспечения качественной обработки обращения в рамках установленного SLA

- **Триггер:** Пользователь выбрал обращение из реестра (UC-LIST-01) или перешёл по прямой ссылке на карточку обращения

- **Частота использования:** 50–150 раз в день на одного сотрудника

- **Приоритет:** Высокий (критично для корректной обработки обращений, соблюдения SLA и качества обслуживания клиентов)

- **Уровень:** Пользовательская задача (User Goal Level)

---

## 2. Предусловия

- Обращение существует в системе с уникальным идентификатором

- Пользователь аутентифицирован в системе

- Пользователь имеет роль «Ответственный сотрудник ДУКО» или «Супервизор ДУКО»

- Пользователь имеет права доступа к данной категории обращений (согласно политике безопасности)

- База данных доступна и выполняет запросы корректно

- Интеграционные API смежных систем (HelpDesk, CRM) доступны или предусмотрен fallback-режим

---

## 3. Основной поток (Main Flow)

### Фаза 1: Инициация просмотра обращения

1. **Пользователь** переходит к карточке обращения одним из способов:

    - Кликает на номер обращения в реестре (UC-LIST-01)

    - Переходит по прямой ссылке (URL с ID обращения)

    - Использует функцию быстрого поиска по номеру обращения

1. **Система** получает идентификатор обращения из URL или контекста перехода.

2. **Система** проверяет валидность идентификатора (формат UUID или числовой ID).

### Фаза 2: Проверка прав доступа

1. **Система** выполняет запрос к базе данных для проверки существования обращения:

    - SELECT FROM `appeals` WHERE `appeal_id = ?`

1. **Система** проверяет права доступа пользователя к обращению:

    - **Проверка роли:** пользователь имеет роль «Ответственный сотрудник ДУКО» или «Супервизор ДУКО»

    - **Проверка ответственности:**

        - Если роль «Ответственный сотрудник»: проверяет, что `responsible_user_id = current_user_id` ИЛИ `created_by_user_id = current_user_id` ИЛИ `department_id = user.department_id`

        - Если роль «Супервизор»: доступ ко всем обращениям подразделения

    - **Проверка прав на просмотр:** пользователь имеет право `appeal.view`

1. **Система** подтверждает наличие прав доступа.

### Фаза 3: Загрузка основных данных обращения

1. **Система** выполняет запрос к базе данных для загрузки основных данных обращения:

    ```SQL
    SELECT 
      appeal_id, appeal_number, status, substatus,
      client_type, communication_channel, category,
      created_at, registered_at, sla_deadline, sla_started_at,
      responsible_user_id, created_by_user_id, department_id,
      appeal_content, internal_decision, response_text,
      work_started_at, decision_made_at, closed_at,
      sla_met, processing_duration, is_blocked,
      applicant_full_name, applicant_phone, applicant_email, applicant_address
    FROM appeals
    WHERE appeal_id = ?
    ```

1. **Система** загружает данные ответственного сотрудника:

    - JOIN с таблицей `users` по `responsible_user_id`

    - Получает: ФИО, должность, email, телефон ответственного

### Фаза 4: Загрузка протокола обработки

1. **Система** выполняет запрос к таблице аудита для загрузки истории действий:

    ```SQL
    SELECT 
      action_type, action_description, performed_by_user_id,
      performed_at, stage, previous_status, new_status, metadata
    FROM appeal_audit_log
    WHERE appeal_id = ?
    ORDER BY performed_at DESC
    ```

1. **Система** группирует записи протокола по этапам процесса (регистрация, аудит, обработка, доведение, закрытие).

### Фаза 5: Загрузка данных из смежных подразделений

1. **Система** проверяет наличие связанных запросов в смежные подразделения:

    - SELECT FROM `inter_department_queries` WHERE `appeal_id = ?`

1. **Система** для каждого найденного запроса загружает:

    - Номер запроса, целевое подразделение, статус, дату отправки

    - Текст запроса, текст ответа (если получен)

    - Плановый и фактический срок ответа

1. **Система** выполняет параллельные запросы к интеграционным API смежных систем (с таймаутом 5 секунд на каждый):

    - **HelpDesk API:** GET `/api/tickets?appeal_id={id}`

        - Получает: статус тикета, дату создания, дату решения, описание решения

    - **CRM API:** GET `/api/customer/history?client_id={id}`

        - Получает: историю взаимодействий с клиентом, предыдущие обращения

    - **Scoring API:** GET `/api/client/score?client_id={id}`

        - Получает: рейтинг клиента, сегмент (VIP/стандарт), риск-профиль

1. **Система** обрабатывает ответы от интеграционных API:

    - Если API доступен: сохраняет полученные данные

    - Если API недоступен или таймаут: помечает источник как недоступный, продолжает работу

### Фаза 6: Загрузка вложений и связанных документов

1. **Система** выполняет запрос к таблице вложений:

    - SELECT FROM `appeal_attachments` WHERE `appeal_id = ?`

1. **Система** для каждого вложения получает:

    - Имя файла, размер, тип, дату загрузки

    - Подписанный URL для скачивания (signed URL с ограниченным сроком действия)

### Фаза 7: Расчёт динамических показателей

1. **Система** рассчитывает текущее состояние SLA:

    - Время до дедлайна: `time_remaining = sla_deadline - current_time`

    - Процент использованного времени: `sla_usage_percent = (current_time - sla_started_at) / (sla_deadline - sla_started_at) * 100`

    - Статус SLA: «в норме» (< 50%), «внимание» (50-80%), «критично» (> 80%), «просрочено» (> 100%)

1. **Система** определяет доступные действия для текущего статуса обращения:

    - Статус «Назначено»: доступны «Принять в работу», «Отказаться от обращения»

    - Статус «В работе»: доступны «Запросить информацию», «Довести ответ», «Сохранить черновик»

    - Статус «На доведении»: доступны «Просмотр истории»

    - Статус «Закрыто»/«Архив»: режим только для чтения

### Фаза 8: Формирование и отображение карточки обращения

1. **Система** компонует данные в структуру карточки обращения с секциями:

    **Секция «Заголовок карточки»:**

    - Номер обращения (крупный шрифт)

    - Тип клиента (badge)

    - Канал поступления (badge)

    - Индикатор SLA (прогресс-бар с цветовой индикацией)

    - Кнопка «Назад к реестру»

    **Секция «Заявитель» (сворачиваемая):**

    - ФИО заявителя

    - Телефон

    - Email (если есть)

    - Адрес проживания/юридический адрес

    - Тип клиента (физическое/юридическое лицо)

    - Сегмент клиента (если загружен из CRM)

    **Секция «Данные обращения» (сворачиваемая):**

    - Дата регистрации (дата + время)

    - Плановый срок исполнения (дата + индикатор просрочки)

    - Текущий статус (badge с цветом)

    - Подстатус (если есть)

    - Ответственный сотрудник (ФИО, должность)

    - Категория обращения

    - Канал поступления

    **Секция «Содержание обращения»:**

    - Текст обращения (multiline, read-only)

    - Дата и время создания

    - Автор записи

    **Секция «История обработки смежными подразделениями» (аккордеон, свернут по умолчанию):**

    - **Подсекция «HelpDesk»:**

        - Статус тикета (badge)

        - Дата создания тикета

        - Дата решения

        - Текст решения (expandable)

        - Исполнитель

    - **Подсекция «Запросы в подразделения»:**

        - Список всех запросов с информацией:

            - Номер запроса, целевое подразделение, статус

            - Дата отправки, плановый срок, фактический срок

            - Текст запроса (expandable)

            - Текст ответа (expandable, если получен)

    - **Подсекция «CRM»:**

        - История предыдущих обращений клиента (если доступно)

        - Оценка удовлетворённости по предыдущим обращениям

    **Секция «Протокол обработки» (сайдбар, скроллируемый):**

    - Фильтр по этапам (dropdown: «Все», «Регистрация», «Аудит», «Обработка», «Доведение»)

    - Хронологический список записей (от новых к старым):

        - Дата и время действия

        - Этап процесса (badge)

        - Описание действия

        - ФИО исполнителя

        - Комментарий (если есть)

    **Секция «Следующее действие» (если статус позволяет редактирование):**

    - Заголовок с описанием текущего этапа

    - Список обязательных полей для заполнения (с чекбоксами)

    - Поля ввода (если статус «В работе»):

        - «Внутреннее решение *» (textarea)

        - «Шаблон ответа клиенту *» (dropdown + textarea)

    - Кнопки действий (в зависимости от статуса)

    **Секция «Вложения»:**

    - Список прикреплённых файлов

    - Для каждого файла: иконка типа, имя, размер, дата загрузки

    - Кнопка скачивания для каждого файла

1. **Система** отображает сформированную карточку обращения пользователю.

2. **Система** устанавливает фокус на основную область карточки для удобства навигации с клавиатуры.

### Фаза 9: Логирование просмотра

1. **Система** создаёт запись в журнале доступа (таблица `appeal_access_log`):

    - `appeal_id` = ID обращения

    - `user_id` = ID пользователя

    - `access_type` = «view»

    - `accessed_at` = текущая дата и время

    - `ip_address` = IP-адрес клиента

    - `session_id` = ID сессии

1. **Система** обновляет счётчик просмотров обращения:

    - UPDATE `appeals` SET `view_count = view_count + 1` WHERE `appeal_id = ?`

### Фаза 10: Ожидание действий пользователя

1. **Пользователь** просматривает данные обращения:

    - Читает содержание обращения

    - Раскрывает/сворачивает секции для детального изучения

    - Просматривает протокол обработки

    - Изучает ответы от смежных подразделений

    - Скачивает вложения (при необходимости)

1. **Пользователь** принимает решение о дальнейших действиях:

    - Принять обращение в работу (UC-ACCEPT-01)

    - Отказаться от обращения (UC-REFUSE-01)

    - Запросить информацию в смежное подразделение (UC-QUERY-01)

    - Приступить к формированию решения и ответа (UC-DECIDE-01)

    - Вернуться к реестру обращений (UC-LIST-01)

1. Сценарий завершается.

## 4. Постусловия

### Успешное завершение:

- Карточка обращения полностью загружена и отображена пользователю

- Все доступные данные из основной базы и интеграционных систем загружены и представлены

- Протокол обработки отображён с возможностью фильтрации по этапам

- Данные смежных подразделений загружены (или отображены заглушки при недоступности)

- Рассчитаны и отображены динамические показатели (SLA, время обработки)

- Определены и отображены доступные действия для текущего статуса

- Просмотр обращения зафиксирован в журнале доступа

- Счётчик просмотров обращения увеличен на 1

- Пользователь имеет полную информацию для принятия решения о дальнейших действиях

- Система готова к выполнению следующих Use Cases (обработка, запросы, доведение)

### Состояние системы:

- Никакие данные обращения не изменились (режим read-only, кроме счётчика просмотров)

- Запись о просмотре создана в журнале доступа (для аудита)

- Сессия пользователя активна

- Интерфейс готов к взаимодействию с пользователем

## 5. Альтернативные потоки

### A1: Недостаточно прав доступа к обращению

- **Точка ветвления:** Шаг 5

- **Условие:** Пользователь не является ответственным по обращению, не является его создателем, принадлежит к другому подразделению и не имеет роли супервизора

- **Действия:**

    1. Система проверяет права доступа и обнаруживает несоответствие

    2. Система не загружает данные обращения (шаги 7–21 пропускаются)

    3. Система отображает уведомление: «Недостаточно прав для просмотра обращения №[номер]. Обращение принадлежит другому подразделению или вы не являетесь ответственным сотрудником.»

    4. Система предлагает действия:

        - Кнопка «Вернуться к реестру обращений»

        - Кнопка «Запросить доступ у супервизора» (отправляет запрос с обоснованием)

    1. Система логирует попытку несанкционированного доступа (SECURITY_AUDIT_LOG):

        - Event: UNAUTHORIZED_ACCESS_ATTEMPT

        - User ID, Appeal ID, Timestamp, IP address

    1. Система автоматически перенаправляет пользователя к реестру обращений через 5 секунд (если не выбрано действие)

    2. Сценарий завершается

### A2: Обращение не найдено

- **Точка ветвления:** Шаг 4

- **Условие:** Запрос к базе данных не вернул записи (обращение с указанным ID не существует или было удалено)

- **Действия:**

    1. Система получает пустой результат запроса

    2. Система отображает экран ошибки:

        - Заголовок: «Обращение не найдено»

        - Сообщение: «Обращение с номером [номер] не существует или было удалено из системы.»

        - Иконка: ⚠️ (предупреждение)

    1. Система предлагает действия:

        - Кнопка «Вернуться к реестру обращений»

        - Кнопка «Проверить правильность номера обращения»

    1. Система логирует событие с уровнем WARNING:

        - Тип: APPEAL_NOT_FOUND

        - Appeal ID (из запроса)

        - User ID

        - Timestamp

    1. Пользователь может вернуться к реестру или проверить номер обращения

    2. Сценарий завершается

### A3: Ошибка загрузки данных из смежной системы (HelpDesk, CRM)

- **Точка ветвления:** Шаг 13

- **Условие:** API смежной системы недоступен, возвращает ошибку или превышен таймаут (5 секунд)

- **Действия:**

    1. Система получает ошибку от интеграционного API (таймаут, HTTP 500, connection refused)

    2. Система НЕ прерывает основной процесс загрузки карточки

    3. Система помечает источник данных как недоступный

    4. Система логирует ошибку интеграции с уровнем WARNING:

        - Тип: INTEGRATION_UNAVAILABLE

        - Source system: [название системы]

        - Error: [текст ошибки]

        - Appeal ID

        - Timestamp

    1. Система продолжает загрузку остальных данных (шаги 14–21)

    2. Система отображает карточку обращения с заглушкой в секции недоступной системы:

        - **Для HelpDesk:** «⚠️ Данные HelpDesk временно недоступны. Попробуйте обновить страницу позже.»

        - **Для CRM:** «⚠️ История клиента из CRM временно недоступна.»

    1. Система добавляет кнопку «🔄 Обновить данные» рядом с заглушкой

    2. Пользователь может:

        - Продолжить работу с доступными данными

        - Нажать «Обновить данные» для повторной попытки загрузки

        - Обновить всю страницу (F5)

    1. Основной поток продолжается с отображением частично загруженной карточки

### A4: Обращение находится в архиве

- **Точка ветвления:** Шаг 18 (определение доступных действий)

- **Условие:** Статус обращения = «Архив» (обращение закрыто более 90 дней назад и перемещено в архив)

- **Действия:**

    1. Система загружает все данные обращения (шаги 7–17 выполняются полностью)

    2. Система определяет, что обращение в архиве

    3. Система отображает карточку обращения в режиме «только для чтения»:

        - В заголовке добавляется badge «АРХИВ» (серый цвет)

        - Все поля ввода заблокированы (disabled)

        - Кнопки действий скрыты или заблокированы

        - Добавляется информационное сообщение: «ℹ️ Обращение находится в архиве. Редактирование невозможно. Для повторного открытия обратитесь к супервизору.»

    1. Система отображает все исторические данные:

        - Полный протокол обработки

        - Все ответы смежных подразделений

        - Финальное решение и ответ клиенту

    1. Пользователь может:

        - Просматривать все данные

        - Скачивать вложения

        - Экспортировать карточку в PDF (если доступна функция)

        - Запросить повторное открытие обращения (если есть права)

    1. Основной поток продолжается в режиме read-only

### A5: Обращение заблокировано для редактирования

- **Точка ветвления:** Шаг 18

- **Условие:** Поле `is_blocked = true` (обращение ожидает ответа от смежного подразделения с установленным флагом блокировки)

- **Действия:**

    1. Система загружает все данные обращения

    2. Система обнаруживает флаг блокировки

    3. Система отображает карточку с предупреждающим баннером:

        - «⏳ Обращение ожидает ответа от [название подразделения]. Редактирование временно заблокировано.»

        - Информация о запросах: «Активные запросы: [список запросов с плановыми сроками]»

    1. Система блокирует поля «Внутреннее решение» и «Шаблон ответа клиенту»

    2. Система отображает доступные действия:

        - «Просмотреть историю запросов»

        - «Снять блокировку» (если есть права супервизора)

        - «Добавить комментарий»

    1. Пользователь может просматривать данные, но не может довести ответ до получения ответа на запрос

    2. Основной поток продолжается с ограниченным функционалом

### A6: Множественные вложения большого размера (медленная загрузка)

- **Точка ветвления:** Шаг 15

- **Условие:** Обращение содержит более 10 вложений общим размером > 100 МБ

- **Действия:**

    1. Система загружает метаданные вложений (имена, размеры, типы) быстро

    2. Система НЕ генерирует signed URLs сразу для всех файлов (ленивая загрузка)

    3. Система отображает карточку обращения с секцией «Вложения»:

        - Список файлов с метаданными

        - Кнопка «Скачать» для каждого файла (неактивна до клика)

    1. Когда пользователь кликает «Скачать» для конкретного файла:

        - Система генерирует signed URL для этого файла (запрос к storage API)

        - Система инициирует скачивание

    1. Система отображает прогресс-индикатор для генерации URL (если занимает > 1 секунды)

    2. Основной поток продолжается с ленивой загрузкой вложений

### A7: Пользователь раскрывает секцию «Протокол обработки» с фильтрацией

- **Точка ветвления:** Шаг 24 (взаимодействие с интерфейсом)

- **Условие:** Пользователь активирует фильтр по этапам в протоколе обработки

- **Действия:**

    1. Пользователь выбирает этап из dropdown (например, «Обработка»)

    2. Система фильтрует записи протокола:

        - WHERE `stage = 'processing'`

    1. Система обновляет отображение протокола только с записями выбранного этапа

    2. Система отображает информацию: «Показано [N] записей этапа "[название этапа]"»

    3. Система сохраняет состояние фильтра в session storage (для восстановления при возврате)

    4. Пользователь может сбросить фильтр (выбрать «Все»)

    5. Основной поток продолжается с отфильтрованным протоколом

### A8: Просмотр карточки закрытого обращения с оценкой клиента

- **Точка ветвления:** Шаг 19 (формирование карточки)

- **Условие:** Статус обращения = «Закрыто», получена оценка удовлетворённости от клиента

- **Действия:**

    1. Система загружает все данные обращения

    2. Система выполняет запрос к таблице `client_feedback`:

        - SELECT FROM `client_feedback` WHERE `appeal_id = ?`

    1. Если найдена оценка:

        - Система добавляет секцию «Оценка клиента» в карточку:

            - Рейтинг (1–5 звёзд или NPS)

            - Комментарий клиента (если есть)

            - Дата получения оценки

    1. Система отображает карточку с дополнительной секцией

    2. Основной поток продолжается с расширенной информацией

## 6. Исключительные ситуации (Exception Flows)

### E1: Критическая ошибка базы данных при загрузке обращения

- **Точка возникновения:** Шаг 7

- **Условие:** Потеря соединения с базой данных, таймаут запроса, ошибка SQL

- **Действия:**

    1. Система прерывает выполнение операции загрузки

    2. Система отображает экран критической ошибки:

        - Заголовок: «Не удалось загрузить обращение»

        - Сообщение: «Сервис временно недоступен. Попробуйте обновить страницу через несколько секунд.»

        - Иконка: ⚠️ (критическая ошибка)

    1. Система предлагает действия:

        - Кнопка «🔄 ОБНОВИТЬ» (повторяет попытку загрузки)

        - Кнопка «← ВЕРНУТЬСЯ К РЕЕСТРУ»

    1. Система логирует ошибку с уровнем CRITICAL:

        - Тип: DATABASE_CONNECTION_FAILED

        - Operation: LOAD_APPEAL

        - Appeal ID

        - User ID

        - Error message

        - Timestamp

    1. Система отправляет алерт команде DevOps

    2. Пользователь может повторить попытку или вернуться к реестру

    3. Сценарий завершается

### E2: Таймаут запроса к интеграционному API (более 5 секунд)

- **Точка возникновения:** Шаг 13

- **Условие:** Интеграционный API не отвечает в течение установленного таймаута (5 секунд)

- **Действия:**

    1. Система прерывает ожидание ответа от API

    2. Система НЕ прерывает загрузку карточки (продолжает работу с основными данными)

    3. Система логирует событие с уровнем WARNING:

        - Тип: INTEGRATION_TIMEOUT

        - Source system: [название API]

        - Timeout duration: 5000ms

        - Appeal ID

    1. Система отображает карточку с предупреждающим сообщением в секции интеграции:

        - «⏱️ Загрузка данных [название системы] занимает больше времени, чем обычно. Данные могут появиться позже или быть недоступными.»

    1. Система продолжает попытку загрузки в фоновом режиме (асинхронно)

    2. Если данные загружаются позже (через 10–30 секунд):

        - Система автоматически обновляет соответствующую секцию карточки без перезагрузки страницы

        - Отображает toast-уведомление: «✅ Данные [название системы] загружены»

    1. Если данные не загружаются через 30 секунд:

        - Система отображает окончательную заглушку с кнопкой ручного обновления

    1. Основной поток продолжается с частично загруженными данными

### E3: Конкурентное изменение обращения другим пользователем

- **Точка возникновения:** Между шагом 7 и шагом 20 (пользователь долго просматривает карточку)

- **Условие:** Другой пользователь (или процесс) изменил данные обращения во время просмотра

- **Действия:**

    1. Система периодически проверяет актуальность данных (long polling или WebSocket)

    2. Система обнаруживает изменение поля `last_modified_at` обращения

    3. Система отображает toast-уведомление в верхней части карточки:

        - «⚠️ Обращение было изменено другим пользователем ([ФИО]). Рекомендуется обновить данные.»

        - Кнопка «ОБНОВИТЬ СЕЙЧАС»

        - Кнопка «ЗАКРЫТЬ» (игнорировать уведомление)

    1. Если пользователь нажимает «ОБНОВИТЬ СЕЙЧАС»:

        - Система перезагружает данные обращения (повтор шагов 7–20)

        - Сохраняет состояние интерфейса (развёрнутые/свёрнутые секции, фильтры)

        - Обновляет карточку с актуальными данными

    1. Если пользователь игнорирует уведомление:

        - Система продолжает отображать устаревшие данные

        - При попытке выполнить действие (например, «Принять в работу») система повторно проверит актуальность и может заблокировать действие

    1. Основной поток продолжается с уведомлением об изменениях

### E4: Ошибка генерации signed URL для вложений

- **Точка возникновения:** Шаг 15

- **Условие:** Сервис хранилища файлов (S3, Azure Blob) недоступен или возвращает ошибку

- **Действия:**

    1. Система пытается сгенерировать signed URLs для вложений

    2. Система получает ошибку от storage API

    3. Система логирует ошибку с уровнем ERROR:

        - Тип: STORAGE_SERVICE_ERROR

        - Appeal ID

        - File names

        - Error message

    1. Система отображает карточку обращения с предупреждением в секции «Вложения»:

        - «⚠️ Временно невозможно получить доступ к вложениям. Служба хранилища недоступна. Попробуйте позже.»

    1. Система отображает список файлов с метаданными, но без возможности скачивания

    2. Система добавляет кнопку «Повторить попытку»

    3. Пользователь может продолжить работу с остальными данными обращения

    4. Основной поток продолжается без доступа к вложениям

### E5: Недействительная сессия пользователя в момент загрузки

- **Точка возникновения:** Шаг 5 (проверка прав доступа)

- **Условие:** Сессия пользователя истекла или была прервана (401 Unauthorized)

- **Действия:**

    1. Система обнаруживает недействительную сессию при проверке прав

    2. Система прерывает загрузку карточки

    3. Система сохраняет URL карточки в session storage (для восстановления после авторизации)

    4. Система перенаправляет пользователя на страницу авторизации

    5. Система отображает сообщение: «Ваша сессия истекла. Пожалуйста, войдите в систему повторно.»

    6. После успешной авторизации:

        - Система автоматически перенаправляет пользователя к сохранённому URL

        - Карточка обращения загружается заново (повтор основного потока с шага 1)

    1. Сценарий возобновляется после авторизации

### E6: Повреждённые данные в базе (NULL в обязательных полях)

- **Точка возникновения:** Шаг 7

- **Условие:** Запрос вернул запись обращения, но критически важные поля содержат NULL (например, `appeal_number`, `status`, `applicant_full_name`)

- **Действия:**

    1. Система загружает запись обращения

    2. Система выполняет валидацию целостности данных:

        - Проверяет наличие обязательных полей

    1. Система обнаруживает повреждённые данные

    2. Система отображает предупреждающее сообщение:

        - «⚠️ Обнаружены повреждённые данные обращения. Некоторая информация может отображаться некорректно.»

    1. Система заполняет отсутствующие поля значениями по умолчанию:

        - NULL → «[не указано]»

        - Пустая строка → «[пусто]»

    1. Система отображает карточку с заполненными заглушками

    2. Система логирует инцидент с уровнем CRITICAL:

        - Тип: DATA_INTEGRITY_VIOLATION

        - Appeal ID

        - Missing fields: [список полей]

        - Timestamp

    1. Система отправляет алерт команде поддержки БД

    2. Пользователь видит карточку с предупреждением и может продолжить работу

    3. Основной поток продолжается с частично повреждёнными данными

## 7. Специальные требования

### 7.1. Производительность

- **NFR-PERF-01:** Время полной загрузки карточки обращения не должно превышать 2 секунд при нормальной нагрузке и доступности всех интеграций

- **NFR-PERF-02:** Время загрузки основных данных обращения (шаги 7–8) не должно превышать 500 миллисекунд

- **NFR-PERF-03:** Время загрузки протокола обработки (шаг 9) не должно превышать 300 миллисекунд при количестве записей до 100

- **NFR-PERF-04:** Таймаут запросов к интеграционным API: 5 секунд (с автоматическим fallback на заглушки)

- **NFR-PERF-05:** Система должна поддерживать одновременный просмотр карточек обращений 200+ пользователями без деградации производительности

- **NFR-PERF-06:** Прокрутка длинного протокола обработки (100+ записей) должна быть плавной (60 FPS) с использованием виртуализации списков

### 7.2. Безопасность

- **NFR-SEC-01:** Проверка прав доступа к обращению должна выполняться на уровне API (backend), а не только на уровне UI

- **NFR-SEC-02:** Система должна применять row-level security для ограничения доступа к обращениям на уровне базы данных

- **NFR-SEC-03:** Все просмотры обращений должны логироваться в журнале доступа с фиксацией:

    - ID пользователя

    - ID обращения

    - Даты и времени просмотра (с точностью до миллисекунд)

    - IP-адреса клиента

    - Session ID

    - User agent

- **NFR-SEC-04:** Персональные данные клиента (телефон, адрес, email) должны быть доступны только пользователям с правом `pii.view`

- **NFR-SEC-05:** Signed URLs для скачивания вложений должны иметь ограниченный срок действия (15 минут)

- **NFR-SEC-06:** Все данные должны передаваться по зашифрованному HTTPS-соединению

- **NFR-SEC-07:** Система должна блокировать попытки SQL injection через параметры URL (валидация формата appeal_id)

### 7.3. Надёжность

- **NFR-REL-01:** Система должна корректно работать при недоступности интеграционных API (graceful degradation с отображением заглушек)

- **NFR-REL-02:** При ошибке загрузки части данных система должна отображать доступную информацию, а не полностью блокировать интерфейс

- **NFR-REL-03:** Система должна поддерживать автоматическое восстановление после временных сбоев БД (retry logic с экспоненциальным back-off)

- **NFR-REL-04:** Кэширование неизменяемых данных (закрытые обращения старше 7 дней) для снижения нагрузки на БД

### 7.4. Доступность (Accessibility)

- **NFR-AVAIL-01:** Интерфейс карточки обращения должен соответствовать стандарту WCAG 2.1 уровня AA

- **NFR-AVAIL-02:** Все интерактивные элементы должны быть доступны с клавиатуры:

    - Tab: переход между элементами

    - Enter/Space: активация кнопок и раскрытие аккордеонов

    - Escape: закрытие модальных окон

    - Arrow keys: навигация по протоколу обработки

- **NFR-AVAIL-03:** Все секции и поля должны иметь ARIA-метки для screen readers:

    - `aria-label` для кнопок без текста

    - `aria-expanded` для сворачиваемых секций

    - `aria-live="polite"` для динамически обновляемых данных

    - `role="region"` для логических секций

- **NFR-AVAIL-04:** Цветовые индикаторы (SLA, статусы) должны сопровождаться текстовыми метками для пользователей с нарушениями цветовосприятия

- **NFR-AVAIL-05:** Поддержка масштабирования интерфейса до 200% без потери функциональности

### 7.5. Удобство использования (Usability)

- **NFR-UX-01:** Система должна сохранять состояние интерфейса при переходах (развёрнутые/свёрнутые секции, фильтры протокола) в session storage

- **NFR-UX-02:** При возврате к карточке из другого UC (например, после отказа от обращения) система должна восстанавливать предыдущее состояние интерфейса

- **NFR-UX-03:** Длинные текстовые поля (содержание обращения, ответы) должны иметь функцию «Развернуть/Свернуть» для экономии экранного пространства

- **NFR-UX-04:** Система должна отображать индикатор загрузки (skeleton screen или spinner) во время загрузки данных

- **NFR-UX-05:** Кнопка «Назад к реестру» должна возвращать пользователя к реестру с сохранением предыдущих фильтров и позиции прокрутки

### 7.6. Интеграции

- **NFR-INT-01:** Система должна поддерживать интеграцию минимум с 3 внешними системами (HelpDesk, CRM, Scoring)

- **NFR-INT-02:** Формат обмена данными с интеграционными API: JSON

- **NFR-INT-03:** Все запросы к интеграционным API должны выполняться параллельно (Promise.all) для минимизации времени загрузки

- **NFR-INT-04:** Система должна кэшировать ответы от интеграционных API на стороне сервера (TTL: 5 минут) для снижения нагрузки

### 7.7. Аудит и мониторинг

- **NFR-AUDIT-01:** Все действия просмотра обращений должны фиксироваться в журнале доступа (`appeal_access_log`)

- **NFR-AUDIT-02:** Система должна отслеживать метрики производительности:

    - Среднее время загрузки карточки

    - Процент успешных загрузок

    - Процент ошибок интеграций

    - Топ-10 медленных запросов

- **NFR-AUDIT-03:** Система должна отправлять алерты при превышении пороговых значений:

    - Время загрузки > 5 секунд: WARNING

    - Ошибки БД: CRITICAL

    - Недоступность интеграций > 10 минут: HIGH

## 8. Бизнес-правила

### BR-VIEW-01: Права доступа к обращениям

**Правило:** Карточка обращения отображается только пользователям с правами доступа:

- **Ответственный сотрудник:** может просматривать обращения, где он является ответственным (`responsible_user_id = current_user_id`) ИЛИ обращения своего подразделения (`department_id = user.department_id`)

- **Супервизор подразделения:** может просматривать все обращения своего подразделения

- **Администратор системы:** может просматривать все обращения без ограничений

- **Аудитор:** может просматривать все обращения в режиме read-only

**Обоснование:** Соблюдение принципа минимальных привилегий и защита персональных данных клиентов.

**Исключения:** Нет.

---

### BR-VIEW-02: Режим «только для чтения» для архивных обращений

**Правило:** Обращения в статусе «Архив» (закрыты более 90 дней) отображаются только в режиме read-only:

- Все поля ввода заблокированы

- Кнопки действий («Принять в работу», «Довести ответ») скрыты

- Доступны только функции просмотра и экспорта

- В заголовке отображается badge «АРХИВ»

**Обоснование:** Предотвращение случайных изменений завершённых обращений и обеспечение целостности исторических данных.

**Исключения:** Супервизор или администратор могут инициировать повторное открытие архивного обращения через специальный UC (с обязательным обоснованием).

---

### BR-VIEW-03: Graceful degradation при ошибках интеграций

**Правило:** При недоступности интеграционных систем (HelpDesk, CRM, Scoring) основная функциональность просмотра обращения НЕ должна блокироваться:

- Система отображает карточку с доступными данными из основной БД

- В секциях недоступных систем отображаются заглушки с предупреждением

- Пользователь может продолжить работу с обращением

- Система предоставляет возможность ручного обновления данных

**Обоснование:** Обеспечение непрерывности бизнес-процесса даже при технических сбоях смежных систем.

**Исключения:** Если основная база данных недоступна, отображается экран критической ошибки (E1).

---

### BR-VIEW-04: Загрузка и отображение протокола обработки

**Правило:** Протокол обработки (audit log) загружается полностью для данного обращения и отображается в хронологическом порядке (от новых к старым):

- Все записи протокола видны пользователю (без ограничений по ролям)

- Доступна фильтрация по этапам процесса (регистрация, аудит, обработка, доведение)

- Записи содержат: дата/время, этап, действие, исполнитель, комментарий

- Максимальное количество отображаемых записей на странице: 50 (с пагинацией)

**Обоснование:** Прозрачность процесса обработки, возможность отследить историю действий для аудита и расследования жалоб.

**Исключения:** Нет.

---

### BR-VIEW-05: Видимость полей решения и ответа

**Правило:** Поля «Внутреннее решение» и «Шаблон ответа клиенту» отображаются в карточке только при определённых статусах:

- **Статус «В работе»:** поля видны, доступны для редактирования (если пользователь является ответственным)

- **Статус «На доведении»:** поля видны, доступны только для чтения (решение принято, ожидается доставка)

- **Статус «Закрыто» / «Архив»:** поля видны, доступны только для чтения (отображается финальная версия)

- **Другие статусы:** поля скрыты или отображаются как «не заполнено»

**Обоснование:** Поэтапное заполнение данных в соответствии с BPMN-процессом.

**Исключения:** Супервизор может просматривать черновики решений на любом этапе (если существуют).

---

### BR-VIEW-06: Индикация SLA

**Правило:** Система отображает текущее состояние SLA с цветовой индикацией:

- **Зеленый:** использовано < 50% времени до дедлайна

- **Желтый:** использовано 50–80% времени

- **Красный:** использовано > 80% времени или дедлайн просрочен

**Расчёт:**

- `sla_usage_percent = ((current_time - sla_started_at) / (sla_deadline - sla_started_at)) * 100`

**Отображение:**

- Прогресс-бар с процентом

- Текстовая метка: «Осталось [N] часов» или «Просрочено на [N] часов»

- Цветовой badge в заголовке карточки

**Обоснование:** Быстрая визуальная оценка срочности обработки обращения.

**Исключения:** Для обращений в статусе «Закрыто» / «Архив» отображается финальный статус SLA («Соблюдён» или «Просрочен»).

---

### BR-VIEW-07: Счётчик просмотров обращения

**Правило:** Каждый просмотр карточки обращения увеличивает счётчик `view_count` на 1:

- Увеличение происходит при успешной загрузке карточки (после шага 20)

- Повторные просмотры одним и тем же пользователем также учитываются

- Счётчик отображается в секции «Данные обращения» (опционально)

**Использование метрики:**

- Аналитика популярности обращений

- Выявление обращений, требующих повышенного внимания

- Контроль загруженности сотрудников

**Обоснование:** Сбор статистики для улучшения процессов.

**Исключения:** Просмотры системными процессами (автоматизация) не учитываются.

---

### BR-VIEW-08: Логирование доступа к обращениям

**Правило:** Каждый просмотр обращения фиксируется в журнале доступа (`appeal_access_log`) с записью:

- `appeal_id` — ID обращения

- `user_id` — ID пользователя

- `access_type` — тип доступа («view»)

- `accessed_at` — дата и время (точность до миллисекунд)

- `ip_address` — IP-адрес клиента

- `session_id` — ID сессии

- `user_agent` — браузер и ОС

**Срок хранения:** 3 года (для соответствия требованиям аудита).

**Обоснование:** Соблюдение требований информационной безопасности, возможность расследования инцидентов доступа к персональным данным.

**Исключения:** Нет.

---

### BR-VIEW-09: Кэширование данных закрытых обращений

**Правило:** Данные обращений в статусе «Закрыто» (старше 7 дней) могут кэшироваться на сервере для повышения производительности:

- TTL кэша: 24 часа

- Кэшируются только неизменяемые данные (основные поля, протокол, ответы)

- Динамические данные (счётчик просмотров, комментарии) загружаются напрямую

**Обоснование:** Снижение нагрузки на БД для часто просматриваемых закрытых обращений.

**Исключения:** Обращения в статусах «В работе», «На доведении» НЕ кэшируются (данные могут изменяться).

---

### BR-VIEW-10: Отображение данных смежных подразделений

**Правило:** Данные от смежных подразделений отображаются в отдельных подсекциях секции «История обработки смежными подразделениями»:

- **HelpDesk:** статус тикета, дата решения, текст решения

- **Запросы в подразделения:** список всех запросов с номерами, статусами, текстами запросов и ответов

- **CRM:** история предыдущих обращений клиента (опционально)

**Порядок отображения:**

1. Запросы в подразделения (если есть)

2. HelpDesk (если есть тикет)

3. CRM (если доступны данные)

**Обоснование:** Структурированное представление информации от разных источников для удобства анализа.

**Исключения:** Если данные от системы недоступны, отображается заглушка с предупреждением (A3).

### 10.4. Метрики для мониторинга

- **Среднее время загрузки карточки** (target: < 2 сек)

- **Процент успешных загрузок** (target: > 99%)

- **Процент ошибок интеграций** (HelpDesk, CRM) — для выявления проблем

- **Топ-10 самых просматриваемых обращений** (по счётчику `view_count`)

- **Среднее количество просмотров на обращение** (для анализа эффективности)

- **Процент просмотров с ошибками доступа** (для контроля безопасности)
