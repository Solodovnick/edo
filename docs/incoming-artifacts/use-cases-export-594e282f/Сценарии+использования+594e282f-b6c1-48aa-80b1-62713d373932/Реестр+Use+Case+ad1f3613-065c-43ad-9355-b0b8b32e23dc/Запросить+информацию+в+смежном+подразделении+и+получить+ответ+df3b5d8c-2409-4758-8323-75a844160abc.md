---

## 1. Паспорт Use Case

- **Идентификатор:** UC-QUERY-01

- **Название:** Запрос информации в смежное подразделение для обработки обращения клиента

- **Актор:** Ответственный сотрудник ДУКО (Группа обработки обращений клиентов)

- **Цель:** Получить экспертную информацию, консультацию или решение от смежного подразделения (юридический отдел, IT-поддержка, служба безопасности, департамент продуктов и др.) для корректной обработки обращения клиента и формирования качественного ответа

- **Триггер:** В процессе обработки обращения ответственный сотрудник выявил необходимость в специализированной экспертизе или информации, которой не обладает

- **Частота использования:** 10–30 раз в день на одного сотрудника

- **Приоритет:** Высокий (критично для соблюдения SLA обращения и качества ответа клиенту)

- **Уровень:** Подзадача (Subfunction Level) в рамках UC-02 «Обработка обращения ответственным»

---

## 2. Предусловия

- Обращение существует в системе с уникальным идентификатором

- Текущий статус обращения — «В работе»

- Пользователь является назначенным ответственным сотрудником по данному обращению

- Пользователь аутентифицирован в системе

- Пользователь имеет роль «Ответственный сотрудник ДУКО» с правами на создание межподразделенческих запросов

- Карточка обращения открыта и отображена пользователю (выполнен UC-01 «Просмотр детализации обращения»)

- База данных доступна и транзакции выполняются корректно

- В системе существует справочник подразделений с активными записями (таблица `departments`)

- Интеграционные API смежных подразделений доступны или существует механизм ручной обработки запросов

---

## 3. Основной поток (Main Flow)

### Фаза 1: Инициация запроса в смежное подразделение

1. **Пользователь** находится в карточке обращения и активирует действие «Направить запрос в подразделение».

2. **Система** проверяет текущий статус обращения.

3. **Система** подтверждает, что статус обращения — «В работе».

4. **Система** проверяет, что пользователь является назначенным ответственным по обращению.

5. **Система** отображает диалоговое окно «Создание запроса в смежное подразделение».

### Фаза 2: Загрузка справочных данных

1. **Система** загружает из базы данных справочник подразделений (таблица `departments`, фильтр `is_active = true`, `accepts_queries = true`).

2. **Система** отображает список доступных подразделений с категориями:

    - «Юридический отдел»

    - «IT-поддержка / HelpDesk»

    - «Служба безопасности»

    - «Департамент продуктов»

    - «Департамент розничного бизнеса»

    - «Департамент корпоративного бизнеса»

    - «Бухгалтерия»

    - «Служба комплаенса»

    - «Другое»

1. **Система** для каждого подразделения отображает:

    - Название подразделения

    - Типовое время ответа (SLA подразделения)

    - Статус доступности (доступно / недоступно / ограниченная работа)

### Фаза 3: Заполнение формы запроса

1. **Пользователь** выбирает целевое подразделение из списка.

2. **Система** подгружает шаблоны типовых запросов для выбранного подразделения (если существуют).

3. **Система** отображает форму запроса с полями:

    - Подразделение (выбрано на шаге 9, read-only)

    - Категория запроса (dropdown: выбор из типовых категорий подразделения)

    - Приоритет (radio buttons: «Обычный», «Высокий», «Критический»)

    - Плановый срок ответа (автоматически рассчитывается на основе приоритета и SLA подразделения)

    - Тема запроса (текстовое поле, обязательное, максимум 200 символов)

    - Текст запроса (textarea, обязательное, минимум 20 символов)

    - Вложения (опционально, кнопка «Прикрепить файлы»)

1. **Пользователь** выбирает категорию запроса из dropdown.

2. **Пользователь** выбирает приоритет запроса (по умолчанию: «Обычный»).

3. **Система** автоматически рассчитывает плановый срок ответа:

    - Обычный приоритет: текущая дата + SLA подразделения (например, 3 рабочих дня)

    - Высокий приоритет: текущая дата + (SLA подразделения / 2)

    - Критический приоритет: текущая дата + 1 рабочий день

1. **Пользователь** вводит тему запроса.

2. **Пользователь** вводит текст запроса с описанием вопроса или запрашиваемой информации.

3. **Пользователь** опционально прикрепляет файлы (документы, скриншоты, логи).

4. **Пользователь** подтверждает создание запроса (активирует кнопку отправки запроса).

### Фаза 4: Валидация введённых данных

1. **Система** проверяет, что выбрано подразделение.

2. **Система** проверяет, что заполнено поле «Тема запроса» (длина >= 5 символов, <= 200 символов).

3. **Система** проверяет, что заполнено поле «Текст запроса» (длина >= 20 символов).

4. **Система** проверяет доступность выбранного подразделения (статус != «недоступно»).

5. **Система** проверяет размер и формат прикреплённых файлов (если есть):

    - Максимальный размер одного файла: 10 МБ

    - Общий размер всех файлов: 50 МБ

    - Допустимые форматы: PDF, DOCX, XLSX, PNG, JPG, TXT

### Фаза 5: Создание записи запроса

1. **Система** начинает транзакцию базы данных.

2. **Система** генерирует уникальный номер запроса (формат: QRY-YYYYMMDD-XXXXX, где XXXXX — автоинкремент).

3. **Система** создаёт запись в таблице `inter_department_queries`:

    - `query_id` = уникальный ID (UUID)

    - `query_number` = сгенерированный номер (шаг 25)

    - `appeal_id` = ID текущего обращения

    - `source_department_id` = ID подразделения пользователя (ДУКО)

    - `target_department_id` = ID выбранного подразделения

    - `category` = выбранная категория запроса

    - `priority` = выбранный приоритет

    - `subject` = тема запроса

    - `query_text` = текст запроса

    - `created_by_user_id` = ID текущего пользователя

    - `created_at` = текущая дата и время

    - `planned_response_date` = рассчитанный срок ответа (шаг 14)

    - `status` = «Отправлен»

    - `is_overdue` = false

1. **Система** сохраняет прикреплённые файлы в хранилище (таблица `query_attachments`):

    - Для каждого файла:

        - `attachment_id` = UUID

        - `query_id` = ID созданного запроса

        - `file_name` = имя файла

        - `file_size` = размер в байтах

        - `file_path` = путь к файлу в хранилище

        - `uploaded_at` = текущая дата и время

### Фаза 6: Определение ответственного в целевом подразделении

1. **Система** запрашивает данные о распределении нагрузки в целевом подразделении (таблица `department_workload`).

2. **Система** применяет алгоритм назначения ответственного:

    - Вариант A (автоматическое назначение): выбирает сотрудника с минимальной текущей загрузкой (количество активных запросов)

    - Вариант B (ручное назначение): направляет запрос координатору подразделения для ручного распределения

    - Выбор варианта определяется настройкой подразделения (`auto_assign_queries`)

1. **Система** обновляет запись запроса:

    - `assigned_to_user_id` = ID назначенного сотрудника (если автоназначение) или NULL (если ручное)

    - `assigned_at` = текущая дата и время (если автоназначение)

### Фаза 7: Отправка уведомлений

1. **Система** создаёт уведомление для назначенного сотрудника целевого подразделения (таблица `notifications`):

    - `recipient_user_id` = ID назначенного сотрудника

    - `notification_type` = «query_received»

    - `query_id` = ID созданного запроса

    - `message` = «Получен запрос №[номер] от ДУКО. Обращение №[номер обращения]. Тема: [тема]. Срок ответа: [дата].»

    - `created_at` = текущая дата и время

    - `is_read` = false

    - `priority` = приоритет запроса (обычный / высокий / критический)

1. **Система** отправляет push-уведомление назначенному сотруднику (через WebSocket).

2. **Система** отправляет email-уведомление (если включена настройка `email_notifications_enabled`).

3. **Система** проверяет наличие интеграции с системой целевого подразделения:

    - Если интеграция существует (API endpoint активен):

        - Отправляет данные запроса через API в систему подразделения

        - Получает подтверждение приёма запроса

        - Сохраняет `external_system_ticket_id` в запись запроса

    - Если интеграции нет:

        - Использует только внутренние уведомления

### Фаза 8: Фиксация в протоколе обращения

1. **Система** создаёт запись в протоколе обработки обращения (таблица `appeal_audit_log`):

    - `appeal_id` = ID текущего обращения

    - `action_type` = «query_sent»

    - `action_description` = «Направлен запрос №[номер] в [название подразделения]»

    - `performed_by_user_id` = ID текущего пользователя

    - `performed_at` = текущая дата и время

    - `stage` = «processing»

    - `metadata` = JSON с данными запроса (query_id, department, priority, subject)

### Фаза 9: Обновление статуса обращения

1. **Система** проверяет, установлен ли флаг «Блокировать обработку до получения ответа» (checkbox в форме запроса, по умолчанию: true).

2. **Система** обновляет запись обращения:

    - Если флаг установлен:

        - `status` = «Ожидает ответа от смежного отдела»

        - `substatus` = название целевого подразделения

        - `is_blocked` = true

    - Если флаг не установлен:

        - Статус обращения не изменяется (остаётся «В работе»)

        - Добавляется запись в массив `pending_queries` (JSON array с ID активных запросов)

### Фаза 10: Завершение создания запроса

1. **Система** фиксирует транзакцию в базе данных (COMMIT).

2. **Система** закрывает диалоговое окно создания запроса.

3. **Система** отображает уведомление пользователю: «Запрос №[номер] успешно отправлен в [название подразделения]. Плановый срок ответа: [дата].»

4. **Система** обновляет карточку обращения:

    - В блоке «История запросов в подразделения» добавляется новая запись с номером запроса, датой отправки, статусом «Отправлен», плановым сроком

    - Обновляется блок «Следующее действие»: «Ожидание ответа на запрос №[номер] от [подразделение]»

1. **Система** остаётся на странице карточки обращения.

---

### Фаза 11: Получение ответа от подразделения (асинхронный процесс)

1. **Система** периодически проверяет статусы активных запросов (scheduled job, каждые 5 минут).

2. **Система** получает ответ от подразделения одним из способов:

    - **Вариант A (через API):** Интегрированная система подразделения отправляет webhook с данными ответа

    - **Вариант B (ручной ввод):** Сотрудник подразделения вводит ответ в форму системы ДУКО

    - **Вариант C (через email):** Система парсит email-ответ и автоматически создаёт запись ответа

1. **Система** валидирует полученные данные ответа:

    - Проверяет наличие обязательных полей: `query_id`, `response_text`, `responded_by_user_id`

    - Проверяет, что запрос существует и имеет статус «Отправлен» или «В обработке»

1. **Система** начинает транзакцию базы данных.

2. **Система** обновляет запись запроса (таблица `inter_department_queries`):

    - `status` = «Получен ответ»

    - `response_text` = текст ответа от подразделения

    - `responded_by_user_id` = ID сотрудника, предоставившего ответ

    - `responded_at` = дата и время получения ответа

    - `actual_response_duration` = `responded_at - created_at` (в часах)

    - `is_overdue` = true (если `responded_at > planned_response_date`) или false

1. **Система** сохраняет прикреплённые к ответу файлы (если есть) в таблицу `query_response_attachments`.

### Фаза 12: Уведомление ответственного сотрудника ДУКО

1. **Система** создаёт уведомление для ответственного сотрудника ДУКО (таблица `notifications`):

    - `recipient_user_id` = ID создателя запроса (`created_by_user_id`)

    - `notification_type` = «query_answered»

    - `query_id` = ID запроса

    - `message` = «Получен ответ на запрос №[номер] от [название подразделения]. Обращение №[номер обращения].»

    - `created_at` = текущая дата и время

    - `is_read` = false

    - `priority` = «high»

1. **Система** отправляет push-уведомление ответственному сотруднику (через WebSocket).

2. **Система** отправляет email-уведомление (если включена настройка).

### Фаза 13: Фиксация ответа в протоколе обращения

1. **Система** создаёт запись в протоколе обработки обращения (таблица `appeal_audit_log`):

    - `appeal_id` = ID обращения

    - `action_type` = «query_response_received»

    - `action_description` = «Получен ответ на запрос №[номер] от [название подразделения]»

    - `performed_by_user_id` = ID сотрудника, предоставившего ответ

    - `performed_at` = дата и время получения ответа

    - `stage` = «processing»

    - `metadata` = JSON с данными ответа (query_id, response_summary, duration)

### Фаза 14: Обновление статуса обращения

1. **Система** проверяет, был ли установлен флаг блокировки обработки (шаг 36).

2. **Система** обновляет запись обращения:

    - Если обращение было заблокировано (`is_blocked = true`):

        - `status` = «В работе» (возврат к обработке)

        - `substatus` = NULL

        - `is_blocked` = false

    - Если обращение не было заблокировано:

        - Удаляет ID запроса из массива `pending_queries`

1. **Система** проверяет наличие других активных запросов для данного обращения:

    - Если существуют другие запросы в статусе «Отправлен» или «В обработке»:

        - Статус обращения остаётся «Ожидает ответа от смежного отдела» (если был установлен)

    - Если это был последний активный запрос:

        - Статус обращения переходит в «В работе»

### Фаза 15: Завершение получения ответа

1. **Система** фиксирует транзакцию в базе данных (COMMIT).

2. **Система** обновляет интерфейс карточки обращения (если она открыта пользователем):

    - В блоке «История запросов в подразделения» обновляется запись с отметкой «Получен ответ», датой ответа

    - Отображается текст ответа (expandable/collapsible секция)

    - Отображаются прикреплённые к ответу файлы (ссылки для скачивания)

    - Обновляется блок «Следующее действие»: «Обработать полученную информацию и продолжить работу с обращением»

## 4. Постусловия

### Успешное завершение (после создания запроса, шаги 1–42):

- В системе создана запись запроса с уникальным номером в статусе «Отправлен»

- Запрос связан с текущим обращением (поле `appeal_id`)

- Назначен ответственный сотрудник в целевом подразделении (или запрос направлен координатору)

- Ответственный сотрудник целевого подразделения получил уведомление о новом запросе

- В протоколе обращения зафиксирована отправка запроса с указанием подразделения, номера запроса, даты и времени

- Статус обращения обновлён на «Ожидает ответа от смежного отдела» (если установлен флаг блокировки) или остаётся «В работе»

- Прикреплённые файлы сохранены в хранилище и связаны с запросом

- Рассчитан плановый срок ответа на основе приоритета и SLA подразделения

- В карточке обращения отображается информация о созданном запросе в блоке «История запросов»

- Пользователь может продолжить работу с обращением или ожидать ответа

### Успешное завершение (после получения ответа, шаги 43–57):

- Запись запроса обновлена со статусом «Получен ответ»

- Текст ответа и прикреплённые файлы сохранены в базе данных

- Рассчитана фактическая длительность обработки запроса

- Определён факт соблюдения/нарушения SLA подразделением (флаг `is_overdue`)

- Ответственный сотрудник ДУКО получил уведомление о получении ответа

- В протоколе обращения зафиксировано получение ответа с указанием исполнителя, даты и времени

- Статус обращения возвращён в «В работе» (если был заблокирован)

- В карточке обращения отображается полученный ответ с возможностью просмотра текста и вложений

- Ответственный сотрудник ДУКО может использовать полученную информацию для завершения обработки обращения

### Состояние системы:

- Транзакции базы данных успешно завершены (committed)

- Целостность данных сохранена

- Аудиторский след зафиксирован для всех этапов (отправка запроса, получение ответа)

- Уведомления отправлены заинтересованным лицам

- Метрики SLA межподразделенческого взаимодействия обновлены

- Система готова к продолжению обработки обращения (UC-02)

## 5. Альтернативные потоки

### A1: Пользователь отменяет создание запроса

- **Точка ветвления:** Между шагом 17 и шагом 18

- **Условие:** Пользователь активирует кнопку «Отмена» в диалоговом окне создания запроса

- **Действия:**

    1. Система закрывает диалоговое окно без сохранения данных

    2. Система не выполняет шаги 19–42

    3. Карточка обращения остаётся в исходном состоянии (статус и данные не изменены)

    4. Пользователь может повторить попытку создания запроса или продолжить обработку обращения

    5. Сценарий завершается

### A2: Не заполнено обязательное поле «Текст запроса»

- **Точка ветвления:** Шаг 21

- **Условие:** Поле «Текст запроса» пустое или содержит менее 20 символов

- **Действия:**

    1. Система не закрывает диалоговое окно

    2. Система отображает валидационное сообщение под полем: «Поле "Текст запроса" обязательно для заполнения. Минимальная длина: 20 символов. Текущая длина: [N].»

    3. Система подсвечивает поле «Текст запроса» (индикация ошибки)

    4. Кнопка отправки запроса остаётся неактивной до устранения ошибки

    5. Пользователь корректирует текст запроса

    6. Основной поток возвращается к шагу 21

### A3: Не заполнено обязательное поле «Тема запроса»

- **Точка ветвления:** Шаг 20

- **Условие:** Поле «Тема запроса» пустое или содержит менее 5 символов или более 200 символов

- **Действия:**

    1. Система не закрывает диалоговое окно

    2. Система отображает валидационное сообщение: «Поле "Тема запроса" обязательно для заполнения. Длина: от 5 до 200 символов.»

    3. Система подсвечивает поле (индикация ошибки)

    4. Кнопка отправки запроса остаётся неактивной

    5. Пользователь корректирует тему запроса

    6. Основной поток возвращается к шагу 20

### A4: Выбранное подразделение недоступно

- **Точка ветвления:** Шаг 22

- **Условие:** Статус выбранного подразделения = «недоступно» (например, технические работы, нерабочее время)

- **Действия:**

    1. Система не закрывает диалоговое окно

    2. Система отображает предупреждающее сообщение: «Подразделение "[название]" в настоящее время недоступно. Причина: [текст причины]. Ожидаемое время доступности: [дата/время]. Вы можете выбрать другое подразделение или сохранить запрос как черновик для отправки позже.»

    3. Система предлагает действия:

        - Кнопка «Выбрать другое подразделение» (возврат к шагу 9)

        - Кнопка «Сохранить как черновик» (сохранение формы без отправки)

        - Кнопка «Отмена»

    1. Пользователь выбирает действие

    2. Основной поток продолжается согласно выбранному действию

### A5: Превышен размер или недопустимый формат прикреплённого файла

- **Точка ветвления:** Шаг 23

- **Условие:** Один или несколько файлов превышают лимиты по размеру или имеют недопустимый формат

- **Действия:**

    1. Система не закрывает диалоговое окно

    2. Система отображает валидационное сообщение:

        - «Файл "[имя файла]" превышает максимальный размер (10 МБ). Текущий размер: [размер].»

        - ИЛИ «Файл "[имя файла]" имеет недопустимый формат. Разрешены: PDF, DOCX, XLSX, PNG, JPG, TXT.»

    1. Система подсвечивает проблемный файл в списке прикреплённых

    2. Система предлагает удалить проблемный файл или заменить его

    3. Кнопка отправки запроса остаётся неактивной до устранения ошибки

    4. Пользователь удаляет или заменяет файл

    5. Основной поток возвращается к шагу 23

### A6: Запрос сохранён как черновик

- **Точка ветвления:** После шага 17 (альтернатива шагу 18)

- **Условие:** Пользователь активирует кнопку «Сохранить как черновик» вместо «Отправить запрос»

- **Действия:**

    1. Система валидирует минимальные обязательные поля (подразделение, тема)

    2. Система создаёт запись в таблице `inter_department_queries` со статусом «Черновик»

    3. Система сохраняет введённые данные (тема, текст, вложения)

    4. Система не выполняет шаги 28–41 (назначение ответственного, отправка уведомлений, изменение статуса обращения)

    5. Система отображает уведомление: «Черновик запроса сохранён. Вы можете вернуться к нему позже для завершения и отправки.»

    6. Пользователь может в любой момент открыть черновик, отредактировать и отправить

    7. Сценарий завершается

### A7: Ответ от подразделения просрочен (не получен в срок)

- **Точка ветвления:** Шаг 43 (периодическая проверка)

- **Условие:** Текущая дата > `planned_response_date`, а статус запроса остаётся «Отправлен» или «В обработке»

- **Действия:**

    1. Система обновляет запись запроса: `is_overdue = true`

    2. Система создаёт уведомление для супервизора ДУКО:

        - `notification_type` = «query_overdue»

        - `message` = «Запрос №[номер] в [подразделение] просрочен. Обращение №[номер обращения]. Плановый срок: [дата]. Текущий статус: [статус].»

        - `priority` = «high»

    1. Система отправляет напоминание ответственному сотруднику целевого подразделения (повторное уведомление)

    2. Система отображает индикацию просрочки в карточке обращения (красная метка рядом с запросом)

    3. Система логирует событие просрочки (AUDIT_LOG)

    4. Основной поток продолжается (ожидание ответа продолжается)

    5. Супервизор может инициировать эскалацию или ручное взаимодействие с подразделением

### A8: Автоматическое назначение ответственного невозможно

- **Точка ветвления:** Шаг 29

- **Условие:** В целевом подразделении нет доступных сотрудников (все перегружены, в отпуске, уволены) или отключено автоназначение

- **Действия:**

    1. Система пропускает шаг 30 (не назначает ответственного)

    2. Система устанавливает `assigned_to_user_id = NULL`

    3. Система направляет запрос координатору подразделения (или заместителю руководителя):

        - Создаёт уведомление с типом «query_requires_manual_assignment»

        - Сообщение: «Требуется ручное назначение ответственного на запрос №[номер] от ДУКО.»

    1. Система устанавливает статус запроса = «Ожидает назначения»

    2. Координатор вручную назначает ответственного через интерфейс

    3. После назначения статус меняется на «В обработке»

    4. Основной поток продолжается с шага 31

### A9: Интеграция с системой подразделения недоступна

- **Точка ветвления:** Шаг 34

- **Условие:** API endpoint целевой системы недоступен (таймаут, ошибка соединения, 500 ответ)

- **Действия:**

    1. Система логирует ошибку интеграции с уровнем WARNING:

        - Тип: INTEGRATION_UNAVAILABLE

        - Target system: [название системы подразделения]

        - Query ID

        - Timestamp

    1. Система НЕ прерывает основной процесс (запрос создан в базе ДУКО)

    2. Система полагается только на внутренние уведомления (шаги 31–33)

    3. Система помечает запрос флагом `integration_failed = true`

    4. Система ставит запрос в очередь повторной отправки (retry queue, 3 попытки с интервалом 10 минут)

    5. Если после 3 попыток интеграция недоступна:

        - Создаётся задача для администратора системы

        - Ответственный сотрудник подразделения получает email-уведомление (fallback)

    1. Основной поток продолжается с шага 35

### A10: Пользователь просматривает историю запросов по обращению

- **Точка ветвления:** В любой момент после шага 42

- **Условие:** Пользователь открывает блок «История запросов в подразделения» в карточке обращения

- **Действия:**

    1. Система загружает все запросы, связанные с текущим обращением (SELECT WHERE `appeal_id = ?`)

    2. Система отображает список запросов с полями:

        - Номер запроса

        - Целевое подразделение

        - Дата отправки

        - Статус (Отправлен / В обработке / Получен ответ / Просрочен)

        - Плановый срок ответа

        - Фактический срок ответа (если статус «Получен ответ»)

    1. Пользователь может развернуть любой запрос для просмотра:

        - Темы и текста запроса

        - Прикреплённых файлов

        - Текста ответа (если получен)

        - Прикреплённых к ответу файлов

    1. Пользователь может скачать файлы вложений

    2. Сценарий продолжается параллельно основному потоку

## 6. Исключительные ситуации (Exception Flows)

### E1: Ошибка подключения к базе данных при создании запроса

- **Точка возникновения:** Шаг 24, 26, 27, 35, 37

- **Условие:** Потеря соединения с базой данных или таймаут запроса

- **Действия:**

    1. Система прерывает выполнение операции

    2. Система выполняет ROLLBACK транзакции (если она была начата)

    3. Система отображает уведомление: «Не удалось создать запрос. Ошибка подключения к базе данных. Попробуйте повторить операцию через несколько секунд.»

    4. Система логирует ошибку с уровнем ERROR:

        - Тип: DATABASE_CONNECTION_FAILED

        - Operation: CREATE_QUERY

        - Appeal ID

        - User ID

        - Timestamp

    1. Система отправляет алерт команде DevOps

    2. Система оставляет диалоговое окно открытым с сохранёнными данными (для повторной попытки)

    3. Кнопка «Отправить запрос» остаётся доступной

    4. Пользователь может повторить попытку или отменить операцию

    5. Сценарий завершается

### E2: Ошибка сохранения прикреплённых файлов

- **Точка возникновения:** Шаг 27

- **Условие:** Файловое хранилище недоступно, недостаточно места на диске, ошибка записи

- **Действия:**

    1. Система прерывает выполнение операции

    2. Система выполняет ROLLBACK транзакции (отменяет создание запроса)

    3. Система отображает уведомление: «Не удалось сохранить прикреплённые файлы. Ошибка файлового хранилища. Попробуйте удалить некоторые файлы или повторить попытку позже.»

    4. Система логирует ошибку с уровнем ERROR:

        - Тип: FILE_STORAGE_ERROR

        - File names: [список файлов]

        - File sizes: [размеры]

        - Error message: [текст ошибки]

    1. Система предлагает варианты:

        - Кнопка «Удалить вложения и отправить запрос без них»

        - Кнопка «Повторить попытку»

        - Кнопка «Отмена»

    1. Пользователь выбирает действие

    2. Если выбрано «Удалить вложения»:

        - Система очищает список прикреплённых файлов

        - Основной поток продолжается с шага 24 (без файлов)

    1. Сценарий завершается

### E3: Нарушение целостности данных (конкурентное изменение обращения)

- **Точка возникновения:** Шаг 38 (COMMIT транзакции)

- **Условие:** Другой процесс одновременно изменил запись обращения, возникла блокировка или конфликт версий

- **Действия:**

    1. Система получает ошибку CONCURRENT_MODIFICATION или LOCK_TIMEOUT

    2. Система выполняет ROLLBACK транзакции

    3. Система повторяет попытку выполнения транзакции (retry logic, максимум 3 попытки с интервалом 500ms)

    4. Если после 3 попыток ошибка сохраняется:

        - Система отображает уведомление: «Не удалось создать запрос. Данные обращения были изменены другим пользователем. Обновите карточку обращения и повторите попытку.»

        - Система логирует ошибку с уровнем WARNING

        - Система предлагает действие: «Обновить карточку обращения»

    1. Пользователь обновляет карточку и повторяет создание запроса

    2. Сценарий завершается

### E4: Ошибка отправки уведомлений

- **Точка возникновения:** Шаг 32, 33, 50, 51

- **Условие:** Служба уведомлений (WebSocket, email-service) недоступна или возвращает ошибку

- **Действия:**

    1. Система фиксирует основную транзакцию (COMMIT на шаге 38 или 56 выполняется успешно)

    2. Система логирует ошибку отправки уведомления с уровнем WARNING:

        - Тип: NOTIFICATION_DELIVERY_FAILED

        - Recipient: user_id

        - Notification type: query_received / query_answered

        - Timestamp

    1. Система сохраняет уведомление в очередь повторной отправки (таблица `notification_queue`)

    2. Система планирует повторную попытку отправки через 5 минут (retry scheduler)

    3. Основной поток продолжается нормально (создание запроса или получение ответа считается успешным)

    4. Пользователь получает подтверждение операции

    5. Сценарий завершается успешно

### E5: Ошибка вызова API внешней системы подразделения

- **Точка возникновения:** Шаг 34

- **Условие:** API endpoint возвращает ошибку (400, 401, 403, 500, таймаут)

- **Действия:**

    1. Система логирует ошибку интеграции с уровнем ERROR:

        - Тип: EXTERNAL_API_ERROR

        - Target system: [название системы]

        - HTTP status: [код ответа]

        - Error message: [текст ошибки]

        - Query ID

    1. Система НЕ прерывает основной процесс (запрос создан в ДУКО)

    2. Система помечает запрос флагом `integration_failed = true`

    3. Система использует fallback-механизм:

        - Отправляет email-уведомление координатору подразделения

        - Создаёт задачу в общей очереди подразделения (если существует интерфейс)

    1. Система ставит запрос в очередь повторной отправки (3 попытки с интервалом 10 минут)

    2. Основной поток продолжается с шага 35

    3. Если после 3 попыток ошибка сохраняется:

        - Создаётся алерт для команды поддержки интеграций

        - Координатор подразделения должен вручную зарегистрировать запрос в своей системе

    1. Сценарий продолжается

### E6: Недостаточно прав доступа для создания запроса

- **Точка возникновения:** Шаг 4

- **Условие:** Между открытием карточки и попыткой создать запрос у пользователя были отозваны права на создание межподразделенческих запросов

- **Действия:**

    1. Система проверяет актуальные разрешения пользователя (permissions check)

    2. Система обнаруживает отсутствие прав `query.create`

    3. Система не отображает диалоговое окно создания запроса

    4. Система отображает уведомление: «Недостаточно прав для создания запросов в смежные подразделения. Обратитесь к администратору системы или супервизору.»

    5. Система логирует событие безопасности (SECURITY_AUDIT_LOG):

        - Event: UNAUTHORIZED_ACCESS_ATTEMPT

        - User ID

        - Action: CREATE_QUERY

        - Timestamp

    1. Карточка обращения остаётся в исходном состоянии

    2. Сценарий завершается

### E7: Ошибка парсинга email-ответа от подразделения

- **Точка возникновения:** Шаг 44 (вариант C)

- **Условие:** Автоматический парсер не может извлечь данные ответа из email (неправильный формат, отсутствие обязательных полей)

- **Действия:**

    1. Система логирует ошибку парсинга с уровнем WARNING:

        - Тип: EMAIL_PARSING_FAILED

        - Email subject: [тема письма]

        - Sender: [отправитель]

        - Timestamp

    1. Система сохраняет исходное письмо в таблицу `unparsed_emails` для ручной обработки

    2. Система создаёт задачу для администратора: «Требуется ручная обработка email-ответа на запрос №[номер]»

    3. Система отправляет уведомление отправителю (сотруднику подразделения):

        - «Ваш ответ на запрос №[номер] не был автоматически обработан. Пожалуйста, введите ответ через интерфейс системы или обратитесь в службу поддержки.»

    1. Запрос остаётся в статусе «Отправлен» или «В обработке»

    2. Администратор или сотрудник подразделения вручную вводит ответ (вариант B)

    3. Сценарий продолжается с шага 45

### E8: Получен ответ на несуществующий или уже закрытый запрос

- **Точка возникновения:** Шаг 45

- **Условие:** `query_id` не существует в базе данных или запрос уже имеет статус «Получен ответ» или «Отменён»

- **Действия:**

    1. Система валидирует данные ответа

    2. Система обнаруживает несоответствие

    3. Система отклоняет ответ (не сохраняет в БД)

    4. Система логирует событие с уровнем WARNING:

        - Тип: INVALID_QUERY_RESPONSE

        - Query ID: [ID из ответа]

        - Current status: [статус запроса или NULL]

        - Responded by: [ID сотрудника]

    1. Система отправляет уведомление отправителю ответа:

        - «Ваш ответ не может быть обработан. Запрос №[номер] не найден или уже закрыт. Если вы считаете это ошибкой, обратитесь в службу поддержки.»

    1. Сценарий завершается без изменения данных

## 7. Специальные требования

### 7.1. Производительность

- **NFR-PERF-01:** Время создания запроса (шаги 1–42) не должно превышать 5 секунд при нормальной нагрузке

- **NFR-PERF-02:** Время загрузки справочника подразделений (шаг 6) не должно превышать 1 секунды

- **NFR-PERF-03:** Операция создания запроса должна выполняться в рамках одной атомарной транзакции БД с уровнем изоляции READ COMMITTED

- **NFR-PERF-04:** Система должна поддерживать одновременное создание запросов 50+ пользователями без деградации производительности

- **NFR-PERF-05:** Периодическая проверка статусов запросов (шаг 43) не должна создавать нагрузку >10% CPU на сервере БД

### 7.2. Безопасность

- **NFR-SEC-01:** Все операции с запросами должны логироваться в аудиторском журнале с фиксацией:

    - ID пользователя, выполнившего действие

    - Точной даты и времени (с точностью до миллисекунд)

    - Типа действия (создание запроса, получение ответа)

    - IP-адреса клиента

    - Session ID

- **NFR-SEC-02:** Создание запросов доступно только пользователям с ролью «Ответственный сотрудник ДУКО» и правом `query.create`

- **NFR-SEC-03:** Доступ к текстам запросов и ответов ограничен правами `query.view` (только создатель запроса, ответственный в подразделении, супервизоры)

- **NFR-SEC-04:** Прикреплённые файлы должны проходить антивирусную проверку перед сохранением

- **NFR-SEC-05:** Текст запроса и ответа должен проходить валидацию на наличие SQL injection, XSS и других паттернов атак

- **NFR-SEC-06:** Все операции должны выполняться в рамках зашифрованного HTTPS-соединения

- **NFR-SEC-07:** Файлы в хранилище должны быть доступны только через авторизованные запросы (signed URLs с ограниченным сроком действия)

### 7.3. Надёжность

- **NFR-REL-01:** Система должна гарантировать атомарность операции создания запроса (все шаги 24–37 выполняются полностью или не выполняются вовсе)

- **NFR-REL-02:** При сбое на любом шаге транзакции все изменения должны быть откачены (ROLLBACK)

- **NFR-REL-03:** Система должна поддерживать механизм повторных попыток (retry) при временных ошибках БД и внешних API (максимум 3 попытки)

- **NFR-REL-04:** Целостность данных должна быть защищена от конкурентных изменений (optimistic locking или row-level locking)

- **NFR-REL-05:** Ошибка отправки уведомлений НЕ должна блокировать основную операцию создания запроса или получения ответа

### 7.4. Доступность

- **NFR-AVAIL-01:** Операция создания запроса должна быть доступна для выполнения с клавиатуры (последовательность Tab + Enter в диалоге)

- **NFR-AVAIL-02:** Валидационные сообщения должны быть доступны для screen readers (ARIA live regions)

- **NFR-AVAIL-03:** Интерфейс должен соответствовать стандарту WCAG 2.1 уровня AA

- **NFR-AVAIL-04:** Система должна поддерживать работу в условиях медленного интернета (graceful degradation)

### 7.5. Масштабируемость

- **NFR-SCALE-01:** Система должна поддерживать обработку до 1000 активных запросов одновременно

- **NFR-SCALE-02:** Хранилище файлов должно поддерживать объём до 1 ТБ вложений

- **NFR-SCALE-03:** Архивация завершённых запросов (старше 1 года) должна выполняться автоматически для оптимизации производительности

### 7.6. Интеграции

- **NFR-INT-01:** Система должна поддерживать интеграцию с минимум 5 внешними системами подразделений через REST API

- **NFR-INT-02:** Формат обмена данными: JSON

- **NFR-INT-03:** Таймаут вызова внешнего API: 30 секунд

- **NFR-INT-04:** Система должна корректно работать при недоступности внешних API (fallback на внутренние уведомления)

- **NFR-INT-05:** Поддержка webhooks для получения ответов от внешних систем в реальном времени

### 7.7. Удобство использования

- **NFR-UX-01:** Система должна сохранять черновики запросов автоматически каждые 2 минуты (auto-save)

- **NFR-UX-02:** Система должна предоставлять шаблоны типовых запросов для каждого подразделения (автозаполнение)

- **NFR-UX-03:** Система должна отображать историю всех запросов по обращению в хронологическом порядке с возможностью фильтрации по подразделению и статусу

## 8. Бизнес-правила

### BR-QUERY-01: Обязательность полей запроса

**Правило:** Для создания запроса обязательны следующие поля:

- Целевое подразделение

- Тема запроса (от 5 до 200 символов)

- Текст запроса (минимум 20 символов)

**Обоснование:** Обеспечивает минимально необходимую информацию для обработки запроса подразделением.

**Исключения:** Нет.

---

### BR-QUERY-02: Расчёт планового срока ответа

**Правило:** Плановый срок ответа рассчитывается автоматически на основе приоритета запроса и базового SLA подразделения:

|Приоритет|Формула расчёта|
|-|-|
|Обычный|текущая дата + базовый SLA подразделения|
|Высокий|текущая дата + (базовый SLA подразделения / 2)|
|Критический|текущая дата + 1 рабочий день|

**Базовые SLA подразделений (примеры):**

- Юридический отдел: 5 рабочих дней

- IT-поддержка: 3 рабочих дня

- Служба безопасности: 7 рабочих дней

- Департамент продуктов: 5 рабочих дней

**Обоснование:** Баланс между скоростью обработки и качеством ответа.

**Исключения:** Супервизор может вручную изменить плановый срок ответа при создании запроса.

---

### BR-QUERY-03: Назначение ответственного в целевом подразделении

**Правило:** Ответственный сотрудник в целевом подразделении назначается автоматически по алгоритму:

1. Выбираются сотрудники с компетенцией по категории запроса

2. Из них выбирается сотрудник с минимальной текущей загрузкой (количество активных запросов)

3. Если несколько сотрудников имеют одинаковую загрузку, выбирается с наилучшим показателем «Средняя оценка ответов»

**Обоснование:** Равномерное распределение нагрузки и обеспечение качества ответов.

**Исключения:** Если в настройках подразделения отключено автоназначение (`auto_assign_queries = false`), запрос направляется координатору для ручного распределения.

---

### BR-QUERY-04: Блокировка обработки обращения при ожидании ответа

**Правило:** При создании запроса пользователь может установить флаг «Блокировать обработку до получения ответа» (по умолчанию: включён).

**Эффекты:**

- Если флаг установлен: статус обращения меняется на «Ожидает ответа от смежного отдела», поля «Внутреннее решение» и «Шаблон ответа клиенту» блокируются для редактирования

- Если флаг снят: статус обращения остаётся «В работе», пользователь может продолжать работу параллельно ожиданию ответа

**Обоснование:** Гибкость процесса — некоторые запросы требуют обязательного ответа перед завершением обработки, другие носят информационный характер.

**Исключения:** Супервизор может изменить статус обращения вручную независимо от флага.

---

### BR-QUERY-05: Уведомление о просрочке ответа

**Правило:** Если запрос не получил ответ до планового срока (`current_date > planned_response_date`):

- Система отмечает запрос как просроченный (`is_overdue = true`)

- Отправляет уведомление супервизору ДУКО (приоритет «high»)

- Отправляет повторное напоминание ответственному сотруднику целевого подразделения

- Отправляет уведомление координатору целевого подразделения

**Периодичность проверки:** каждые 5 минут (scheduled job).

**Обоснование:** Обеспечение соблюдения SLA и своевременного реагирования на просрочки.

**Исключения:** Нет.

---

### BR-QUERY-06: Лимиты на размер и количество файлов

**Правило:**

- Максимальный размер одного файла: 10 МБ

- Максимальное количество файлов в одном запросе: 10

- Общий размер всех файлов в запросе: 50 МБ

- Допустимые форматы: PDF, DOCX, XLSX, PNG, JPG, JPEG, TXT

**Обоснование:** Предотвращение перегрузки хранилища и сетевых каналов, защита от злоупотреблений.

**Исключения:** Администратор системы может увеличить лимиты для конкретных пользователей или подразделений.

---

### BR-QUERY-07: Антивирусная проверка файлов

**Правило:** Все прикреплённые файлы должны пройти антивирусную проверку перед сохранением в хранилище.

**Действия при обнаружении угрозы:**

- Файл отклоняется, не сохраняется

- Пользователю отображается сообщение: «Файл "[имя]" содержит потенциальную угрозу и не может быть прикреплён.»

- Событие логируется в SECURITY_AUDIT_LOG

- Отправляется алерт команде безопасности

**Обоснование:** Защита от вредоносного ПО.

**Исключения:** Нет.

---

### BR-QUERY-08: Фиксация времени обработки запроса

**Правило:** Система автоматически рассчитывает и сохраняет метрики обработки запроса:

- `actual_response_duration` = `responded_at - created_at` (в часах)

- `is_overdue` = true (если `responded_at > planned_response_date`) или false

**Использование метрик:**

- Аналитика эффективности подразделений

- Расчёт среднего времени ответа по категориям запросов

- Выявление проблемных зон межподразделенческого взаимодействия

**Обоснование:** Основа для принятия решений по оптимизации процессов.

**Исключения:** Нет.

---

### BR-QUERY-09: Множественные запросы по одному обращению

**Правило:** По одному обращению может быть создано неограниченное количество запросов в разные подразделения.

**Логика обработки:**

- Если установлен флаг блокировки хотя бы для одного запроса — обращение остаётся в статусе «Ожидает ответа»

- Статус возвращается в «В работе» только после получения ответов на ВСЕ запросы с флагом блокировки

- Запросы без флага блокировки не влияют на статус обращения

**Обоснование:** Комплексные обращения могут требовать консультаций нескольких подразделений.

**Исключения:** Нет.

---

### BR-QUERY-10: Автосохранение черновиков

**Правило:** Система автоматически сохраняет черновик запроса каждые 2 минуты, если пользователь начал заполнение формы.

**Условия автосохранения:**

- Заполнено хотя бы одно поле (подразделение, тема или текст)

- Диалоговое окно остаётся открытым

- Пользователь не активировал кнопку «Отмена»

**Восстановление черновика:**

- При повторном открытии формы создания запроса для того же обращения система предлагает: «Обнаружен несохранённый черновик. Восстановить?»

- Черновики хранятся 24 часа, после чего удаляются автоматически

**Обоснование:** Предотвращение потери данных при случайном закрытии окна или сбое браузера.

**Исключения:** Нет.

## 9. Связь с другими Use Cases

### Предшествующие Use Cases:

- **UC-00:** Регистрация нового обращения

    - *Связь:* Обращение должно быть зарегистрировано до создания запросов в подразделения

- **UC-ACCEPT-01:** Принять обращение в работу ответственным

    - *Связь:* Обращение должно быть в статусе «В работе» перед созданием запроса

- **UC-01:** Просмотр детализации обращения

    - *Связь:* Пользователь просматривает карточку обращения перед принятием решения о создании запроса

### Последующие Use Cases:

- **UC-02:** Обработка обращения ответственным

    - *Связь:* После получения ответа пользователь использует информацию для завершения обработки обращения

- **UC-SUBMIT-01:** Отправка обращения на доведение

    - *Связь:* После получения всех необходимых ответов пользователь может завершить обработку и отправить обращение дальше

### Параллельные Use Cases:

- **UC-COMMENT-01:** Добавление комментария к обращению

    - *Связь:* Пользователь может добавлять комментарии параллельно ожиданию ответов на запросы

- **UC-HISTORY-01:** Просмотр протокола обработки

    - *Связь:* Создание и получение ответа на запрос фиксируется в протоколе обработки

- **UC-NOTIFY-01:** Просмотр уведомлений

    - *Связь:* Сотрудники подразделений и ДУКО получают уведомления о запросах и ответах

- **UC-DRAFT-01:** Сохранение черновика обращения

    - *Связь:* Черновики запросов сохраняются независимо от черновиков обращения

### Связанные Use Cases других подразделений:

- **UC-DEPT-QUERY-01:** Обработка входящего запроса сотрудником подразделения

    - *Связь:* Сотрудник целевого подразделения получает запрос, созданный в рамках UC-QUERY-01, и обрабатывает его

- **UC-DEPT-RESPONSE-01:** Предоставление ответа на запрос

    - *Связь:* Ответ, предоставленный сотрудником подразделения, поступает в систему и обрабатывается в рамках UC-QUERY-01 (фаза 11–15)

### Зависимые Use Cases:

- **UC-ESCALATE-01:** Эскалация просроченного запроса

    - *Связь:* Если запрос просрочен, супервизор может инициировать эскалацию к руководству подразделения

- **UC-STATS-01:** Просмотр статистики обработки

    - *Связь:* Метрики запросов (количество, средняя длительность, процент просрочек) учитываются в общей аналитике

### 10.6. Метрики для мониторинга

- **Количество запросов за период** (по подразделениям, по категориям)

- **Среднее время ответа** (actual_response_duration) по подразделениям

- **Процент просроченных запросов** (is_overdue = true) target: < 5%

- **Процент успешных интеграций** (integration_failed = false) target: > 95%

- **Топ-5 подразделений по объёму запросов** (для планирования ресурсов)

- **Топ-5 категорий запросов** (для создания типовых шаблонов и FAQ)

- **Среднее количество запросов на одно обращение** (для анализа сложности обращений)

- Система должна поддерживать работу в режиме частичной недоступности внешних интеграций (graceful degradation)
