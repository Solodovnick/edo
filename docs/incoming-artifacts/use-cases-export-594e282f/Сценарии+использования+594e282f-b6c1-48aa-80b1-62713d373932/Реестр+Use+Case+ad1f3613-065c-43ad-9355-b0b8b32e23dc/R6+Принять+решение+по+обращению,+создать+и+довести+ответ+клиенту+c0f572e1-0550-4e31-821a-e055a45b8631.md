---

## 1. Паспорт Use Case

- **Идентификатор:** UC-DECIDE-01

- **Название:** Принятие решения по обращению, формирование и доведение ответа клиенту

- **Актор:** Ответственный сотрудник ДУКО (Группа обработки обращений клиентов)

- **Цель:** Принять обоснованное решение по обращению клиента, сформировать корректный ответ на основе шаблонов или произвольного текста, довести ответ до клиента через указанный канал коммуникации и зафиксировать результат в системе для завершения цикла обработки обращения

- **Триггер:** Обращение находится в статусе «В работе», ответственный сотрудник собрал все необходимые данные (включая ответы от смежных подразделений, если требовалось) и готов сформировать финальное решение

- **Частота использования:** 30–100 раз в день на одного сотрудника

- **Приоритет:** Критический (напрямую влияет на соблюдение SLA, клиентский опыт и репутацию банка)

- **Уровень:** Пользовательская задача (User Goal Level)

---

## 2. Предусловия

- Обращение существует в системе с уникальным идентификатором

- Текущий статус обращения — «В работе»

- Пользователь является назначенным ответственным сотрудником по данному обращению

- Пользователь аутентифицирован в системе

- Пользователь имеет роль «Ответственный сотрудник ДУКО» с правами на обработку обращений и доведение ответов

- Карточка обращения открыта и отображена пользователю (выполнен UC-01 «Просмотр детализации обращения»)

- Все обязательные данные обращения собраны (содержание претензии, данные заявителя, результаты проверок)

- Если были отправлены запросы в смежные подразделения (UC-QUERY-01), получены ответы на все критически важные запросы

- База данных доступна и транзакции выполняются корректно

- В системе существует справочник шаблонов ответов (таблица `response_templates`) с активными записями

- Канал доведения ответа определён (email, телефон, личный кабинет, письменное уведомление)

- Интеграционный сервис доставки ответов (email-service, SMS-gateway) доступен

---

## 3. Основной поток (Main Flow)

### Фаза 1: Открытие карточки обращения для финализации

1. **Пользователь** находится в карточке обращения в статусе «В работе».

2. **Система** отображает рабочую область с блоками:

    - «Внутреннее решение *» (обязательное поле, textarea)

    - «Шаблон ответа клиенту *» (обязательное поле, dropdown + textarea)

    - Кнопки действий: «Сохранить черновик», «Довести ответ»

1. **Система** проверяет наличие сохранённого черновика решения и ответа:

    - Если черновик существует (ранее пользователь сохранял данные через «Сохранить черновик»):

        - Загружает данные из таблицы `appeal_drafts`

        - Заполняет поля «Внутреннее решение» и «Шаблон ответа клиенту» сохранёнными значениями

        - Отображает информацию о черновике: «Восстановлен черновик от [дата, время]»

    - Если черновика нет:

        - Поля остаются пустыми

### Фаза 2: Формирование внутреннего решения

1. **Пользователь** анализирует данные обращения (содержание претензии, данные клиента, ответы от смежных подразделений, документы).

2. **Пользователь** вводит текст внутреннего решения в поле «Внутреннее решение *».

3. **Система** выполняет валидацию в реальном времени (при потере фокуса):

    - Проверяет, что поле не пустое

    - Проверяет минимальную длину (>= 20 символов)

    - Отображает счётчик символов под полем

1. **Пользователь** опционально активирует автосохранение черновика (система автоматически сохраняет каждые 2 минуты).

### Фаза 3: Выбор и настройка шаблона ответа клиенту

1. **Пользователь** активирует dropdown «Выберите шаблон ответа».

2. **Система** загружает список доступных шаблонов из базы данных (таблица `response_templates`):

    - Фильтр: `is_active = true`

    - Группировка по категориям (соответствующим типу обращения)

1. **Система** отображает список шаблонов с названиями:

    - «Удовлетворение претензии полностью»

    - «Частичное удовлетворение претензии»

    - «Отказ в удовлетворении претензии (обоснованный)»

    - «Разъяснение условий договора»

    - «Извинение за технический сбой»

    - «Перенаправление в другое подразделение»

    - «Произвольный ответ (без шаблона)»

1. **Пользователь** выбирает подходящий шаблон из списка.

2. **Система** загружает текст выбранного шаблона из поля `template_text`.

3. **Система** выполняет подстановку переменных в шаблон:

    - `{client_name}` → ФИО заявителя из обращения

    - `{appeal_number}` → номер обращения

    - `{appeal_date}` → дата регистрации обращения

    - `{resolution_date}` → текущая дата

    - `{responsible_name}` → ФИО ответственного сотрудника

    - Другие переменные согласно настройке шаблона

1. **Система** отображает обработанный текст шаблона в поле «Текст ответа клиенту» (textarea, редактируемое).

2. **Пользователь** просматривает предзаполненный текст ответа.

3. **Пользователь** опционально редактирует текст ответа:

    - Добавляет/удаляет абзацы

    - Корректирует формулировки

    - Добавляет специфичные детали по данному обращению

### Фаза 4: Валидация данных перед доведением

1. **Пользователь** завершает заполнение полей и активирует кнопку «Довести ответ».

2. **Система** выполняет валидацию обязательных полей:

    - Проверяет, что поле «Внутреннее решение» заполнено (длина >= 20 символов)

    - Проверяет, что поле «Текст ответа клиенту» заполнено (длина >= 50 символов)

    - Проверяет, что выбран шаблон (если «Произвольный ответ», проверяет наличие текста)

1. **Система** выполняет дополнительные проверки:

    - Проверяет, что статус обращения по-прежнему «В работе» (защита от конкурентных изменений)

    - Проверяет, что пользователь по-прежнему является ответственным

    - Проверяет, что нет незакрытых критических запросов в смежные подразделения (если были установлены блокировки)

### Фаза 5: Подтверждение доведения ответа

1. **Система** отображает диалоговое окно подтверждения: «Вы уверены, что готовы довести ответ клиенту?»

2. **Система** в диалоге отображает краткую информацию:

    - Тип обращения и номер

    - ФИО заявителя

    - Канал доведения (email / телефон / письменное уведомление)

    - Предпросмотр текста ответа (первые 200 символов)

1. **Пользователь** подтверждает действие (активирует кнопку подтверждения).

### Фаза 6: Обновление данных обращения

1. **Система** начинает транзакцию базы данных.

2. **Система** обновляет запись обращения (таблица `appeals`):

    - `internal_decision` = текст внутреннего решения

    - `response_template_id` = ID выбранного шаблона (или NULL для произвольного)

    - `response_text` = финальный текст ответа клиенту

    - `status` = «На доведении»

    - `decision_made_at` = текущая дата и время

    - `decision_made_by_user_id` = ID текущего пользователя

    - `last_modified_at` = текущая дата и время

    - `last_modified_by` = ID текущего пользователя

1. **Система** вычисляет длительность обработки:

    - `processing_duration` = `decision_made_at - work_started_at` (в часах)

1. **Система** проверяет соблюдение SLA:

    - Если `decision_made_at <= sla_deadline`:

        - `sla_met` = true

    - Если `decision_made_at > sla_deadline`:

        - `sla_met` = false

        - `sla_overdue_hours` = разница в часах

### Фаза 7: Фиксация в протоколе обработки

1. **Система** создаёт запись в протоколе обработки (таблица `appeal_audit_log`):

    - `appeal_id` = ID текущего обращения

    - `action_type` = «decision_made»

    - `action_description` = «Принято решение по обращению, подготовлен ответ клиенту»

    - `performed_by_user_id` = ID текущего пользователя

    - `performed_at` = текущая дата и время

    - `stage` = «completion»

    - `previous_status` = «В работе»

    - `new_status` = «На доведении»

    - `metadata` = JSON с данными:

        - `template_used` = название шаблона

        - `processing_duration` = длительность обработки

        - `sla_met` = соблюдён ли SLA

        - `decision_summary` = первые 200 символов внутреннего решения

### Фаза 8: Отправка ответа клиенту

1. **Система** определяет канал доведения ответа из данных обращения (поле `communication_channel`):

    - Email → вызов email-service

    - Телефон → создание задачи для колл-центра

    - Личный кабинет → публикация в системе интернет-банкинга

    - Письменное уведомление → формирование PDF и постановка в очередь печати

1. **Система** подготавливает данные для отправки:

    - Получатель: email/телефон заявителя

    - Тема (для email): «Ответ на обращение №[номер] от [дата]»

    - Тело сообщения: текст ответа клиенту

    - Вложения (если применимо): PDF с официальным ответом

1. **Система** вызывает интеграционный сервис доставки:

    - **Для email:**

        - POST-запрос к email-service API

        - Параметры: `recipient`, `subject`, `body`, `attachments`

        - Ожидание подтверждения отправки (таймаут 30 секунд)

    - **Для SMS:**

        - POST-запрос к SMS-gateway API

        - Параметры: `phone`, `message` (сокращённая версия ответа)

    - **Для личного кабинета:**

        - POST-запрос к API интернет-банкинга

        - Создание уведомления в личном кабинете клиента

    - **Для письменного уведомления:**

        - Генерация PDF-документа с официальным бланком

        - Постановка в очередь печати и отправки почтой

1. **Система** получает подтверждение успешной отправки от интеграционного сервиса:

    - Статус: `success` / `failed`

    - `delivery_id` = уникальный ID отправки (для трекинга)

    - `delivered_at` = дата и время доставки

### Фаза 9: Обновление статуса после доведения

1. **Система** обновляет запись обращения после успешной отправки:

    - `status` = «Закрыто»

    - `response_delivered_at` = дата и время доставки

    - `delivery_channel` = канал доставки (email / sms / personal_cabinet / mail)

    - `delivery_id` = ID отправки из интеграционного сервиса

    - `delivery_status` = «delivered»

    - `closed_at` = текущая дата и время

    - `closed_by_user_id` = ID текущего пользователя

1. **Система** создаёт запись в протоколе обработки:

    - `action_type` = «response_delivered»

    - `action_description` = «Ответ доведён до клиента через [канал]»

    - `performed_at` = текущая дата и время

    - `new_status` = «Закрыто»

    - `metadata` = JSON с данными доставки

### Фаза 10: Удаление черновика и уведомления

1. **Система** удаляет сохранённый черновик (если существует) из таблицы `appeal_drafts`:

    - DELETE WHERE `appeal_id = ?` AND `user_id = ?`

1. **Система** создаёт уведомление для супервизора ДУКО:

    - `notification_type` = «appeal_closed»

    - `message` = «Обращение №[номер] закрыто сотрудником [ФИО]. SLA: [соблюдён/просрочен]. Длительность обработки: [часы].»

    - `priority` = «normal» (если SLA соблюдён) или «high» (если просрочен)

1. **Система** опционально отправляет уведомление в смежные подразделения, которые участвовали в обработке (если были запросы):

    - «Обращение №[номер], по которому вы предоставляли информацию, закрыто. Спасибо за сотрудничество.»

### Фаза 11: Обновление метрик и статистики

1. **Система** обновляет статистику сотрудника (таблица `user_statistics`):

    - Увеличивает счётчик `closed_appeals_count`

    - Обновляет `last_closed_at` = текущая дата и время

    - Пересчитывает среднее время обработки

    - Пересчитывает процент соблюдения SLA

1. **Система** обновляет общую статистику подразделения:

    - Увеличивает счётчик закрытых обращений за день/месяц

    - Обновляет агрегированные метрики SLA

### Фаза 12: Завершение операции

1. **Система** фиксирует транзакцию в базе данных (COMMIT).

2. **Система** закрывает диалоговое окно подтверждения.

3. **Система** отображает уведомление пользователю: «Ответ успешно доведён до клиента. Обращение №[номер] закрыто.»

4. **Система** выполняет одно из действий (согласно настройке `post_close_behavior`):

    - Вариант A: Автоматически возвращает пользователя к реестру обращений (UC-LIST-01)

    - Вариант B: Остаётся на карточке обращения в режиме read-only с отображением результата

1. **Система** удаляет обращение из списка «Мои обращения» (фильтр активных обращений).

## 4. Постусловия

### Успешное завершение:

- Статус обращения в базе данных изменён на «Закрыто»

- Поля `internal_decision` и `response_text` содержат финальные версии решения и ответа

- Ответ доведён до клиента через указанный канал коммуникации

- Получено подтверждение доставки от интеграционного сервиса (delivery_id)

- В протоколе обработки зафиксированы два события:

    - «Принято решение по обращению»

    - «Ответ доведён до клиента»

- Рассчитана длительность обработки обращения

- Определён факт соблюдения/нарушения SLA

- Супервизор получил уведомление о закрытии обращения

- Статистика сотрудника и подразделения обновлена

- Черновик удалён из системы (если существовал)

- Смежные подразделения уведомлены о завершении обращения (если участвовали)

- Обращение удалено из активного реестра «Мои обращения»

- Обращение доступно для просмотра в режиме read-only или в архиве

### Состояние системы:

- Транзакция базы данных успешно завершена (committed)

- Целостность данных сохранена

- Аудиторский след полностью зафиксирован

- Метрики SLA обновлены

- Клиент получил ответ на своё обращение

- Система готова к обработке следующих обращений

## 5. Альтернативные потоки

### A1: Не заполнено обязательное поле «Внутреннее решение»

- **Точка ветвления:** Шаг 18

- **Условие:** Поле «Внутреннее решение» пустое или содержит менее 20 символов

- **Действия:**

    1. Система не закрывает карточку обращения

    2. Система отображает валидационное сообщение под полем: «Поле "Внутреннее решение" обязательно для заполнения. Минимальная длина: 20 символов. Текущая длина: [N].»

    3. Система подсвечивает поле (индикация ошибки)

    4. Система блокирует кнопку «Довести ответ» до устранения ошибки

    5. Пользователь корректирует текст решения

    6. Основной поток возвращается к шагу 18

### A2: Не заполнено обязательное поле «Текст ответа клиенту»

- **Точка ветвления:** Шаг 18

- **Условие:** Поле «Текст ответа клиенту» пустое или содержит менее 50 символов

- **Действия:**

    1. Система не закрывает карточку обращения

    2. Система отображает валидационное сообщение: «Поле "Текст ответа клиенту" обязательно для заполнения. Минимальная длина: 50 символов. Текущая длина: [N].»

    3. Система подсвечивает поле (индикация ошибки)

    4. Система блокирует кнопку «Довести ответ»

    5. Пользователь выбирает шаблон или вводит произвольный текст

    6. Основной поток возвращается к шагу 18

### A3: Выбранный шаблон ответа недоступен или устарел

- **Точка ветвления:** Шаг 12

- **Условие:** Между загрузкой списка шаблонов и выбором конкретного шаблона он был деактивирован администратором (`is_active = false`)

- **Действия:**

    1. Система пытается загрузить текст шаблона

    2. Система обнаруживает, что шаблон неактивен или удалён

    3. Система отображает уведомление: «Выбранный шаблон больше недоступен. Пожалуйста, выберите другой шаблон или введите произвольный текст.»

    4. Система очищает поле «Текст ответа клиенту»

    5. Система сбрасывает выбор шаблона в dropdown

    6. Пользователь выбирает другой шаблон или вариант «Произвольный ответ»

    7. Основной поток возвращается к шагу 11

### A4: Пользователь нажимает «Сохранить черновик»

- **Точка ветвления:** После шага 16 (альтернатива шагу 17)

- **Условие:** Пользователь активирует кнопку «Сохранить черновик» вместо «Довести ответ»

- **Действия:**

    1. Система НЕ выполняет валидацию обязательных полей (черновик может быть неполным)

    2. Система начинает транзакцию БД

    3. Система проверяет наличие существующего черновика для данного обращения и пользователя:

        - Если черновик существует: UPDATE запись

        - Если черновика нет: INSERT новую запись

    1. Система сохраняет данные в таблицу `appeal_drafts`:

        - `appeal_id` = ID обращения

        - `user_id` = ID пользователя

        - `internal_decision_draft` = текст из поля «Внутреннее решение»

        - `response_template_id_draft` = ID выбранного шаблона

        - `response_text_draft` = текст из поля «Текст ответа клиенту»

        - `saved_at` = текущая дата и время

    1. Система НЕ изменяет статус обращения (остаётся «В работе»)

    2. Система фиксирует транзакцию (COMMIT)

    3. Система отображает уведомление: «Черновик сохранён. Вы можете вернуться к обработке позже.»

    4. Система опционально возвращает пользователя к реестру обращений или остаётся на карточке

    5. Сценарий завершается

### A5: Пользователь отменяет подтверждение доведения ответа

- **Точка ветвления:** Между шагом 21 и шагом 22

- **Условие:** Пользователь активирует кнопку «Отмена» в диалоговом окне подтверждения

- **Действия:**

    1. Система закрывает диалоговое окно подтверждения

    2. Система не выполняет шаги 23–43

    3. Карточка обращения остаётся открытой в исходном состоянии

    4. Статус обращения не изменяется («В работе»)

    5. Пользователь может продолжить редактирование решения и ответа или повторить попытку доведения

    6. Сценарий завершается

### A6: Ошибка отправки ответа клиенту (сбой интеграции)

- **Точка ветвления:** Шаг 31

- **Условие:** Интеграционный сервис доставки (email-service, SMS-gateway) возвращает ошибку или не отвечает в течение 30 секунд

- **Действия:**

    1. Система получает ответ `status = failed` или таймаут

    2. Система выполняет ROLLBACK изменения статуса на «Закрыто» (откатывает шаги 32–33)

    3. Система сохраняет статус обращения как «На доведении» (промежуточный статус)

    4. Система обновляет запись обращения:

        - `status` = «На доведении»

        - `delivery_status` = «failed»

        - `delivery_error_message` = текст ошибки от интеграционного сервиса

        - `delivery_retry_count` = 1

    1. Система фиксирует транзакцию (COMMIT)

    2. Система создаёт запись в протоколе:

        - `action_type` = «delivery_failed»

        - `action_description` = «Ошибка доведения ответа через [канал]: [текст ошибки]»

    1. Система отображает уведомление пользователю: «Не удалось довести ответ клиенту. Причина: [текст ошибки]. Ответ сохранён, система автоматически повторит попытку отправки через 5 минут.»

    2. Система логирует ошибку с уровнем ERROR

    3. Система ставит обращение в очередь повторной отправки (таблица `delivery_retry_queue`)

    4. Система остаётся на карточке обращения

    5. Пользователь может:

        - Ожидать автоматической повторной отправки

        - Вручную повторить попытку доведения (кнопка «Повторить отправку»)

        - Изменить канал доведения (если применимо)

    1. Сценарий продолжается с автоматической повторной попыткой (шаг 30)

### A7: Обращение переназначено другому сотруднику

- **Точка ветвления:** Шаг 19

- **Условие:** Между открытием карточки и попыткой довести ответ обращение было переназначено другому сотруднику супервизором

- **Действия:**

    1. Система проверяет поле `responsible_user_id` обращения

    2. Система обнаруживает несоответствие с `current_user_id`

    3. Система прерывает выполнение операции

    4. Система отображает уведомление: «Обращение №[номер] переназначено другому сотруднику. Вы больше не можете довести ответ. Все введённые данные будут сохранены как черновик для нового ответственного.»

    5. Система автоматически сохраняет данные как черновик (выполняет A4, шаги 2–4)

    6. Система переводит карточку в режим read-only

    7. Система предлагает вернуться к реестру обращений

    8. Сценарий завершается

### A8: Обнаружены незакрытые критические запросы в смежные подразделения

- **Точка ветвления:** Шаг 19

- **Условие:** Существуют активные запросы в смежные подразделения со статусом «Отправлен» или «В обработке» и флагом `is_blocked = true`

- **Действия:**

    1. Система выполняет запрос к таблице `inter_department_queries`:

        - WHERE `appeal_id = ?` AND `status IN ('sent', 'in_progress')` AND `is_blocked = true`

    1. Система обнаруживает незакрытые запросы

    2. Система отображает предупреждающее сообщение: «По данному обращению существуют незакрытые запросы в следующие подразделения:

        - [Подразделение 1]: Запрос №[номер], статус: [статус], плановый срок: [дата]

        - 

    1. Система предлагает варианты действий:

        - Кнопка «Ожидать ответов» (отменяет доведение, оставляет обращение в статусе «В работе»)

        - Кнопка «Снять блокировку запросов и продолжить» (изменяет флаг `is_blocked = false` для запросов и продолжает доведение)

        - Кнопка «Отмена»

    1. Пользователь выбирает действие

    2. Если выбрано «Снять блокировку»:

        - Система обновляет запросы (UPDATE `is_blocked = false`)

        - Основной поток продолжается с шага 20

    1. Если выбрано «Ожидать ответов»:

        - Сценарий завершается, обращение остаётся в статусе «В работе»

### A9: Автоматическое восстановление черновика после потери сессии

- **Точка ветвления:** Шаг 3 (при открытии карточки)

- **Условие:** Произошла потеря сессии (разрыв соединения, закрытие браузера), но данные были сохранены в `sessionStorage`

- **Действия:**

    1. Система проверяет наличие данных в `sessionStorage` для ключа `appeal_[ID]_draft`

    2. Если данные найдены:

        - Система отображает уведомление: «Обнаружены несохранённые данные. Восстановить?»

        - Кнопки: «Восстановить», «Отклонить»

    1. Если пользователь выбирает «Восстановить»:

        - Система заполняет поля из `sessionStorage`

        - Система очищает `sessionStorage`

    1. Если пользователь выбирает «Отклонить»:

        - Система очищает `sessionStorage`

        - Поля остаются пустыми или загружаются из черновика в БД (если есть)

    1. Основной поток продолжается с шага 4

## 6. Исключительные ситуации (Exception Flows)

### E1: Ошибка подключения к базе данных при сохранении решения

- **Точка возникновения:** Шаг 23, 24, 27, 32

- **Условие:** Потеря соединения с базой данных или таймаут запроса

- **Действия:**

    1. Система прерывает выполнение операции

    2. Система выполняет ROLLBACK транзакции (если она была начата)

    3. Система отображает уведомление: «Не удалось сохранить решение. Ошибка подключения к базе данных. Ваши данные сохранены локально. Попробуйте повторить операцию через несколько секунд.»

    4. Система автоматически сохраняет данные в `sessionStorage` (временное хранилище)

    5. Система логирует ошибку с уровнем ERROR:

        - Тип: DATABASE_CONNECTION_FAILED

        - Operation: SAVE_DECISION

        - Appeal ID

        - User ID

        - Timestamp

    1. Система отправляет алерт команде DevOps

    2. Карточка обращения остаётся открытой с введёнными данными

    3. Кнопка «Довести ответ» остаётся доступной для повторной попытки

    4. Пользователь может повторить попытку или сохранить черновик

    5. Сценарий завершается

### E2: Таймаут вызова API доставки ответа

- **Точка возникновения:** Шаг 30

- **Условие:** Интеграционный сервис не отвечает в течение 30 секунд (настроенный таймаут)

- **Действия:**

    1. Система прерывает ожидание ответа от API

    2. Система НЕ откатывает основную транзакцию (решение и ответ сохранены)

    3. Система устанавливает статус обращения = «На доведении» (промежуточный)

    4. Система сохраняет информацию об ошибке:

        - `delivery_status` = «timeout»

        - `delivery_error_message` = «Превышено время ожидания ответа от сервиса доставки»

    1. Система фиксирует транзакцию (COMMIT)

    2. Система отображает уведомление: «Превышено время ожидания доставки ответа. Решение сохранено. Система автоматически повторит попытку отправки в фоновом режиме.»

    3. Система логирует событие с уровнем WARNING

    4. Система ставит обращение в очередь повторной отправки (retry queue)

    5. Система создаёт задачу для мониторинга (таблица `monitoring_tasks`)

    6. Фоновый процесс (scheduled job) повторяет попытку отправки каждые 5 минут (максимум 10 попыток)

    7. При успешной доставке система обновляет статус на «Закрыто» и уведомляет пользователя

    8. Сценарий продолжается с промежуточным статусом «На доведении»

### E3: Нарушение целостности данных (конкурентное изменение)

- **Точка возникновения:** Шаг 39 (COMMIT транзакции)

- **Условие:** Другой процесс одновременно изменил запись обращения, возникла блокировка или конфликт версий

- **Действия:**

    1. Система получает ошибку CONCURRENT_MODIFICATION или LOCK_TIMEOUT

    2. Система выполняет ROLLBACK транзакции

    3. Система повторяет попытку выполнения транзакции (retry logic, максимум 3 попытки с интервалом 500ms)

    4. Если после 3 попыток ошибка сохраняется:

        - Система отображает уведомление: «Не удалось довести ответ. Данные обращения были изменены другим пользователем или процессом. Обновите карточку обращения и повторите попытку.»

        - Система сохраняет введённые данные в черновик (автоматически)

        - Система логирует ошибку с уровнем WARNING

        - Система предлагает действие: «Обновить карточку обращения»

    1. Пользователь обновляет карточку и проверяет актуальность данных

    2. Пользователь повторяет операцию доведения

    3. Сценарий завершается

### E4: Ошибка генерации PDF-документа для письменного уведомления

- **Точка возникновения:** Шаг 30 (вариант генерации PDF)

- **Условие:** Сервис генерации PDF недоступен или возвращает ошибку

- **Действия:**

    1. Система получает ошибку от PDF-генератора

    2. Система логирует ошибку с уровнем ERROR

    3. Система НЕ прерывает основной процесс (решение и ответ сохранены)

    4. Система устанавливает статус обращения = «На доведении»

    5. Система создаёт задачу для оператора: «Требуется вручную сформировать письменное уведомление для обращения №[номер]»

    6. Система уведомляет пользователя: «Ответ сохранён, но не удалось автоматически создать письменное уведомление. Задача передана в службу документооборота.»

    7. Оператор службы документооборота вручную формирует и отправляет письмо

    8. После подтверждения отправки статус обращения обновляется на «Закрыто»

    9. Сценарий продолжается с ручной обработкой

### E5: Недостаточно прав доступа для доведения ответа

- **Точка возникновения:** Шаг 19

- **Условие:** Между открытием карточки и попыткой довести ответ у пользователя были отозваны права на доведение ответов

- **Действия:**

    1. Система проверяет актуальные разрешения пользователя (permissions check)

    2. Система обнаруживает отсутствие прав `appeal.deliver_response`

    3. Система прерывает выполнение операции

    4. Система отображает уведомление: «Недостаточно прав для доведения ответа. Обратитесь к администратору системы или супервизору. Ваши данные сохранены как черновик.»

    5. Система автоматически сохраняет данные как черновик (A4)

    6. Система логирует событие безопасности (SECURITY_AUDIT_LOG):

        - Event: UNAUTHORIZED_ACCESS_ATTEMPT

        - User ID

        - Appeal ID

        - Action: DELIVER_RESPONSE

        - Timestamp

    1. Карточка обращения остаётся в статусе «В работе»

    2. Сценарий завершается

### E6: Критическая ошибка интеграционного сервиса (500 Internal Server Error)

- **Точка возникновения:** Шаг 30

- **Условие:** API доставки возвращает HTTP 500 или другую критическую ошибку

- **Действия:**

    1. Система получает ошибку от API

    2. Система сохраняет статус обращения = «На доведении»

    3. Система сохраняет детали ошибки:

        - `delivery_status` = «error»

        - `delivery_error_code` = HTTP-код ответа

        - `delivery_error_message` = текст ошибки

    1. Система фиксирует транзакцию (COMMIT)

    2. Система отображает уведомление: «Критическая ошибка службы доставки. Ваш ответ сохранён и будет автоматически отправлен после восстановления службы.»

    3. Система логирует ошибку с уровнем CRITICAL:

        - Тип: INTEGRATION_SERVICE_ERROR

        - Service: [название сервиса]

        - HTTP status: [код]

        - Error message: [текст]

    1. Система отправляет экстренный алерт команде поддержки интеграций

    2. Система ставит обращение в приоритетную очередь повторной отправки

    3. Администратор расследует проблему и восстанавливает сервис

    4. После восстановления система автоматически повторяет отправку

    5. Сценарий продолжается с промежуточным статусом

### E7: Потеря сессии в момент доведения ответа

- **Точка возникновения:** Любой шаг между 23 и 39

- **Условие:** Сессия пользователя истекла или была прервана в момент выполнения операции

- **Действия:**

    1. Система обнаруживает недействительную сессию (401 Unauthorized)

    2. Если транзакция была начата:

        - Система выполняет ROLLBACK (откат изменений)

    1. Система сохраняет данные в `sessionStorage` (локальное хранилище)

    2. Система перенаправляет пользователя на страницу авторизации

    3. После успешной авторизации система отображает уведомление: «Обнаружены несохранённые данные обращения №[номер]. Восстановить?»

    4. Если пользователь подтверждает:

        - Система восстанавливает данные из `sessionStorage`

        - Система открывает карточку обращения с восстановленными данными

        - Пользователь может повторить операцию доведения

    1. Если пользователь отклоняет:

        - Данные очищаются, пользователь переходит к реестру

    1. Сценарий возобновляется после авторизации

## 7. Специальные требования

### 7.1. Производительность

- **NFR-PERF-01:** Время сохранения черновика (A4) не должно превышать 500 миллисекунд

- **NFR-PERF-02:** Время доведения ответа (шаги 17–42) не должно превышать 2 секунд при успешной работе интеграционных сервисов

- **NFR-PERF-03:** Время загрузки списка шаблонов ответов (шаг 9) не должно превышать 300 миллисекунд

- **NFR-PERF-04:** Автосохранение черновика (каждые 2 минуты) должно выполняться в фоновом режиме без блокировки интерфейса

- **NFR-PERF-05:** Операция должна выполняться в рамках одной атомарной транзакции БД с уровнем изоляции READ COMMITTED

- **NFR-PERF-06:** Система должна поддерживать одновременное доведение ответов 100+ пользователями без деградации производительности

### 7.2. Безопасность

- **NFR-SEC-01:** Все операции принятия решения и доведения ответа должны логироваться в аудиторском журнале с фиксацией:

    - ID пользователя, выполнившего действие

    - Точной даты и времени (с точностью до миллисекунд)

    - Предыдущего и нового статуса обращения

    - IP-адреса клиента

    - Session ID

    - Использованного шаблона ответа

- **NFR-SEC-02:** Операция доведения ответа доступна только пользователям с ролью «Ответственный сотрудник ДУКО» и правом `appeal.deliver_response`

- **NFR-SEC-03:** Проверка прав доступа должна выполняться на уровне API (backend), а не только на уровне UI

- **NFR-SEC-04:** Текст решения и ответа должен проходить валидацию на наличие персональных данных третьих лиц (PII leak detection)

- **NFR-SEC-05:** Все изменения должны выполняться в рамках зашифрованного HTTPS-соединения

- **NFR-SEC-06:** Доступ к отправленным ответам должен быть ограничен правами `appeal.view_responses` (только создатель, супервизор, аудитор)

- **NFR-SEC-07:** Черновики должны быть доступны только их создателю (изоляция на уровне БД: `WHERE user_id = current_user_id`)

### 7.3. Надёжность

- **NFR-REL-01:** Система должна гарантировать атомарность операции доведения ответа (все шаги 23–38 выполняются полностью или не выполняются вовсе)

- **NFR-REL-02:** При сбое на любом шаге транзакции все изменения должны быть откачены (ROLLBACK)

- **NFR-REL-03:** Система должна поддерживать механизм повторных попыток (retry) при временных ошибках интеграционных сервисов (максимум 10 попыток с экспоненциальным back-off)

- **NFR-REL-04:** Целостность данных должна быть защищена от конкурентных изменений (optimistic locking или row-level locking)

- **NFR-REL-05:** Ошибка отправки ответа НЕ должна блокировать сохранение решения — решение сохраняется, обращение переходит в статус «На доведении»

- **NFR-REL-06:** Автосохранение черновика должно работать даже при нестабильном интернет-соединении (offline-first approach с синхронизацией при восстановлении связи)

### 7.4. Доступность (Accessibility)

- **NFR-AVAIL-01:** Операция доведения ответа должна быть доступна для выполнения с клавиатуры (последовательность Tab + Enter)

- **NFR-AVAIL-02:** Все обязательные поля должны иметь ARIA-метки `aria-required="true"`

- **NFR-AVAIL-03:** Валидационные сообщения должны быть доступны для screen readers (ARIA live regions с `aria-live="polite"`)

- **NFR-AVAIL-04:** Интерфейс должен соответствовать стандарту WCAG 2.1 уровня AA

- **NFR-AVAIL-05:** Поддержка высококонтрастных тем для пользователей с нарушениями зрения

### 7.5. Отказоустойчивость

- **NFR-FAULT-01:** При недоступности службы доставки ответов система должна сохранить решение и перевести обращение в статус «На доведении» с автоматической повторной отправкой

- **NFR-FAULT-02:** При недоступности справочника шаблонов система должна предоставить возможность ввода произвольного ответа (fallback)

- **NFR-FAULT-03:** Система должна сохранять данные в `sessionStorage` каждые 30 секунд для восстановления при сбое браузера

### 7.6. Удобство использования (Usability)

- **NFR-UX-01:** Система должна отображать предупреждение при попытке закрыть карточку обращения с несохранёнными данными: «У вас есть несохранённые изменения. Сохранить черновик?»

- **NFR-UX-02:** Система должна отображать счётчик символов для полей с ограничениями (минимум/максимум)

- **NFR-UX-03:** Система должна подсвечивать обязательные поля красной звёздочкой (*)

- **NFR-UX-04:** Система должна предоставлять автозаполнение переменных в шаблонах с предпросмотром в реальном времени

- **NFR-UX-05:** Система должна отображать прогресс-индикатор при отправке ответа (шаги 28–31)

### 7.7. Интеграции

- **NFR-INT-01:** Система должна поддерживать интеграцию минимум с 4 каналами доставки ответов (email, SMS, личный кабинет, почта)

- **NFR-INT-02:** Таймаут вызова интеграционного API: 30 секунд

- **NFR-INT-03:** Формат обмена данными с интеграционными сервисами: JSON

- **NFR-INT-04:** Система должна получать подтверждение доставки (delivery confirmation) от всех интеграционных сервисов

## 8. Бизнес-правила

### BR-DECIDE-01: Обязательность поля «Внутреннее решение»

**Правило:** Ответ можно довести клиенту только при заполненном поле «Внутреннее решение» (минимум 20 символов).

**Обоснование:** Внутреннее решение необходимо для:

- Аудита действий сотрудника

- Анализа качества обработки обращений

- Обоснования позиции банка в случае эскалации или судебного разбирательства

**Исключения:** Супервизор с правом `appeal.override_validation` может довести ответ без внутреннего решения в исключительных случаях (с обязательным комментарием).

---

### BR-DECIDE-02: Справочник шаблонов ответов

**Правило:** Шаблоны ответов хранятся в справочнике (таблица `response_templates`) и актуализируются регулярно (минимум раз в квартал).

**Атрибуты шаблона:**

- `template_id` — уникальный ID

- `template_name` — название шаблона

- `category` — категория (удовлетворение / отказ / разъяснение и т.д.)

- `template_text` — текст с переменными ({client_name}, {appeal_number} и др.)

- `is_active` — флаг активности

- `created_at`, `updated_at` — даты создания и обновления

- `approved_by_legal` — флаг юридического одобрения (обязательно для шаблонов отказа)

**Обоснование:** Стандартизация ответов повышает качество коммуникации и снижает риски юридических претензий.

**Исключения:** Для уникальных ситуаций доступен вариант «Произвольный ответ (без шаблона)».

---

### BR-DECIDE-03: Статус «Закрыто» устанавливается только после успешной доставки

**Правило:** Статус обращения изменяется на «Закрыто» только после получения подтверждения доставки ответа от интеграционного сервиса.

**Промежуточные статусы:**

- «В работе» → «На доведении» (решение принято, ответ готов, но ещё не отправлен)

- «На доведении» → «Закрыто» (ответ успешно доставлен клиенту)

**Обоснование:** Обеспечивает точность отчётности и предотвращает преждевременное закрытие обращений.

**Исключения:** При критической ошибке интеграции супервизор может вручную закрыть обращение с обоснованием (если доставка подтверждена альтернативным способом).

---

### BR-DECIDE-04: Аудиторский след всех черновиков и отправленных ответов

**Правило:** Система должна сохранять полную историю изменений решения и ответа:

- Все версии черновиков (с датой и временем сохранения)

- Финальная версия отправленного ответа

- Факт использования шаблона (какой шаблон, какие изменения внесены)

**Хранение:** Таблицы `appeal_drafts` (черновики) и `appeal_audit_log` (финальные версии).

**Обоснование:** Прозрачность процесса, возможность аудита и расследования жалоб клиентов.

**Исключения:** Нет.

---

### BR-DECIDE-05: Управление статусом при ошибке доставки

**Правило:** При ошибке отправки ответа клиенту:

- Статус обращения остаётся «На доведении» (если решение уже сохранено)

- ИЛИ остаётся «В работе» (если ошибка произошла до сохранения решения)

- Обращение автоматически попадает в очередь повторной отправки

- Максимальное количество автоматических попыток: 10

- Интервал между попытками: экспоненциальный back-off (5 мин, 10 мин, 20 мин, ..., 8 часов)

**Действия при исчерпании попыток:**

- Создаётся задача для администратора системы

- Уведомляется супервизор ДУКО

- Обращение помечается флагом `requires_manual_delivery = true`

- Супервизор может инициировать доставку альтернативным способом или изменить канал доведения

**Обоснование:** Обеспечивает гарантированную доставку ответа клиенту даже при технических сбоях.

**Исключения:** Нет.

---

### BR-DECIDE-06: Расчёт метрик SLA

**Правило:** При доведении ответа система автоматически рассчитывает и фиксирует метрики:

- **Длительность обработки:**

    - `processing_duration = decision_made_at - work_started_at` (в часах)

- **Соблюдение SLA:**

    - `sla_met = true` (если `decision_made_at <= sla_deadline`)

    - `sla_met = false` (если `decision_made_at > sla_deadline`)

    - `sla_overdue_hours` = разница в часах при просрочке

- **Полная длительность:**

    - `total_duration = closed_at - created_at` (от регистрации до закрытия)

**Использование метрик:**

- Персональный KPI сотрудника (процент соблюдения SLA)

- Агрегированная статистика подразделения

- Выявление проблемных категорий обращений

**Обоснование:** Основа для управления производительностью и качеством обслуживания.

**Исключения:** Нет.

---

### BR-DECIDE-07: Подстановка переменных в шаблоны ответов

**Правило:** Система автоматически подставляет значения переменных в текст шаблона на основе данных обращения:

|Переменная|Источник данных|
|-|-|
|`{client_name}`|`appeals.applicant_full_name`|
|`{appeal_number}`|`appeals.appeal_number`|
|`{appeal_date}`|`appeals.created_at` (форматированная дата)|
|`{resolution_date}`|Текущая дата|
|`{responsible_name}`|`users.full_name` (ответственный сотрудник)|
|`{department_name}`|`departments.name` (ДУКО)|
|`{contact_phone}`|Контактный телефон ДУКО|
|`{contact_email}`|Контактный email ДУКО|

**Формат переменных:** `{variable_name}` (в фигурных скобках).

**Обработка отсутствующих данных:**

- Если значение переменной = NULL: заменяется на `[не указано]`

- Если переменная неизвестна: остаётся без изменений (отображается как есть)

**Обоснование:** Автоматизация заполнения ответов, снижение ошибок ручного ввода.

**Исключения:** Супервизор может создавать дополнительные переменные через администраторский интерфейс.

---

### BR-DECIDE-08: Каналы доведения ответа

**Правило:** Канал доведения ответа определяется полем `communication_channel` в обращении и приоритетом клиента:

|Канал|Условие использования|Время доставки|
|-|-|-|
|Email|`client_email != NULL`|1–5 минут|
|SMS|Для коротких уведомлений (до 160 символов)|1–3 минуты|
|Личный кабинет|Клиент зарегистрирован в интернет-банкинге|Мгновенно|
|Письменное уведомление|Запрос клиента или регуляторное обращение|3–7 рабочих дней|
|Телефонный звонок|Для VIP-клиентов или критичных ситуаций|По графику|

**Множественные каналы:**

- Для регуляторных обращений: обязательно письменное уведомление + email (дублирование)

- Для VIP-клиентов: личный кабинет + SMS-уведомление

**Обоснование:** Обеспечение гарантированной доставки с учётом предпочтений клиента.

**Исключения:** Супервизор может изменить канал доведения вручную.

---

### BR-DECIDE-09: Автосохранение черновиков

**Правило:** Система автоматически сохраняет черновик решения и ответа каждые 2 минуты, если:

- Заполнено хотя бы одно поле («Внутреннее решение» или «Текст ответа клиенту»)

- Карточка обращения остаётся открытой

- Пользователь не активировал кнопку «Отмена» или не закрыл карточку

**Хранение черновиков:**

- Таблица `appeal_drafts`

- Один черновик на пользователя и обращение (при повторном сохранении — UPDATE)

- Срок хранения: 7 дней (автоматическое удаление устаревших черновиков)

**Восстановление:**

- При повторном открытии карточки обращения система автоматически загружает последний черновик

- Отображается информация: «Восстановлен черновик от [дата, время]»

**Обоснование:** Предотвращение потери данных при сбоях, отключениях, случайном закрытии браузера.

**Исключения:** Нет.

---

### BR-DECIDE-10: Валидация длины полей

**Правило:** Система применяет следующие ограничения на длину текстовых полей:

|Поле|Минимум|Максимум|Обоснование|
|-|-|-|-|
|Внутреннее решение|20 символов|5000 символов|Обеспечение содержательности решения|
|Текст ответа клиенту|50 символов|10000 символов|Полнота и читаемость ответа|

**Отображение:**

- Счётчик символов под полем: «[N] / [MAX]»

- Предупреждение при приближении к лимиту: «Осталось [N] символов»

- Блокировка ввода при достижении максимума

**Обоснование:** Баланс между полнотой информации и читаемостью.

**Исключения:** Супервизор может отключить валидацию длины для конкретного обращения.

### 10.6. Метрики для мониторинга

- **Количество закрытых обращений за период** (по сотрудникам, по подразделению)

- **Процент соблюдения SLA** (target: >= 99%)

- **Среднее время обработки** (от принятия в работу до доведения ответа)

- **Процент использования шаблонов** vs произвольных ответов

- **Топ-5 используемых шаблонов** (для оптимизации справочника)

- **Процент успешной доставки** (delivery_status = 'delivered') target: >= 98%

- **Количество обращений с ошибками доставки** (для выявления проблем интеграций)

- **Среднее время доставки ответа** (по каналам)
