---

## 1. Паспорт Use Case

- **Идентификатор:** UC-REFUSE-01

- **Название:** Отказ ответственного сотрудника от обработки обращения

- **Актор:** Ответственный сотрудник ДУКО (Группа обработки обращений клиентов)

- **Цель:** Официально отказаться от назначенного обращения с указанием причины для снятия ответственности, возврата обращения в пул перераспределения, фиксации причины отказа в аудите и уведомления заинтересованных лиц

- **Триггер:** Пользователь открыл карточку обращения в статусе «Назначено» или «В работе» и принял решение не обрабатывать данное обращение

- **Частота использования:** 5–20 раз в день на одного сотрудника

- **Приоритет:** Средний (важно для корректного распределения рабочей нагрузки и предотвращения простоя обращений)

- **Уровень:** Пользовательская задача (User Goal Level)

---

## 2. Предусловия

- Обращение существует в системе с уникальным идентификатором

- Текущий статус обращения — «Назначено» или «В работе»

- Пользователь аутентифицирован в системе

- Пользователь является ответственным сотрудником, назначенным на данное обращение (поле `responsible_user_id` содержит ID текущего пользователя)

- Пользователь имеет роль «Ответственный сотрудник ДУКО» с правами на отказ от обработки обращения

- Карточка обращения открыта и отображена пользователю (выполнен UC-01 «Просмотр детализации обращения»)

- База данных доступна и транзакции выполняются корректно

- В системе существует справочник причин отказа от обращения (таблица `refusal_reasons`)

- Роль супервизора подразделения определена и активна

---

## 3. Основной поток (Main Flow)

### Фаза 1: Инициация отказа от обращения

1. **Пользователь** активирует действие «Отказаться от обращения».

2. **Система** проверяет текущий статус обращения в базе данных.

3. **Система** подтверждает, что статус обращения — «Назначено» или «В работе».

4. **Система** проверяет, что пользователь является назначенным ответственным (сравнивает `current_user_id` с `responsible_user_id` обращения).

5. **Система** отображает диалоговое окно «Отказ от обработки обращения».

### Фаза 2: Ввод причины отказа

1. **Система** загружает из базы данных справочник причин отказа (таблица `refusal_reasons`, поле `is_active = true`).

2. **Система** отображает список типовых причин отказа:

    - «Недостаточная квалификация / профиль не подходит»

    - «Конфликт интересов с клиентом»

    - «Перегрузка (превышен лимит обращений)»

    - «Отсутствие необходимой информации для обработки»

    - «Технические проблемы / отсутствие доступа к системам»

    - «Другое (требуется указать причину)»

1. **Пользователь** выбирает одну причину из списка.

2. **Пользователь** вводит текстовый комментарий в поле «Комментарий к отказу» (обязательное поле, минимум 10 символов).

3. **Пользователь** подтверждает действие (активирует кнопку подтверждения отказа).

### Фаза 3: Валидация введённых данных

1. **Система** проверяет, что выбрана причина отказа.

2. **Система** проверяет, что заполнено поле «Комментарий к отказу» (длина >= 10 символов).

3. **Система** проверяет, что комментарий не содержит запрещённых символов или недопустимого содержимого (profanity filter, SQL injection patterns).

### Фаза 4: Обновление данных обращения

1. **Система** начинает транзакцию базы данных.

2. **Система** сохраняет текущие значения полей для аудита:

    - `previous_status` = текущий статус

    - `previous_responsible_user_id` = ID текущего ответственного

1. **Система** обновляет запись обращения:

    - Устанавливает поле `status` = «Ожидает перераспределения»

    - Устанавливает поле `responsible_user_id` = NULL (снимает назначение)

    - Устанавливает поле `refusal_reason_id` = ID выбранной причины

    - Устанавливает поле `refusal_comment` = текст комментария пользователя

    - Устанавливает поле `refused_by_user_id` = ID текущего пользователя

    - Устанавливает поле `refused_at` = текущая дата и время (timestamp)

    - Устанавливает поле `last_modified_at` = текущая дата и время

    - Устанавливает поле `last_modified_by` = ID текущего пользователя

    - Увеличивает счётчик отказов: `refusal_count = refusal_count + 1`

1. **Система** проверяет значение `refusal_count`:

    - Если `refusal_count >= 3`, устанавливает флаг `requires_supervisor_review = true` (обращение требует проверки супервизором перед повторным назначением)

### Фаза 5: Фиксация в протоколе обработки

1. **Система** создаёт запись в протоколе обработки (таблица `appeal_audit_log`):

    - `appeal_id` = ID текущего обращения

    - `action_type` = «refused»

    - `action_description` = «Отказ от обработки обращения»

    - `performed_by_user_id` = ID текущего пользователя

    - `performed_at` = текущая дата и время

    - `stage` = «redistribution»

    - `previous_status` = сохранённый статус (шаг 15)

    - `new_status` = «Ожидает перераспределения»

    - `refusal_reason` = текст выбранной причины

    - `refusal_comment` = текст комментария пользователя

    - `metadata` = JSON с дополнительными данными (IP-адрес, session ID, user agent)

### Фаза 6: Обновление метрик пользователя

1. **Система** обновляет статистику сотрудника (таблица `user_statistics`):

    - Увеличивает счётчик `refusal_count` для текущего пользователя

    - Обновляет `last_refusal_at` = текущая дата и время

1. **Система** проверяет лимит отказов:

    - Если `refusal_count` за текущий месяц >= настроенного порога (например, 20), создаёт запись в таблице `supervisor_alerts` для проверки супервизором

### Фаза 7: Отправка уведомлений

1. **Система** определяет супервизора подразделения (запрос к таблице `departments` → `supervisor_user_id`).

2. **Система** создаёт уведомление для супервизора (таблица `notifications`):

    - `recipient_user_id` = ID супервизора

    - `notification_type` = «appeal_refused»

    - `appeal_id` = ID текущего обращения

    - `message` = «Сотрудник [ФИО] отказался от обработки обращения №[номер]. Причина: [причина]. Требуется перераспределение.»

    - `created_at` = текущая дата и время

    - `is_read` = false

    - `priority` = «high» (если `refusal_count >= 3`) или «normal»

1. **Система** отправляет push-уведомление супервизору (через WebSocket или push-service).

2. **Система** отправляет email супервизору (если включена настройка `email_notifications_enabled`).

### Фаза 8: Остановка таймеров SLA (если обращение было в работе)

1. **Система** проверяет, был ли установлен `work_started_at` (обращение было принято в работу):

    - Если `work_started_at` != NULL:

        - Рассчитывает время обработки: `processing_duration = refused_at - work_started_at`

        - Сохраняет `processing_duration` в поле обращения

        - Фиксирует в протоколе время, затраченное на обработку

    - Если `work_started_at` = NULL:

        - Пропускает расчёт (обращение не было принято в работу)

### Фаза 9: Завершение операции

1. **Система** фиксирует транзакцию в базе данных (COMMIT).

2. **Система** закрывает диалоговое окно отказа.

3. **Система** отображает уведомление пользователю: «Вы отказались от обращения №[номер]. Обращение отправлено на перераспределение.»

4. **Система** выполняет одно из действий (согласно настройке `refusal_redirect_behavior`):

    - Вариант A: Закрывает карточку обращения и возвращает пользователя к реестру обращений (UC-LIST-01)

    - Вариант B: Остаётся на карточке обращения в режиме read-only с отображением факта отказа

1. **Система** удаляет обращение из списка «Мои обращения» текущего пользователя.

---

## 4. Постусловия

### Успешное завершение:

- Статус обращения в базе данных изменён на «Ожидает перераспределения»

- Поле `responsible_user_id` установлено в NULL (назначение снято)

- Причина отказа и комментарий зафиксированы в полях `refusal_reason_id` и `refusal_comment`

- Поля `refused_by_user_id` и `refused_at` содержат данные об отказавшемся сотруднике и времени отказа

- Счётчик отказов обращения увеличен (`refusal_count`)

- В протоколе обработки создана запись о отказе с фиксацией причины, исполнителя, даты и времени

- Супервизор получил уведомление о необходимости перераспределения обращения

- Обращение удалено из списка «Мои обращения» отказавшегося сотрудника

- Обращение доступно для назначения другому ответственному (появилось в пуле необработанных обращений)

- Если обращение было в работе, зафиксировано время обработки до отказа

- Статистика сотрудника обновлена (увеличен счётчик отказов)

- Если достигнут лимит отказов обращения (>= 3), установлен флаг `requires_supervisor_review`

### Состояние системы:

- Транзакция базы данных успешно завершена (committed)

- Целостность данных сохранена

- Аудиторский след зафиксирован

- Уведомления отправлены заинтересованным лицам

- Пользователь возвращён к реестру или остаётся на карточке в режиме просмотра

- Обращение готово к перераспределению супервизором (UC-ASSIGN-01)

## 5. Альтернативные потоки

### A1: Пользователь отменяет отказ в диалоговом окне

- **Точка ветвления:** Между шагом 9 и шагом 10

- **Условие:** Пользователь активирует кнопку «Отмена» в диалоговом окне отказа

- **Действия:**

    1. Система закрывает диалоговое окно без сохранения данных

    2. Система не выполняет шаги 11–29

    3. Карточка обращения остаётся в исходном состоянии (статус и назначение не изменены)

    4. Пользователь может продолжить работу с обращением или повторить попытку отказа

    5. Сценарий завершается

### A2: Обращение уже переназначено другому сотруднику

- **Точка ветвления:** Шаг 4

- **Условие:** Поле `responsible_user_id` не совпадает с `current_user_id` (обращение переназначено между открытием карточки и попыткой отказа)

- **Действия:**

    1. Система прерывает выполнение операции (не отображает диалог отказа)

    2. Система отображает уведомление: «Обращение №[номер] уже переназначено другому сотруднику. Вы больше не являетесь ответственным.»

    3. Система отображает информацию о текущем ответственном сотруднике (ФИО, должность)

    4. Система автоматически обновляет данные карточки обращения из базы данных

    5. Система переводит карточку в режим read-only

    6. Система предлагает действие: «Вернуться к списку обращений»

    7. Сценарий завершается

### A3: Статус обращения изменён на последующий этап

- **Точка ветвления:** Шаг 3

- **Условие:** Текущий статус обращения — «На доведении», «Закрыто» или «Архив» (обращение прошло дальше по процессу)

- **Действия:**

    1. Система прерывает выполнение операции

    2. Система отображает уведомление: «Обращение №[номер] уже находится на этапе "[текущий статус]". Отказ от обработки невозможен.»

    3. Система автоматически обновляет карточку обращения из базы данных

    4. Система отображает актуальный статус и историю обработки

    5. Система переводит все поля в режим read-only

    6. Сценарий завершается

### A4: Попытка отказа при статусе, не допускающем отказ

- **Точка ветвления:** Шаг 3

- **Условие:** Статус обращения — «Готово к регистрации», «Зарегистрировано», «На аудите» или другой статус, не входящий в список разрешённых

- **Действия:**

    1. Система прерывает выполнение операции

    2. Система отображает уведомление: «Отказ от обращения возможен только в статусах "Назначено" или "В работе". Текущий статус: "[текущий статус]".»

    3. Система остаётся на карточке обращения

    4. Кнопка «Отказаться от обращения» остаётся недоступной или скрытой

    5. Сценарий завершается

### A5: Выбрана причина «Другое» без указания комментария

- **Точка ветвления:** Шаг 12

- **Условие:** Пользователь выбрал причину «Другое (требуется указать причину)», но не заполнил поле комментария или ввёл менее 10 символов

- **Действия:**

    1. Система не закрывает диалоговое окно

    2. Система отображает валидационное сообщение под полем комментария: «Поле обязательно для заполнения. Минимальная длина: 10 символов.»

    3. Система подсвечивает поле комментария (индикация ошибки)

    4. Кнопка подтверждения отказа остаётся неактивной до устранения ошибки

    5. Пользователь может исправить ошибку или отменить отказ

    6. Основной поток возвращается к шагу 9

### A6: Обращение с высоким приоритетом или VIP-клиентом

- **Точка ветвления:** Шаг 5 (перед отображением диалога)

- **Условие:** Обращение имеет флаг `is_high_priority = true` или `client_segment = 'VIP'`

- **Действия:**

    1. Система отображает предупреждающее сообщение в диалоговом окне: «⚠ Внимание! Обращение имеет высокий приоритет / относится к VIP-клиенту. Отказ от обработки должен быть согласован с супервизором.»

    2. Система добавляет дополнительное обязательное поле: «Согласовано с супервизором (ФИО, дата)»

    3. Пользователь заполняет информацию о согласовании

    4. Система валидирует заполнение дополнительного поля (шаг 12)

    5. Система добавляет информацию о согласовании в запись протокола (шаг 18)

    6. Основной поток продолжается с шага 6

### A7: Достигнут лимит отказов сотрудника за период

- **Точка ветвления:** Шаг 20

- **Условие:** Счётчик отказов сотрудника за текущий месяц достиг или превысил настроенный лимит (например, 20 отказов)

- **Действия:**

    1. Система выполняет все шаги основного потока (отказ разрешён)

    2. Система создаёт дополнительную запись в таблице `supervisor_alerts`:

        - `alert_type` = «high_refusal_rate»

        - `user_id` = ID текущего пользователя

        - `message` = «Сотрудник [ФИО] превысил допустимый лимит отказов за месяц ([количество] отказов). Требуется проверка загрузки и компетенций.»

        - `created_at` = текущая дата и время

    1. Система отправляет уведомление супервизору с повышенным приоритетом

    2. Система логирует событие в журнал HR-аналитики для анализа эффективности сотрудника

    3. Основной поток завершается нормально

## 6. Исключительные ситуации (Exception Flows)

### E1: Ошибка подключения к базе данных

- **Точка возникновения:** Шаг 2, 6, 14, 16, 18, 19, 21

- **Условие:** Потеря соединения с базой данных или таймаут запроса

- **Действия:**

    1. Система прерывает выполнение операции

    2. Система выполняет ROLLBACK транзакции (если она была начата на шаге 14)

    3. Система отображает уведомление: «Не удалось отказаться от обращения. Ошибка подключения к базе данных. Попробуйте повторить операцию через несколько секунд.»

    4. Система логирует ошибку с уровнем ERROR:

        - Тип ошибки: DATABASE_CONNECTION_FAILED

        - Appeal ID

        - User ID

        - Action: REFUSE_APPEAL

        - Timestamp

        - Текст ошибки БД

    1. Система отправляет алерт команде DevOps

    2. Система оставляет диалоговое окно открытым

    3. Пользователь может повторить попытку отказа или отменить операцию

    4. Сценарий завершается

### E2: Нарушение целостности данных (конкурентное изменение)

- **Точка возникновения:** Шаг 26 (COMMIT транзакции)

- **Условие:** Другой процесс одновременно изменил запись обращения, возникла блокировка или конфликт версий

- **Действия:**

    1. Система получает ошибку CONCURRENT_MODIFICATION или LOCK_TIMEOUT

    2. Система выполняет ROLLBACK транзакции

    3. Система повторяет попытку выполнения транзакции (retry logic, максимум 3 попытки с интервалом 500ms)

    4. Если после 3 попыток ошибка сохраняется:

        - Система отображает уведомление: «Не удалось отказаться от обращения. Данные обращения были изменены другим пользователем. Обновите карточку и повторите попытку.»

        - Система логирует ошибку с уровнем WARNING

        - Система предлагает действие: «Обновить карточку обращения»

    1. Пользователь может обновить карточку и повторить отказ

    2. Сценарий завершается

### E3: Валидация обнаружила запрещённое содержимое в комментарии

- **Точка возникновения:** Шаг 13

- **Условие:** Комментарий содержит недопустимые паттерны (SQL injection, XSS, оскорбления)

- **Действия:**

    1. Система не закрывает диалоговое окно

    2. Система отображает валидационное сообщение: «Комментарий содержит недопустимые символы или выражения. Пожалуйста, измените текст.»

    3. Система подсвечивает поле комментария (индикация ошибки)

    4. Система логирует попытку ввода запрещённого содержимого (SECURITY_AUDIT_LOG):

        - Event: INVALID_INPUT_DETECTED

        - User ID

        - Input field: refusal_comment

        - Detected pattern: [тип нарушения]

        - Timestamp

    1. Кнопка подтверждения отказа остаётся неактивной до устранения ошибки

    2. Пользователь корректирует текст комментария

    3. Основной поток возвращается к шагу 13

### E4: Недостаточно прав доступа

- **Точка возникновения:** Шаг 4

- **Условие:** Между открытием карточки и попыткой отказа у пользователя были отозваны права на отказ от обращений

- **Действия:**

    1. Система проверяет актуальные разрешения пользователя (permissions check)

    2. Система обнаруживает отсутствие прав `appeal.refuse`

    3. Система отменяет операцию

    4. Система отображает уведомление: «Недостаточно прав для отказа от обращения. Обратитесь к администратору системы.»

    5. Система логирует событие безопасности (SECURITY_AUDIT_LOG):

        - Event: UNAUTHORIZED_ACCESS_ATTEMPT

        - User ID

        - Appeal ID

        - Action: REFUSE_APPEAL

        - Timestamp

    1. Система закрывает диалоговое окно отказа

    2. Карточка обращения остаётся в исходном состоянии

    3. Сценарий завершается

### E5: Ошибка отправки уведомлений

- **Точка возникновения:** Шаг 23 или 24

- **Условие:** Служба уведомлений (WebSocket, email-service) недоступна или возвращает ошибку

- **Действия:**

    1. Система фиксирует основную транзакцию (COMMIT на шаге 26 выполняется успешно)

    2. Система логирует ошибку отправки уведомления с уровнем WARNING:

        - Тип ошибки: NOTIFICATION_DELIVERY_FAILED

        - Recipient: supervisor_user_id

        - Notification type: appeal_refused

        - Timestamp

    1. Система сохраняет уведомление в очередь повторной отправки (таблица `notification_queue`)

    2. Система планирует повторную попытку отправки через 5 минут (retry scheduler)

    3. Основной поток продолжается нормально (отказ считается успешным, несмотря на ошибку уведомления)

    4. Пользователь получает подтверждение отказа (шаг 28)

    5. Сценарий завершается успешно

### E6: Не найден супервизор подразделения

- **Точка возникновения:** Шаг 21

- **Условие:** Поле `supervisor_user_id` в таблице `departments` = NULL или ссылается на несуществующего/неактивного пользователя

- **Действия:**

    1. Система логирует предупреждение с уровнем WARNING:

        - Тип: SUPERVISOR_NOT_FOUND

        - Department ID

        - Appeal ID

        - Timestamp

    1. Система пропускает отправку уведомления супервизору (шаги 22–24)

    2. Система создаёт задачу для администратора системы: «Назначить супервизора для подразделения [название]»

    3. Система отправляет общее уведомление в канал администраторов

    4. Основной поток продолжается нормально (шаги 25–29)

    5. Отказ считается успешным, несмотря на отсутствие уведомления

    6. Сценарий завершается

## 7. Специальные требования

### 7.1. Производительность

- **NFR-PERF-01:** Время выполнения операции отказа от обращения (шаги 2–26) не должно превышать 3 секунд при нормальной нагрузке

- **NFR-PERF-02:** Время загрузки справочника причин отказа (шаг 6) не должно превышать 500 миллисекунд

- **NFR-PERF-03:** Операция должна выполняться в рамках одной атомарной транзакции БД с уровнем изоляции READ COMMITTED

- **NFR-PERF-04:** Система должна поддерживать одновременный отказ от обращений 20+ пользователями без деградации производительности

### 7.2. Безопасность

- **NFR-SEC-01:** Все операции отказа от обращения должны логироваться в аудиторском журнале с фиксацией:

    - ID пользователя, выполнившего отказ

    - Точной даты и времени (с точностью до миллисекунд)

    - Причины отказа и комментария

    - Предыдущего и нового статуса

    - IP-адреса клиента

    - Session ID

- **NFR-SEC-02:** Операция отказа доступна только пользователям с ролью «Ответственный сотрудник ДУКО» и правом `appeal.refuse`

- **NFR-SEC-03:** Проверка прав доступа должна выполняться на уровне API (backend), а не только на уровне UI

- **NFR-SEC-04:** Комментарий к отказу должен проходить валидацию на наличие SQL injection, XSS и других паттернов атак

- **NFR-SEC-05:** Система должна блокировать повторное выполнение операции при дублировании запроса (idempotency check)

- **NFR-SEC-06:** Все изменения статуса должны выполняться в рамках зашифрованного HTTPS-соединения

### 7.3. Надёжность

- **NFR-REL-01:** Система должна гарантировать атомарность операции (все шаги 14–18 выполняются полностью или не выполняются вовсе)

- **NFR-REL-02:** При сбое на любом шаге транзакции все изменения должны быть откачены (ROLLBACK)

- **NFR-REL-03:** Система должна поддерживать механизм повторных попыток (retry) при временных ошибках БД (максимум 3 попытки)

- **NFR-REL-04:** Целостность данных должна быть защищена от конкурентных изменений (optimistic locking или row-level locking)

- **NFR-REL-05:** Ошибка отправки уведомлений НЕ должна блокировать основную операцию отказа

### 7.4. Доступность

- **NFR-AVAIL-01:** Операция отказа должна быть доступна для выполнения с клавиатуры (последовательность Tab + Enter в диалоге)

- **NFR-AVAIL-02:** Валидационные сообщения должны быть доступны для screen readers (ARIA live regions)

- **NFR-AVAIL-03:** Интерфейс должен соответствовать стандарту WCAG 2.1 уровня AA

### 7.5. Отказоустойчивость

- **NFR-FAULT-01:** При недоступности службы уведомлений операция отказа должна завершиться успешно (уведомление — некритичный компонент)

- **NFR-FAULT-02:** Ошибки при отправке уведомлений должны логироваться отдельно и помещаться в очередь повторной отправки

- **NFR-FAULT-03:** При недоступности справочника причин отказа система должна предложить базовый набор причин (hardcoded fallback)

### 7.6. Удобство использования

- **NFR-UX-01:** Система должна предоставлять автозаполнение или подсказки при вводе комментария (типовые формулировки)

- **NFR-UX-02:** Диалоговое окно отказа должно сохранять введённые данные при случайном закрытии (session storage)

- **NFR-UX-03:** Система должна предупреждать пользователя о последствиях отказа (потеря назначения, влияние на статистику)

## 8. Бизнес-правила

### BR-REFUSE-01: Допустимые статусы для отказа

**Правило:** Операция «Отказаться от обращения» разрешена только для обращений в статусах «Назначено» или «В работе».

**Обоснование:** Обеспечивает логичность процесса — отказаться можно только от обращения, которое ещё не обработано и не передано дальше.

**Исключения:** Супервизор с правом `appeal.force_refuse` может отказать от обращения в любом статусе (для исключительных ситуаций).

---

### BR-REFUSE-02: Обязательность причины отказа

**Правило:** Пользователь обязан выбрать причину отказа из справочника и указать текстовый комментарий (минимум 10 символов).

**Обоснование:** Обеспечивает прозрачность процесса, позволяет анализировать причины отказов и оптимизировать распределение обращений.

**Исключения:** Нет.

---

### BR-REFUSE-03: Снятие назначения ответственного

**Правило:** При отказе от обращения поле `responsible_user_id` устанавливается в NULL, и обращение переходит в статус «Ожидает перераспределения».

**Обоснование:** Обращение должно быть доступно для назначения другому сотруднику через механизм перераспределения.

**Исключения:** Нет.

---

### BR-REFUSE-04: Счётчик отказов обращения

**Правило:** Каждый отказ увеличивает счётчик `refusal_count` обращения на 1.

**Обоснование:** Позволяет отслеживать «проблемные» обращения, которые неоднократно отклонялись сотрудниками.

**Действия при превышении лимита:**

- Если `refusal_count >= 3`, устанавливается флаг `requires_supervisor_review = true`

- Обращение может быть назначено повторно только после проверки и одобрения супервизором

- Супервизор должен оценить сложность обращения и назначить более опытного сотрудника или взять в работу сам

**Исключения:** Нет.

---

### BR-REFUSE-05: Лимит отказов сотрудника

**Правило:** Система отслеживает количество отказов сотрудника за календарный месяц.

**Пороговые значения:**

- **Предупреждение:** >= 15 отказов за месяц (уведомление супервизору)

- **Критический уровень:** >= 20 отказов за месяц (создание алерта для HR и супервизора)

**Обоснование:** Высокая частота отказов может указывать на:

- Неправильное распределение обращений (не по компетенции)

- Перегрузку сотрудника

- Недостаточную квалификацию

- Проблемы с мотивацией

**Действия:** Супервизор должен провести встречу с сотрудником для выяснения причин и корректировки нагрузки.

**Исключения:** Новые сотрудники (стаж < 1 месяца) имеют увеличенный лимит — до 30 отказов (период адаптации).

---

### BR-REFUSE-06: Уведомление супервизора

**Правило:** При каждом отказе от обращения система отправляет уведомление супервизору подразделения.

**Формат уведомления:**

- Приоритет: «normal» (если `refusal_count < 3`) или «high» (если `refusal_count >= 3`)

- Содержание: номер обращения, ФИО отказавшегося сотрудника, причина отказа, счётчик отказов обращения

**Обоснование:** Супервизор должен оперативно перераспределить обращение для соблюдения SLA.

**Исключения:** Если в настройках подразделения отключены уведомления (`notify_on_refusal = false`), уведомление не отправляется, но запись в `supervisor_alerts` создаётся.

---

### BR-REFUSE-07: Влияние на статистику сотрудника

**Правило:** Отказ от обращения влияет на следующие метрики сотрудника:

- Увеличивается счётчик `refusal_count` (персональная статистика)

- Учитывается в расчёте KPI «Процент отказов» = (refusal_count / assigned_count) × 100%

- Целевое значение KPI: < 10% отказов от общего числа назначенных обращений

**Обоснование:** Стимулирует ответственный подход к обработке назначенных обращений.

**Исключения:** Отказы по причинам «Конфликт интересов» и «Технические проблемы» не учитываются в негативных метриках KPI (классифицируются как объективные причины).

---

### BR-REFUSE-08: Остановка таймера SLA

**Правило:** Если обращение было принято в работу (статус «В работе»), при отказе система:

- Рассчитывает время обработки: `processing_duration = refused_at - work_started_at`

- Сохраняет `processing_duration` в истории обработки

- НЕ засчитывает это время в общий расчёт SLA обращения (время обнуляется для нового ответственного)

**Обоснование:** Новый ответственный должен получить полное время SLA для обработки, без учёта времени предыдущего сотрудника.

**Исключения:** Если обращение было в статусе «Назначено» (не было принято в работу), время не рассчитывается.

---

### BR-REFUSE-09: Блокировка повторного назначения тому же сотруднику

**Правило:** Сотрудник, отказавшийся от обращения, не может быть назначен на это обращение повторно в течение 7 календарных дней.

**Обоснование:** Предотвращение ситуации, когда обращение «ходит по кругу» между одними и теми же сотрудниками.

**Исключения:** Супервизор может вручную назначить обращение тому же сотруднику с указанием обоснования.

---

### BR-REFUSE-10: Приоритетные и VIP-обращения

**Правило:** Отказ от обращений с флагами `is_high_priority = true` или `client_segment = 'VIP'` требует:

- Обязательного указания дополнительной информации о согласовании с супервизором

- Немедленного уведомления супервизора с приоритетом «urgent»

- Ручного перераспределения супервизором (автоматическое назначение не применяется)

**Обоснование:** Обеспечивает особый контроль за обработкой критически важных обращений.

**Исключения:** Нет.

### 10.5. Метрики для мониторинга

- **Общее количество отказов за период** (по подразделению, по сотруднику)

- **Процент отказов от общего числа назначений** (target: < 10%)

- **Топ-5 причин отказов** (для оптимизации процесса назначения)

- **Среднее время от назначения до отказа** (для анализа скорости принятия решений)

- **Количество обращений с `refusal_count >= 3`** (проблемные обращения)

- **Среднее время перераспределения после отказа** (эффективность супервизора)

### 10.6. Рекомендации по автоматизации

**Автоматическое перераспределение:**

- Если `refusal_count < 3`, система может автоматически предложить следующего кандидата на основе:

    - Загрузки сотрудников (количество обращений в работе)

    - Профиля компетенций (соответствие категории обращения)

    - Истории успешной обработки аналогичных обращений

**Предиктивная аналитика:**
