## 1. Паспорт Use Case

- **Идентификатор:** UC-LIST-01

- **Название:** Поиск и навигация по реестру обращений ответственного сотрудника

- **Актор:**

    - Ответственный сотрудник ДУКО (основной актор)

    - Супервизор ДУКО (расширенные права)

- **Цель:** Найти конкретное обращение по критериям поиска (номер, ФИО заявителя, содержание), отфильтровать список обращений по статусу, типу заявителя, ответственности, отсортировать по дате регистрации или дедлайну для приоритизации работы, перейти к детализированному просмотру обращения или инициировать регистрацию нового обращения

- **Триггер:** Пользователь авторизовался в системе и перешёл в раздел «Кабинет ответственного» через главное меню или выбрал вкладку «Мои обращения» в навигации

- **Частота использования:** 20–50 раз в день на одного сотрудника

- **Приоритет:** Критический (основной инструмент управления рабочей нагрузкой и планирования работы)

- **Уровень:** Пользовательская задача (User Goal Level)

---

## 2. Предусловия

- Пользователь успешно аутентифицирован в системе

- Пользователь имеет роль «Ответственный сотрудник ДУКО» или «Супервизор ДУКО»

- База данных доступна и выполняет запросы корректно

- В системе существует минимум 1 обращение в любом статусе

- Пользователю назначено минимум 1 обращение (для фильтра «Мои обращения») ИЛИ пользователь имеет права супервизора для просмотра всех обращений подразделения

- Интеграционные сервисы (для синхронизации данных) доступны или предусмотрен fallback-режим

---

## 3. Основной поток (Main Flow)

### Фаза 1: Инициализация реестра обращений

1. **Пользователь** переходит в раздел «Кабинет ответственного» одним из способов:

    - Кликает на пункт меню «Мои обращения» в навигации

    - Выбирает вкладку «Кабинет ответственного» в header

    - Переходит по прямой ссылке (URL `/appeals`)

1. **Система** получает запрос на отображение реестра обращений.

2. **Система** проверяет аутентификацию пользователя:

    - Валидирует JWT-токен из заголовка `Authorization: Bearer <token>`

    - Проверяет срок действия токена

    - Извлекает `user_id` и `role` из токена

1. **Система** определяет роль пользователя и применяет политики видимости данных:

    - **Ответственный сотрудник ДУКО:**

        - Фильтр по умолчанию: «Мои обращения» (`WHERE responsible_user_id = current_user_id`)

        - Дополнительный доступ: обращения подразделения в режиме read-only (`WHERE department_id = user.department_id`)

    - **Супервизор ДУКО:**

        - Фильтр по умолчанию: «Мои обращения» (личные назначения)

        - Полный доступ: все обращения подразделения в режиме read-write

### Фаза 2: Загрузка данных с параметрами по умолчанию

1. **Система** формирует SQL-запрос с параметрами по умолчанию:

    ```SQL
    SELECT 
      appeal_id, appeal_number, client_type, communication_channel,
      status, substatus, applicant_full_name, appeal_content,
      created_at, registered_at, sla_deadline, responsible_user_id,
      is_overdue, sla_usage_percent
    FROM appeals
    WHERE responsible_user_id = ?
    ORDER BY sla_deadline ASC
    LIMIT 50 OFFSET 0
    ```

1. **Система** выполняет запрос к базе данных с таймаутом 10 секунд.

2. **Система** получает результаты запроса (массив записей обращений).

3. **Система** для каждой записи выполняет дополнительную обработку:

    - Вычисляет `sla_usage_percent = ((current_time - sla_started_at) / (sla_deadline - sla_started_at)) * 100`

    - Определяет `is_overdue = (current_time > sla_deadline)`

    - Обрезает `appeal_content` до 80 символов для колонки «Содержание (краткое)»

    - Загружает ФИО ответственного сотрудника (JOIN с таблицей `users`)

1. **Система** подсчитывает общее количество записей для пагинации:

    - SELECT COUNT(*) FROM appeals WHERE responsible_user_id = ?

1. **Система** рассчитывает счётчики для табов фильтрации:

    - «Все обращения»: COUNT(*) WHERE department_id = user.department_id

    - «Мои обращения»: COUNT(*) WHERE responsible_user_id = current_user_id

    - «Физ. лицо»: COUNT(*) WHERE client_type = 'individual'

    - «Юр. лицо»: COUNT(*) WHERE client_type = 'legal_entity'

    - По статусам: COUNT(*) для каждого статуса

### Фаза 3: Отображение интерфейса реестра

1. **Система** отображает экран «Кабинет ответственного» со следующей структурой:

    **Header страницы:**

    - Заголовок: «Кабинет ответственного»

    - Навигационные табы: «Мои обращения», «Дашборд», «Архив»

    **Toolbar (панель управления):**

    - Строка глобального поиска (full-width input, плейсхолдер: «🔍 Поиск по номеру, заявителю или содержанию...»)

    - Табы фильтрации (Segmented Controls):

        - Группа 1 (Ответственность): «Все обращения ([N])», «Мои обращения ([N])» (активен)

        - Группа 2 (Тип заявителя): «Все типы», «Физ. лицо», «Юр. лицо»

        - Группа 3 (Статус): «Все статусы», «В обработке», «Ожидает доведения», «Просрочено»

    - Кнопка «➕ СОЗДАТЬ ОБРАЩЕНИЕ»

    **Таблица обращений:**

    - Заголовки колонок с индикаторами сортировки:

        - № | Номер обращения | Тип клиента | Канал | Статус | Заявитель | Содержание (краткое) | Дата рег. ↕ | Дедлайн ↑ | Ответственный | Действия

    - Строки данных (50 записей)

    - Hover-эффект на строках

    - Цветовая индикация дедлайнов (зелёный/жёлтый/красный)

    **Footer (пагинация):**

    - Информация: «Показано 1-50 из [N] обращений»

    - Кнопки: «← Назад» | «Страница 1 из [M]» | «Вперед →»

    - Dropdown: «Показывать по: [10 / 25 / 50 / 100]»

1. **Система** устанавливает фокус на поле глобального поиска для быстрого начала работы.

2. **Система** сохраняет текущее состояние реестра в `sessionStorage`:

    ```JSON
    {
      "filters": {
        "responsibility": "my",
        "client_type": "all",
        "status": "all"
      },
      "sort": {
        "field": "sla_deadline",
        "direction": "ASC"
      },
      "pagination": {
        "page": 1,
        "limit": 50
      },
      "search": ""
    }
    ```

### Фаза 4: Взаимодействие с глобальным поиском

1. **Пользователь** вводит текст в строку глобального поиска (например, «Иванов»).

2. **Система** применяет debounce 300 миллисекунд (ожидает паузу в вводе).

3. **Система** после паузы выполняет поисковый запрос:

    ```SQL
    SELECT ... FROM appeals
    WHERE (
      appeal_number ILIKE '%Иванов%'
      OR applicant_full_name ILIKE '%Иванов%'
      OR appeal_content ILIKE '%Иванов%'
    )
    AND responsible_user_id = ?
    ORDER BY sla_deadline ASC
    LIMIT 50
    ```

1. **Система** валидирует поисковый запрос:

    - Проверяет длину (минимум 2 символа, максимум 200 символов)

    - Фильтрует специальные символы для защиты от SQL injection

1. **Система** обновляет таблицу с результатами поиска.

2. **Система** отображает информацию о результатах: «Найдено [N] обращений по запросу '[текст]'»

3. **Система** подсвечивает совпадающие фрагменты в результатах (опционально).

4. **Система** отображает кнопку очистки поиска (крестик справа от поля).

### Фаза 5: Применение фильтров

1. **Пользователь** кликает на таб «Все обращения» (Группа 1).

2. **Система** снимает фильтр по ответственному и выполняет запрос:

    ```SQL
    SELECT ... FROM appeals
    WHERE department_id = ?
    ORDER BY sla_deadline ASC
    LIMIT 50
    ```

1. **Система** проверяет права пользователя:

    - Если роль «Ответственный сотрудник»: применяет ограничение подразделением

    - Если роль «Супервизор»: отображает все обращения подразделения

1. **Система** обновляет счётчик таба: «Все обращения ([N])»

2. **Система** обновляет таблицу с новым набором данных.

3. **Система** сохраняет изменение фильтра в `sessionStorage`.

4. **Пользователь** дополнительно выбирает таб «Физ. лицо» (Группа 2).

5. **Система** применяет комбинированный фильтр:

    ```SQL
    WHERE department_id = ? AND client_type = 'individual'
    ```

1. **Система** обновляет таблицу и счётчики.

2. **Пользователь** выбирает таб «Просрочено» (Группа 3).

3. **Система** применяет дополнительное условие:

    ```SQL
    WHERE department_id = ? 
      AND client_type = 'individual'
      AND sla_deadline < CURRENT_TIMESTAMP
    ```

1. **Система** обновляет таблицу с просроченными обращениями.

2. **Система** отображает только обращения с красными дедлайнами.

### Фаза 6: Сортировка данных

1. **Пользователь** кликает на заголовок колонки «Дата рег.»

2. **Система** определяет текущее направление сортировки для этой колонки (по умолчанию: нет).

3. **Система** переключает направление сортировки:

    - Первый клик → ASC (↑)

    - Второй клик → DESC (↓)

    - Третий клик → сброс (возврат к сортировке по умолчанию «Дедлайн ASC»)

1. **Система** выполняет запрос с новой сортировкой:

    ```SQL
    ORDER BY registered_at ASC
    ```

1. **Система** обновляет индикатор направления в заголовке колонки (иконка ↑ или ↓).

2. **Система** перестраивает таблицу согласно новой сортировке.

3. **Система** сохраняет параметры сортировки в `sessionStorage`.

### Фаза 7: Работа с пагинацией

1. **Пользователь** кликает на кнопку «Вперед →» (переход на страницу 2).

2. **Система** вычисляет OFFSET для следующей страницы: `OFFSET = (page - 1) * limit = 1 * 50 = 50`

3. **Система** выполняет запрос с новым OFFSET:

    ```SQL
    LIMIT 50 OFFSET 50
    ```

1. **Система** загружает следующие 50 записей.

2. **Система** обновляет таблицу с новыми данными.

3. **Система** обновляет информацию пагинации: «Показано 51-100 из [N] обращений», «Страница 2 из [M]»

4. **Система** активирует кнопку «← Назад», деактивирует «Вперед →» если достигнута последняя страница.

5. **Система** сохраняет номер текущей страницы в `sessionStorage`.

### Фаза 8: Переход к детализации обращения

1. **Пользователь** кликает на номер обращения «№348678» (фиолетовая кликабельная ссылка).

2. **Система** сохраняет полное состояние реестра в `sessionStorage`:

    - Активные фильтры

    - Параметры сортировки

    - Текущая страница пагинации

    - Текст поиска

    - Позиция прокрутки таблицы

1. **Система** выполняет проверку прав доступа к обращению:

    - Проверяет, что пользователь имеет право просмотра (`appeal.view`)

    - Проверяет, что пользователь является ответственным ИЛИ имеет права супервизора

1. **Система** генерирует URL карточки обращения: `/appeals/{appeal_id}`

2. **Система** переходит к UC-VIEW-01 «Просмотр детализации обращения».

3. Сценарий приостанавливается (возврат к реестру произойдёт после закрытия карточки).

### Фаза 9: Возврат к реестру с восстановлением состояния

1. **Пользователь** возвращается к реестру (кнопка «Назад к реестру» в карточке обращения или навигация).

2. **Система** загружает сохранённое состояние из `sessionStorage`.

3. **Система** восстанавливает:

    - Активные фильтры (применяет к запросу)

    - Параметры сортировки

    - Текущую страницу пагинации

    - Текст поиска (заполняет поле)

    - Позицию прокрутки таблицы

1. **Система** выполняет запрос с восстановленными параметрами.

2. **Система** отображает реестр в том же состоянии, в котором пользователь его покинул.

## 4. Постусловия

### Успешное завершение:

- Реестр обращений отображён с таблицей данных

- Применены фильтры и сортировка согласно выбору пользователя

- Счётчики в табах фильтров отражают актуальное количество записей

- Просроченные дедлайны подсвечены красным цветом

- Пользователь получил список обращений для планирования работы

- Состояние реестра (фильтры, сортировка, пагинация, поиск) сохранено в `sessionStorage`

- При переходе к карточке обращения состояние реестра зафиксировано для последующего восстановления

- Пользователь может выполнить следующие действия: просмотр обращения, создание нового обращения, изменение фильтров

### Состояние системы:

- Никакие данные в базе данных не изменились (режим read-only)

- Действия пользователя НЕ логируются в основной аудит (просмотр списка не фиксируется как критичное событие)

- Состояние фильтров и сортировки сохранено на стороне клиента (`sessionStorage`)

- Таймеры SLA продолжают работу

- Система готова к выполнению связанных Use Cases (UC-VIEW-01, UC-00)

## 5. Альтернативные потоки

### A1: Нет обращений для отображения (пустой результат)

- **Точка ветвления:** Шаг 7

- **Условие:** Запрос к БД вернул 0 записей (у пользователя нет назначенных обращений или фильтры дали пустой результат)

- **Действия:**

    1. Система отображает экран реестра со всеми элементами управления (поиск, фильтры, кнопки)

    2. Система показывает заглушку вместо таблицы:

        - Иконка 📋 (пустой список)

        - Текст: «У вас пока нет назначенных обращений» (если фильтр «Мои обращения»)

        - ИЛИ «Нет обращений, соответствующих выбранным фильтрам» (если применены фильтры)

        - Подсказка: «Обращения появятся здесь после назначения ответственным» или «Попробуйте изменить критерии фильтрации»

    1. Таб «Мои обращения» показывает счётчик «(0)»

    2. Кнопка «СОЗДАТЬ ОБРАЩЕНИЕ» остаётся активной

    3. Система предлагает действия:

        - Кнопка «Сбросить все фильтры» (если фильтры применены)

        - Кнопка «Показать все обращения подразделения» (переключает на таб «Все обращения»)

    1. Сценарий продолжается, система ожидает действий пользователя

### A2: Поиск не дал результатов

- **Точка ветвления:** Шаг 18

- **Условие:** Поисковый запрос не совпал ни с одной записью

- **Действия:**

    1. Система отображает сообщение в области таблицы:

        - «🔍 По запросу '[текст]' ничего не найдено»

        - «Попробуйте изменить поисковый запрос или сбросить фильтры»

    1. Система отображает кнопки:

        - «✕ Очистить поиск» (очищает поле, возвращает полный список)

        - «Сбросить фильтры» (если фильтры применены)

    1. Счётчик показывает: «Найдено 0 обращений»

    2. Фильтры остаются активными (поиск комбинируется с фильтрами)

    3. Пользователь может изменить запрос или очистить поле поиска

    4. Основной поток продолжается с шага 14

### A3: Переключение на «Все обращения» при недостаточных правах

- **Точка ветвления:** Шаг 22

- **Условие:** Пользователь с ролью «Ответственный сотрудник» (не Супервизор) кликает на «Все обращения»

- **Действия:**

    1. Система проверяет роль пользователя

    2. Система применяет ограничение: показывает только обращения подразделения (`WHERE department_id = user.department_id`)

    3. Система НЕ отображает модальное окно с предупреждением (тихое ограничение)

    4. Таб «Все обращения» остаётся активным, показывает ограниченную выборку

    5. Счётчик таба отражает количество доступных обращений подразделения

    6. Основной поток продолжается с шага 26

### A4: Комбинированные фильтры не дали результатов

- **Точка ветвления:** Шаг 33

- **Условие:** Применено несколько фильтров, но выборка пустая (например, «Мои обращения + Юр. лицо + Просрочено» = 0)

- **Действия:**

    1. Система отображает информационное сообщение:

        - «📊 Нет обращений, соответствующих выбранным фильтрам»

        - Активные фильтры отображаются как badges: [Мои обращения] [Юр. лицо] [Просрочено] [✕]

    1. Система предлагает действия:

        - Кнопка «Сбросить все фильтры» (возврат к параметрам по умолчанию)

        - Кнопка «Снять последний фильтр» (удаляет последний применённый фильтр)

    1. Пользователь может:

        - Снять отдельные фильтры (кликнув на ✕ в badge)

        - Сбросить все фильтры

        - Изменить комбинацию фильтров

    1. Основной поток продолжается с шага 22

### A5: Пользователь кликает «СОЗДАТЬ ОБРАЩЕНИЕ»

- **Точка ветвления:** После шага 13 (в любой момент просмотра реестра)

- **Условие:** Пользователь нажал кнопку «➕ СОЗДАТЬ ОБРАЩЕНИЕ»

- **Действия:**

    1. Система сохраняет текущее состояние реестра (фильтры, сортировка, пагинация, поиск) в `sessionStorage`

    2. Система переходит к UC-00 «Регистрация нового обращения»

    3. Текущий сценарий приостанавливается

    4. После завершения регистрации (успешной или отменённой):

        - Система возвращает пользователя к реестру

        - Восстанавливает сохранённые фильтры и сортировку

        - Обновляет данные (новое обращение появляется в списке, если соответствует фильтрам)

    1. Сценарий возобновляется с шага 56

### A6: Изменение лимита пагинации

- **Точка ветвления:** После шага 13 (в footer пагинации)

- **Условие:** Пользователь выбирает новое значение в dropdown «Показывать по: [10 / 25 / 50 / 100]»

- **Действия:**

    1. Пользователь выбирает, например, «100»

    2. Система обновляет параметр `limit = 100`

    3. Система сбрасывает пагинацию на первую страницу (`page = 1`, `OFFSET = 0`)

    4. Система выполняет запрос с новым лимитом:

        ```SQL
        LIMIT 100 OFFSET 0
        ```

    1. Система обновляет таблицу (отображает до 100 записей)

    2. Система обновляет информацию пагинации: «Показано 1-100 из [N] обращений»

    3. Система пересчитывает количество страниц

    4. Система сохраняет новый лимит в `sessionStorage`

    5. Основной поток продолжается с отображением большего количества записей

### A7: Переход на конкретную страницу пагинации

- **Точка ветвления:** Шаг 42

- **Условие:** Пользователь кликает на номер страницы (если отображается список страниц: «1 2 3 ... 10»)

- **Действия:**

    1. Пользователь кликает на «Страница 5»

    2. Система вычисляет OFFSET: `OFFSET = (5 - 1) * 50 = 200`

    3. Система выполняет запрос: `LIMIT 50 OFFSET 200`

    4. Система загружает записи 201-250

    5. Система обновляет таблицу

    6. Система обновляет информацию пагинации: «Показано 201-250 из [N] обращений», «Страница 5 из [M]»

    7. Система сохраняет номер страницы в `sessionStorage`

    8. Основной поток продолжается с новой страницей

### A8: Попытка сортировки несортируемой колонки

- **Точка ветвления:** Шаг 35

- **Условие:** Пользователь кликает на заголовок колонки, которая не поддерживает сортировку (например, «Действия»)

- **Действия:**

    1. Система проверяет конфигурацию колонки (атрибут `sortable: false`)

    2. Система игнорирует клик (не выполняет запрос сортировки)

    3. Курсор над заголовком несортируемой колонки остаётся стандартным (не меняется на `pointer`)

    4. Таблица не изменяется

    5. Сценарий продолжается без изменений

### A9: Поиск возвращает слишком много результатов (>1000)

- **Точка ветвления:** Шаг 18

- **Условие:** Поисковый запрос слишком общий и вернул более 1000 записей

- **Действия:**

    1. Система выполняет запрос и получает COUNT(*) > 1000

    2. Система ограничивает выборку первыми 1000 записями

    3. Система отображает предупреждающее сообщение:

        - «⚠️ Найдено более 1000 обращений. Показаны первые 1000 результатов. Пожалуйста, уточните поисковый запрос для более точных результатов.»

    1. Система отображает таблицу с первыми 1000 записями

    2. Пагинация работает в пределах 1000 записей (20 страниц по 50)

    3. Пользователь может уточнить запрос для сужения результатов

    4. Основной поток продолжается с ограниченной выборкой

### A10: Сброс всех фильтров

- **Точка ветвления:** После применения фильтров (шаги 22–34)

- **Условие:** Пользователь кликает на кнопку «Сбросить все фильтры» или выбирает «Все» во всех группах фильтров

- **Действия:**

    1. Система сбрасывает все фильтры к значениям по умолчанию:

        - Ответственность: «Мои обращения»

        - Тип клиента: «Все типы»

        - Статус: «Все статусы»

    1. Система очищает поле поиска (если было заполнено)

    2. Система сбрасывает сортировку к «Дедлайн ASC»

    3. Система сбрасывает пагинацию на первую страницу

    4. Система выполняет запрос с параметрами по умолчанию (повтор шага 5)

    5. Система обновляет таблицу с полным списком обращений пользователя

    6. Система обновляет счётчики табов

    7. Система очищает сохранённое состояние в `sessionStorage`

    8. Основной поток продолжается с параметрами по умолчанию

### A11: Автоматическое обновление данных реестра

- **Точка ветвления:** В любой момент просмотра реестра (фоновый процесс)

- **Условие:** Прошло 60 секунд с момента последней загрузки данных ИЛИ система получила WebSocket-уведомление об изменении обращений

- **Действия:**

    1. Система выполняет фоновый запрос для проверки изменений (polling или WebSocket)

    2. Если обнаружены изменения (новые обращения, изменение статусов):

        - Система отображает toast-уведомление: «🔔 Обнаружены изменения в реестре обращений. [Обновить сейчас]»

    1. Если пользователь кликает «Обновить сейчас»:

        - Система перезагружает данные с текущими фильтрами и сортировкой

        - Сохраняет позицию прокрутки

        - Обновляет таблицу и счётчики

        - Подсвечивает изменённые/новые строки (на 3 секунды)

    1. Если пользователь игнорирует уведомление:

        - Уведомление автоматически скрывается через 10 секунд

        - Данные остаются неизменными до следующего ручного обновления

    1. Основной поток продолжается с опционально обновлёнными данными

## 6. Исключительные ситуации (Exception Flows)

### E1: Критическая ошибка базы данных при загрузке реестра

- **Точка возникновения:** Шаг 6

- **Условие:** Потеря соединения с БД, таймаут запроса (>10 секунд), ошибка SQL

- **Действия:**

    1. Система прерывает выполнение запроса

    2. Система логирует ошибку с уровнем CRITICAL:

        - Тип: DATABASE_CONNECTION_FAILED

        - Operation: LOAD_APPEALS_LIST

        - User ID

        - Error message

        - Timestamp

        - Stack trace

    1. Система отправляет алерт команде DevOps

    2. Система отображает экран ошибки:

        - Заголовок: «Не удалось загрузить реестр обращений»

        - Сообщение: «Сервис временно недоступен. Попробуйте обновить страницу через несколько секунд.»

        - Код ошибки: DB-001

    1. Система предлагает действия:

        - Кнопка «🔄 ОБНОВИТЬ» (перезагружает страницу, повтор с шага 1)

        - Кнопка «📞 СООБЩИТЬ О ПРОБЛЕМЕ» (открывает форму обратной связи)

    1. Сценарий завершается

### E2: Таймаут загрузки данных (медленный запрос)

- **Точка возникновения:** Шаг 6

- **Условие:** Запрос выполняется дольше 10 секунд из-за большого объёма данных или высокой нагрузки на БД

- **Действия:**

    1. Система отображает индикатор загрузки с сообщением:

        - «⏳ Загрузка большого объема данных... Это может занять до 30 секунд.»

    1. Система продолжает ожидание до 30 секунд (extended timeout)

    2. Если данные загружаются в течение 30 секунд:

        - Система отображает таблицу

        - Показывает предупреждение: «⚠️ Загрузка заняла больше времени, чем обычно. Попробуйте применить фильтры для ускорения работы.»

    1. Если таймаут превышен (>30 секунд):

        - Система отображает ошибку: «⏱️ Превышено время ожидания. Попробуйте уменьшить выборку, применив фильтры.»

        - Код ошибки: DB-002

    1. Система логирует событие с уровнем WARNING

    2. Система предлагает:

        - Кнопка «Показать только мои обращения» (быстрый фильтр)

        - Кнопка «Обновить страницу»

    1. Сценарий продолжается с уменьшенной выборкой или завершается

### E3: Ошибка прав доступа при переходе к обращению

- **Точка возникновения:** Шаг 52

- **Условие:** Между загрузкой реестра и кликом на обращение права пользователя были изменены или обращение было переназначено

- **Действия:**

    1. Система проверяет права доступа к обращению

    2. Система обнаруживает несоответствие (403 Forbidden)

    3. Система НЕ переходит к карточке обращения

    4. Система отображает toast-уведомление:

        - «⚠️ Недостаточно прав для просмотра обращения №[номер]. Возможно, обращение было переназначено другому сотруднику.»

    1. Система автоматически обновляет таблицу реестра (перезапрашивает данные)

    2. Обращение исчезает из списка (если переназначено) или остаётся с обновлённым ответственным

    3. Система логирует попытку доступа (SECURITY_AUDIT_LOG):

        - Event: UNAUTHORIZED_ACCESS_ATTEMPT

        - User ID, Appeal ID, Timestamp

    1. Основной поток продолжается с обновлёнными данными

### E4: Ошибка парсинга даты в фильтре

- **Точка возникновения:** Если пользователь вводит дату вручную в расширенном фильтре (опциональная функция)

- **Условие:** Пользователь ввёл невалидный формат даты (например, «32.13.2026»)

- **Действия:**

    1. Система пытается распарсить дату

    2. Система обнаруживает ошибку формата

    3. Система отображает валидационное сообщение под полем:

        - «❌ Неверный формат даты. Используйте формат: ДД.ММ.ГГГГ»

    1. Система подсвечивает поле с ошибкой

    2. Система НЕ применяет фильтр (блокирует кнопку «Применить»)

    3. Пользователь корректирует дату

    4. Основной поток возвращается к шагу применения фильтра

### E5: Превышение квоты запросов (rate limiting)

- **Точка возникновения:** Шаг 6 или шаг 16 (поиск)

- **Условие:** Пользователь выполнил более 100 запросов в минуту (защита от злоупотреблений или ошибок клиента)

- **Действия:**

    1. Система обнаруживает превышение квоты (HTTP 429 Too Many Requests)

    2. Система блокирует дальнейшие запросы от данного пользователя на 60 секунд

    3. Система отображает сообщение:

        - «⚠️ Слишком много запросов. Пожалуйста, подождите 1 минуту перед продолжением работы.»

        - Обратный отсчёт: «Доступ восстановится через: 00:59»

    1. Система логирует событие с уровнем WARNING:

        - Тип: RATE_LIMIT_EXCEEDED

        - User ID

        - Request count: [N]

        - Timestamp

    1. Система автоматически разблокирует доступ через 60 секунд

    2. Пользователь может продолжить работу после разблокировки

    3. Сценарий возобновляется после истечения блокировки

### E6: Повреждённые данные в БД (NULL в обязательных полях)

- **Точка возникновения:** Шаг 8

- **Условие:** Запрос вернул записи с NULL в критически важных полях (например, `appeal_number`, `status`)

- **Действия:**

    1. Система обрабатывает результаты запроса

    2. Система обнаруживает записи с NULL в обязательных полях

    3. Система пропускает повреждённые записи (не отображает в таблице)

    4. Система логирует инцидент с уровнем CRITICAL:

        - Тип: DATA_INTEGRITY_VIOLATION

        - Appeal IDs: [список ID повреждённых записей]

        - Missing fields: [список полей]

        - Timestamp

    1. Система отправляет алерт команде поддержки БД

    2. Система отображает предупреждающий баннер в верхней части реестра:

        - «⚠️ Обнаружены повреждённые данные. Некоторые обращения могут не отображаться. Техническая поддержка уведомлена.»

    1. Система продолжает работу с корректными записями

    2. Основной поток продолжается с частично отображённым реестром

### E7: Истечение сессии пользователя

- **Точка возникновения:** Шаг 3 или любой последующий запрос к API

- **Условие:** JWT-токен пользователя истёк или был отозван (401 Unauthorized)

- **Действия:**

    1. Система получает ответ 401 Unauthorized при попытке запроса

    2. Система сохраняет текущее состояние реестра (фильтры, поиск, пагинация) в `sessionStorage`:

        - Ключ: `unsaved_registry_state`

    1. Система очищает устаревший токен из localStorage

    2. Система перенаправляет пользователя на страницу авторизации

    3. Система передаёт URL возврата: `/login?redirect=/appeals`

    4. После успешной авторизации:

        - Система перенаправляет пользователя обратно к реестру (`/appeals`)

        - Загружает сохранённое состояние из `sessionStorage`

        - Восстанавливает фильтры, поиск, пагинацию

        - Выполняет запрос с восстановленными параметрами

    1. Сценарий возобновляется с восстановленным состоянием

### E8: Ошибка сохранения состояния в sessionStorage

- **Точка возникновения:** Шаг 51 (сохранение состояния перед переходом к карточке)

- **Условие:** Браузер блокирует доступ к `sessionStorage` (режим инкогнито с ограничениями, квота превышена)

- **Действия:**

    1. Система пытается записать данные в `sessionStorage`

    2. Система перехватывает ошибку (QuotaExceededError или SecurityError)

    3. Система логирует предупреждение с уровнем WARNING:

        - Тип: SESSIONSTORAGE_WRITE_FAILED

        - Error: [текст ошибки]

    1. Система НЕ блокирует переход к карточке обращения

    2. Система использует fallback: сохраняет минимальное состояние в памяти приложения (state management)

    3. При возврате к реестру состояние НЕ восстанавливается (пользователь видит реестр с параметрами по умолчанию)

    4. Система отображает информационное сообщение:

        - «ℹ️ Не удалось сохранить фильтры. При возврате к реестру параметры будут сброшены.»

    1. Основной поток продолжается с предупреждением

## 7. Специальные требования

### 7.1. Производительность

- **NFR-PERF-01:** Время загрузки реестра (50 записей) не должно превышать 1.5 секунд при нормальной нагрузке на БД

- **NFR-PERF-02:** Время отклика поиска (после debounce) не должно превышать 500 миллисекунд

- **NFR-PERF-03:** Время применения фильтра не должно превышать 500 миллисекунд

- **NFR-PERF-04:** Время сортировки таблицы не должно превышать 200 миллисекунд

- **NFR-PERF-05:** Debounce поиска установлен на 300 миллисекунд для снижения количества запросов к БД

- **NFR-PERF-06:** Таймаут запроса к БД: 10 секунд (нормальный), 30 секунд (extended для больших выборок)

- **NFR-PERF-07:** Система должна поддерживать одновременный просмотр реестра 200+ пользователями без деградации производительности

- **NFR-PERF-08:** Пагинация: ленивая подгрузка при скролле к концу страницы (опционально, вместо кнопок пагинации)

- **NFR-PERF-09:** Индексы БД должны быть созданы на колонках: `responsible_user_id`, `department_id`, `status`, `sla_deadline`, `created_at`, `appeal_number`, `applicant_full_name`

### 7.2. Безопасность

- **NFR-SEC-01:** Все запросы к API должны выполняться с JWT-токеном в заголовке `Authorization: Bearer <token>`

- **NFR-SEC-02:** Применение row-level security (RLS) на уровне базы данных:

    - Ответственный сотрудник: `WHERE responsible_user_id = current_user_id OR department_id = user.department_id` (read-only для подразделения)

    - Супервизор: `WHERE department_id = user.department_id` (read-write)

    - Администратор: без ограничений

- **NFR-SEC-03:** Логирование всех действий пользователя в реестре:

    - Просмотр реестра (session start)

    - Применение фильтров (filter_applied)

    - Выполнение поиска (search_performed)

    - Переход к карточке обращения (appeal_accessed)

    - Данные логов: user_id, action, timestamp, query_params, ip_address, session_id

- **NFR-SEC-04:** Маскирование персональных данных в таблице:

    - Телефон: показывать последние 4 цифры (****1234) для ответственного сотрудника

    - Полный телефон доступен только супервизору или в карточке обращения

    - Адрес: не отображается в таблице, доступен только в карточке обращения

- **NFR-SEC-05:** Защита от SQL injection:

    - Все параметры поиска и фильтров передаются через подготовленные запросы (prepared statements)

    - Валидация формата `appeal_id` (UUID или числовой ID)

    - Экранирование специальных символов в поисковых запросах

- **NFR-SEC-06:** Rate limiting:

    - Максимум 100 запросов в минуту на пользователя

    - Блокировка на 60 секунд при превышении

    - HTTP 429 Too Many Requests

- **NFR-SEC-07:** Проверка прав доступа перед каждым запросом на уровне API (backend), а не только на уровне UI

### 7.3. Доступность (Accessibility)

- **NFR-AVAIL-01:** Интерфейс реестра должен соответствовать стандарту WCAG 2.1 уровня AA

- **NFR-AVAIL-02:** Навигация с клавиатуры:

    - Tab: переход между элементами управления (поиск, фильтры, строки таблицы, пагинация)

    - Enter: применение фильтра, переход к обращению, отправка поискового запроса

    - Escape: очистка поиска, сброс фильтров

    - Arrow keys: навигация по строкам таблицы (↑/↓), переключение между табами фильтров (←/→)

- **NFR-AVAIL-03:** ARIA-метки для всех интерактивных элементов:

    - `aria-label` для кнопок без текста (иконки)

    - `aria-live="polite"` для динамически обновляемых счётчиков

    - `aria-expanded` для табов фильтров

    - `role="table"`, `role="row"`, `role="columnheader"`, `role="cell"` для таблицы

    - `aria-sort` для заголовков сортируемых колонок (`ascending` / `descending` / `none`)

- **NFR-AVAIL-04:** Контраст текста должен быть не менее 4.5:1 для обычного текста и 3:1 для крупного текста

- **NFR-AVAIL-05:** Фокус-индикаторы должны быть видимы для всех интерактивных элементов (outline, border, box-shadow)

- **NFR-AVAIL-06:** Поддержка screen readers:

    - Озвучивание изменений при применении фильтров («Применён фильтр: Мои обращения»)

    - Озвучивание количества найденных записей («Найдено 15 обращений»)

    - Озвучивание направления сортировки («Сортировка по дедлайну, по возрастанию»)

### 7.4. Удобство использования (Usability)

- **NFR-UX-01:** Sticky header таблицы: заголовки колонок остаются видимыми при вертикальной прокрутке (position: sticky, top: 0)

- **NFR-UX-02:** Анимация применения фильтров и поиска: fade-in 150 миллисекунд для плавного обновления таблицы

- **NFR-UX-03:** Сохранение состояния реестра в `sessionStorage`:

## 8. Бизнес-правила

### BR-LIST-01: Фильтр по умолчанию

**Правило:** При первом входе в реестр обращений всегда активирован фильтр «Мои обращения».

**Логика:**

- SQL: `WHERE responsible_user_id = current_user_id`

- Отображаются только обращения, где пользователь является назначенным ответственным

**Обоснование:** Предотвращение информационной перегрузки, фокусировка внимания на личных задачах сотрудника.

**Исключения:** Если сохранено состояние фильтров в `sessionStorage` от предыдущей сессии, восстанавливаются сохранённые фильтры.

---

### BR-LIST-02: Цветовая индикация дедлайнов

**Правило:** Поле «Дедлайн» в таблице отображается с цветовой индикацией в зависимости от процента использованного времени SLA.

**Формула:** `sla_usage_percent = ((current_time - sla_started_at) / (sla_deadline - sla_started_at)) * 100`

**Цвета:**

- **Зелёный:** `sla_usage_percent < 50%` (до дедлайна осталось >50% времени)

- **Жёлтый:** `50% ≤ sla_usage_percent < 80%` (осталось 20-50% времени)

- **Красный:** `sla_usage_percent ≥ 80%` (осталось <20% времени или просрочено)

- **Красный + жирный + иконка ⚠:** `current_time > sla_deadline` (дедлайн прошёл)

**Обоснование:** Быстрая визуальная оценка срочности обращений для приоритизации работы.

**Исключения:** Нет.

---

### BR-LIST-03: Сортировка по умолчанию

**Правило:** По умолчанию таблица отсортирована по полю «Дедлайн» в порядке возрастания (ASC) — ближайшие дедлайны первыми.

**SQL:** `ORDER BY sla_deadline ASC`

**Обоснование:** Помогает сотруднику сфокусироваться на обращениях с наиболее близкими или просроченными дедлайнами.

**Исключения:** Если пользователь изменил сортировку и состояние сохранено в `sessionStorage`, восстанавливается последняя выбранная сортировка.

---

### BR-LIST-04: Счётчики в табах фильтров

**Правило:** Счётчики в табах фильтров обновляются в реальном времени при изменении данных в БД или применении фильтров.

**Формат:** «Мои обращения ([N])», где [N] — количество обращений, соответствующих фильтру.

**Расчёт:**

- Выполняется отдельный COUNT-запрос для каждого таба

- Счётчики загружаются параллельно с основными данными таблицы

- При изменении фильтров пересчитываются только затронутые счётчики

**Обоснование:** Помогает пользователю оценить объём данных перед применением фильтра.

**Исключения:** Если счётчик = 0, таб остаётся доступным, показывает «(0)».

---

### BR-LIST-05: Комбинирование фильтров

**Правило:** Фильтры из разных групп комбинируются по логике AND, фильтры внутри одной группы — взаимоисключающие (radio button logic).

**Пример:**

- Выбраны: «Мои обращения» + «Физ. лицо» + «Просрочено»

- SQL: `WHERE (responsible_user_id = current_user_id) AND (client_type = 'individual') AND (sla_deadline < current_time)`

**Группы фильтров:**

1. Ответственность: «Все обращения» OR «Мои обращения» (взаимоисключающие)

2. Тип клиента: «Все типы» OR «Физ. лицо» OR «Юр. лицо» (взаимоисключающие)

3. Статус: «Все статусы» OR «В обработке» OR «Ожидает доведения» OR «Просрочено» (взаимоисключающие)

**Обоснование:** Логичное сочетание фильтров для точной выборки данных.

**Исключения:** Нет.

---

### BR-LIST-06: Поиск по множественным полям

**Правило:** Поисковый запрос ищет совпадения в следующих полях:

- `appeal_number` — номер обращения (точное или частичное совпадение)

- `applicant_full_name` — ФИО заявителя (частичное совпадение)

- `appeal_content` — первые 1000 символов содержания обращения (полнотекстовый поиск)

**Логика:** Логическое OR (совпадение хотя бы в одном поле).

**SQL (PostgreSQL):**

```SQL
WHERE (
  appeal_number ILIKE '%query%'
  OR applicant_full_name ILIKE '%query%'
  OR appeal_content ILIKE '%query%'
)
```

**Регистронезависимость:** Поиск выполняется без учёта регистра (ILIKE).

**Ограничения:**

- Минимальная длина запроса: 2 символа

- Максимальная длина запроса: 200 символов

- Специальные символы экранируются

**Обоснование:** Универсальный поиск по ключевым полям обращения.

**Исключения:** Полнотекстовый поиск (Full-Text Search) может быть включён опционально для повышения производительности при больших объёмах данных.

---

### BR-LIST-07: Видимость данных по ролям

**Правило:** Набор доступных обращений зависит от роли пользователя.

**Роли и видимость:**

|Роль|Фильтр «Мои обращения»|Фильтр «Все обращения»|Права редактирования|
|-|-|-|-|
|Ответственный сотрудник|`WHERE responsible_user_id = current_user_id`|`WHERE department_id = user.department_id` (read-only)|Только свои обращения|
|Супервизор ДУКО|`WHERE responsible_user_id = current_user_id`|`WHERE department_id = user.department_id` (read-write)|Все обращения подразделения|
|Администратор|`WHERE responsible_user_id = current_user_id`|Все обращения без ограничений|Все обращения|

**Обоснование:** Разграничение доступа согласно должностным обязанностям и принципу минимальных привилегий.

**Исключения:** Нет.

---

### BR-LIST-08: Пагинация и производительность

**Правило:** Максимальный размер выборки на одной странице: 100 записей.

**Рекомендованные значения:** 10, 25, 50 (по умолчанию), 100.

**Логика:**

- При выборке >100 записей система автоматически включает пагинацию

- Пользователь может изменить лимит через dropdown «Показывать по: [10/25/50/100]»

- При изменении лимита пагинация сбрасывается на первую страницу

**SQL:** `LIMIT [limit] OFFSET [offset]`

**Расчёт OFFSET:** `offset = (page - 1) * limit`

**Обоснование:** Баланс между удобством просмотра и производительностью запросов к БД.

**Исключения:** Супервизор может запросить экспорт всех данных в CSV/Excel без ограничения пагинации (отдельная функция).

---

### BR-LIST-09: Поддержка частичного совпадения и полнотекстового поиска

**Правило:** Поиск поддерживает частичное совпадение (LIKE) по умолчанию, с возможностью переключения на полнотекстовый поиск (Full-Text Search) при наличии соответствующих индексов в БД.

**Типы поиска:**

- **Частичное совпадение (LIKE/ILIKE):** `ILIKE '%query%'` — ищет подстроку в любой части текста

- **Полнотекстовый поиск (FTS):** использует индексы PostgreSQL (GIN/GiST) для быстрого поиска по словам

**Переключение:** Автоматически определяется системой в зависимости от наличия FTS-индексов.

**Обоснование:** Гибкость поиска с оптимизацией производительности.

**Исключения:** Нет.

---

### BR-LIST-10: Сохранение фильтров между сеансами

**Правило:** Фильтры, сортировка, пагинация и поисковый запрос сохраняются в `sessionStorage` и восстанавливаются при:

- Возврате из карточки обращения

- Обновлении страницы (F5)

- Возврате из других разделов системы

**Срок хранения:** До закрытия вкладки браузера (session storage) или до явной очистки пользователем.

**Данные сохранения:**

```JSON
{
  "filters": { ... },
  "sort": { ... },
  "pagination": { ... },
  "search": "текст поиска",
  "scroll_position": 1234
}
```

**Обоснование:** Удобство работы, предотвращение потери контекста при переходах.

**Исключения:** Если пользователь очистил `sessionStorage` или работает в режиме инкогнито с ограничениями, состояние не сохраняется.

---

### BR-LIST-11: Перезагрузка реестра при смене роли

**Правило:** Если роль пользователя изменяется во время сеанса (например, повышение до супервизора), реестр автоматически перезагружается с новыми правами доступа.

**Логика:**

- Система отслеживает изменения в JWT-токене (через WebSocket или polling)

- При обнаружении изменения роли отображается уведомление: «Ваши права изменены. Обновляем данные...»

- Автоматическая перезагрузка реестра с новыми фильтрами и правами

**Обоснование:** Актуальность отображаемых данных при изменении прав доступа.

**Исключения:** Нет.

---

### BR-LIST-12: Автоматическое включение пагинации

**Правило:** Пагинация включается автоматически, если количество записей в выборке превышает 100.

**Логика:**

- Если `COUNT(*) > 100`: отображается пагинация с кнопками «Назад», «Вперёд», выбором страниц

- Если `COUNT(*) ≤ 100`: пагинация скрыта, все записи отображаются на одной странице

**Обоснование:** Упрощение интерфейса при малом количестве записей, обеспечение производительности при большом.

**Исключения:** Нет.

---

### BR-LIST-13: Учёт часового пояса при сортировке по дедлайну

**Правило:** Сортировка по полю «Дедлайн» всегда учитывает часовой пояс пользователя.

**Логика:**

- Даты в БД хранятся в UTC

- При отображении и сортировке конвертируются в локальный часовой пояс пользователя (из профиля или автоопределение)

- Сортировка выполняется по конвертированным значениям

**Обоснование:** Корректное отображение времени для пользователей в разных часовых поясах.

**Исключения:** Нет.

---

### BR-LIST-14: Обрезка поля «Содержание (краткое)»

**Правило:** В колонке «Содержание (краткое)» отображаются только первые 80 символов текста обращения, остальное обрезается с добавлением «...».

**Отображение:** `"Клиент обратился с жалобой на некорректное начисление комиссии за SMS-информир..."`

**Tooltip:** При наведении курсора отображается полный текст содержания (первые 500 символов).

**Обоснование:** Экономия места в таблице, сохранение читаемости.

**Исключения:** Полный текст доступен в карточке обращения.

---

### BR-LIST-15: Доступность кнопки «Создать обращение»

**Правило:** Кнопка «➕ СОЗДАТЬ ОБРАЩЕНИЕ» доступна всегда, независимо от количества обращений в реестре (даже при 0 записей).

**Логика:**

- Кнопка отображается в toolbar реестра

- Активна для всех пользователей с правом `appeal.create`

- Переход к UC-00 «Регистрация нового обращения»

**Обоснование:** Обеспечение возможности создания обращений в любой момент.

**Исключения:** Нет.

## 10. Связь с другими Use Cases

### Предшествующие Use Cases:

- Нет прямых предшествующих UC (реестр — точка входа в систему обработки обращений)

### Последующие Use Cases:

- **UC-VIEW-01:** Просмотр детализации обращения

    - *Связь:* Пользователь кликает на номер обращения в реестре и переходит к детальному просмотру

- **UC-ACCEPT-01:** Принять обращение в работу ответственным

    - *Связь:* После просмотра в реестре или карточке пользователь может принять обращение в работу

    - *Связь:* Пользователь может отказаться от обращения, назначенного ему

    - *Связь:* Пользователь кликает на кнопку «➕ СОЗДАТЬ ОБРАЩЕНИЕ» в реестре

### Параллельные Use Cases:

- **UC-DECIDE-01:** Принять решение, создать и довести ответ клиенту

    - *Связь:* После просмотра обращения в реестре пользователь может перейти к обработке

- **UC-QUERY-01:** Запросить информацию в смежном подразделении

    - *Связь:* Может быть инициировано из карточки обращения, открытой из реестра

- **UC-COMMENT-01:** Добавление комментария к обращению

### 11.4. Примечания для разработчиков

- Реализация должна использовать серверную пагинацию (не клиентскую) для работы с большими объёмами данных

- Все запросы к БД должны использовать подготовленные выражения (prepared statements) для защиты от SQL injection

- Рекомендуется использование Redis или Memcached для кэширования счётчиков фильтров

- Индексы БД критически важны для производительности поиска и сортировки

- Состояние фильтров в `sessionStorage` должно иметь версионирование для совместимости при обновлении интерфейса

- WebSocket-уведомления об изменениях обращений опциональны, но рекомендуются для real-time обновлений

- Skeleton screen предпочтительнее spinner'а для индикации загрузки (лучше UX)

- Для mobile responsive версии может потребоваться адаптация таблицы (cards вместо table)

### 11.5. Метрики успеха (KPI)

- **Среднее время поиска обращения:** < 10 секунд (от входа в реестр до открытия карточки)

- **Процент использования фильтров:** > 60% пользователей используют фильтры регулярно

- **Процент использования поиска:** > 40% пользователей используют поиск

- **Удовлетворённость пользователей:** > 4.5/5 (по результатам опросов)

- **Время загрузки реестра:** медианное значение < 1 секунда, 95-й перцентиль < 2 секунды
