# ADR-004: Учебный демо-бэкенд (REST + одна СУБД) для итогового проекта

**Status:** Accepted | **Date:** 2026-05-03 | **Version:** 1.0.0

---

## Контекст

- По [ADR-001](ADR-001-frontend-spa.md) основной контур UI — SPA с **mock** и `localStorage`.
- Для **финального проекта** по курсу **«Системный анализ»** (школа) требуется **демонстрация полного стека**: фронтенд + **сервер** + **хотя бы одна база данных**, без претензии на промышленный запуск.
- В импорте ограничений зафиксирован ориентир стека: **Java / Spring Boot 2.7+**, **PostgreSQL 15.2+** ([`docs/incoming-artifacts/constraints-import-b0b7abff/`](../incoming-artifacts/constraints-import-b0b7abff/README.md)).

## Решение

1. Вводится **отдельный учебный контур**: один **REST API** (JSON) и **одна** реляционная СУБД (**PostgreSQL**), разворачиваемая локально или в Docker.
2. Реализация бэкенда **вне** обязательного объёма репозитория фронтенда: отдельный модуль/репозиторий или подпапка `backend/` — по выбору команды; в данном репозитории фиксируется **план и контракт**, а не обязательный код.
3. Фронтенд переключается с mock на API через **конфигурацию** (например переменная окружения базового URL API), без изменения экранов сверх требований FR.
4. Объём API — **минимально достаточный для демонстрации** сценариев из плана ([`docs/backend-development-plan.md`](../backend-development-plan.md)): CRUD по обращению, справочник статусов, заглушка пользователя/роли при отсутствии полноценного IAM.

## Связь с ADR-001

- **ADR-001 не отменяется:** по умолчанию проект остаётся работоспособным **без** бэкенда.
- ADR-004 задаёт **дополнительный** режим для защиты учебного проекта и репетиции интеграции.

## Последствия

- Нужно поддерживать согласованность модели данных с [`docs/er-diagram.md`](../er-diagram.md) на уровне **ключевых сущностей** (хотя бы **Обращение** и связанные справочники).
- Безопасность и интеграции с CRM/МОКИ — **вне** минимального учебного контура (заглушки допустимы).
