# Роли команды и ожидания по опыту — EDO Bank

**Version:** 1.0.0 | **Date:** 2026-05-03 | **Status:** Active  

Документ задаёт **кто за что отвечает** в проекте модуля ЭДО для ДУКО и **какой опыт** считается достаточным. Технические правила агентов Cursor по-прежнему в `.cursor/rules/*.mdc` и [`AGENTS.md`](../AGENTS.md).

**Контекст заказа:** [`core-source-context.md`](core-source-context.md) (KPI сроков, устные обращения).

---

## Матрица ролей: зона, область, опыт

### 1. Аналитик требований (FR / BR)

| Параметр | Содержание |
|----------|------------|
| **Зона** | Формулировка и согласование **функциональных** и **бизнес**-требований; приоритеты P0–P2; связка UC ↔ FR; версии документов. |
| **Предметная область** | Обработка обращений в финансовой организации: жизненный цикл обращения, роли (ответственный, аудитор, администратор), SLA и регламенты, CRM-контуры **на уровне требований** (не интеграционная разработка). |
| **Ожидаемый опыт** | **Middle+** системный аналитик: 3+ года в продуктовой или заказной разработке; опыт **BR/FR**, user stories, acceptance criteria; желательно опыт **банка / страхования / крупного контакт-центра** или систем класса CRM/BPM/Service Desk. Понимание **регуляторного давления по срокам** (как в `core-source-context.md`). |
| **Артефакты** | `functional-requirements.md`, `business-requirements.md`, `use-case.md`, `state-diagram.md`, `glossary.md`, реестры в `docs/`. |
| **Не входит** | Вёрстка и код; выбор npm-пакетов без согласования с FR/ADR. |

---

### 2. Специалист по UI/UX и дизайн-системе

| Параметр | Содержание |
|----------|------------|
| **Зона** | Принципы интерфейса, **дизайн-токены**, согласованность экранов кабинетов; бриф и план дизайн-системы **в рамках утверждённого ТЗ** — без расширения функций. |
| **Предметная область** | Корпоративные **web-кабинеты** (плотные таблицы, карточки, фильтры), тёмная тема, доступность базового уровня (подписи, фокус, контраст по правилам проекта). |
| **Ожидаемый опыт** | **Middle** UX/UI в B2B или банковских интерфейсах **2+ года**; работа с **дизайн-системами** (аналог shadcn/Material — понимание компонентного контура); умение описывать поведение в markdown-спеках, а не только в Figma. |
| **Артефакты** | `ui-ux-brief.md`, `design-system-plan.md`; опора на `docs/ui-artifacts/` как на исторические заметки. |
| **Не входит** | Реализация React-компонентов (это фронтенд); изменение FR без процесса согласования. |

---

### 3. Разработчик фронтенда (React / TypeScript)

| Параметр | Содержание |
|----------|------------|
| **Зона** | Реализация UI и логики по **FR**: компоненты, сервисы, mock-данные, состояния загрузки/ошибка/пусто там, где это в ТЗ. |
| **Предметная область** | SPA на **Vite**, **React 18**, **TypeScript**, **Tailwind**, **shadcn/ui**, Radix, MUI — как в ADR; работа с **localStorage** через сервисный слой. |
| **Ожидаемый опыт** | **Middle** фронтенд: **2+ года** React+TS; опыт **корпоративных админок** или финтеха желателен; строгая типизация, умение читать спецификацию и не добавлять фичи «от себя». |
| **Артефакты** | Код в `src/`; при изменении стека — `artifacts-catalog.md`, ADR в `docs/adr/`. |
| **Не входит** | Бэкенд Spring/Java; правка FR без аналитика; подключение новых библиотек без ADR. |

---

### 4. QA / ревью по ТЗ

| Параметр | Содержание |
|----------|------------|
| **Зона** | **Статическая проверка** соответствия реализации документам; отчёт о расхождениях; критичность дефектов с отсылкой к разделам FR. |
| **Предметная область** | Тест-анализ по текстовым требованиям; сценарии из UC и state diagram; граничные состояния UI. |
| **Ожидаемый опыт** | **Middle** QA или аналитик с сильной QA-частью: **2+ года**; опыт **чек-листов и приёмочных критериев** в Agile; понимание SPA; желательно опыт **регулируемой** среды (сроки, аудит следов). Навык автоматизации E2E **не обязателен** для текущего контура (mock, SPA). |
| **Артефакты** | Запросы в чате с явной ссылкой на FR; при необходимости — записи в задаче/PR. |
| **Не входит** | Изменение ТЗ; написание прод-кода исправлений (можно описать баг — исправляет фронтенд). |

---

### 5. Интеграция зависимостей (npm)

| Параметр | Содержание |
|----------|------------|
| **Зона** | `package.json` / lockfile; обновления зависимостей **в рамках согласованного стека**; отсутствие лишних пакетов. |
| **Предметная область** | Экосистема **Node/npm**, уязвимости и совместимость версий с **Vite 5/6** и **React 18**. |
| **Ожидаемый опыт** | **Junior+** с опытом сопровождения фронтенд-репозитория **1+ год** или **Middle** фронтенд с фокусом на инфраструктуру. |
| **Артефакты** | Те же; при новом пакете — запись в каталог артефактов и ADR по правилам проекта. |
| **Не входит** | Реализация бизнес-фич без задачи; произвольное добавление UI-библиотек. |

---

### 6. Инженер сборки (Vite / TypeScript / PostCSS)

| Параметр | Содержание |
|----------|------------|
| **Зона** | `vite.config.*`, `tsconfig.json`, `postcss.config.*`, пути и алиасы, сборка и превью. |
| **Предметная область** | **Vite**, **TypeScript** strict, **PostCSS** / Tailwind pipeline. |
| **Ожидаемый опыт** | **Junior+** **1+ год** со сборкой фронтенд-приложений или **Middle** фронтенд; понимание различий dev/prod и влияния конфига на бандл. |
| **Не входит** | Прод-DevOps (K8s, CI вне репозитория) — только если отдельно введены в объём. |

## Минимальный состав для потока «ТЗ → код → проверка»

| Роль | Нужна для MVP? |
|------|----------------|
| Аналитик | Да (можно совмещение с PM заказчика, если владеет FR) |
| UI/UX | Желательно; допускается совмещение с фронтендом при соблюдении `design-system-plan` |
| Фронтенд | Да |
| QA по ТЗ | Да, хотя бы частично (ревью перед релизом демо) |
| npm / Сборка | Обычно совмещается с фронтендом |

## Обновление матрицы

При смене стека (ADR), появлении бэкенд-команды или CI — дополнить этот файл и поднять **PATCH** версии; зафиксировать в `docs/VERSIONS.md`.
