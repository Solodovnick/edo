# Роли команды и должности — EDO Bank

**Version:** 1.1.1 | **Date:** 2026-05-03 | **Status:** Active  

Документ задаёт **официальные названия должностей**, зоны ответственности и соответствие правилам Cursor.  

**Правило по грейду:** для всех должностей ниже установлен минимум **Senior** (не ниже уровня старшего специалиста в соответствующей области).

**Контекст заказа:** [`core-source-context.md`](core-source-context.md).

---

## 1. Системный аналитик

| Параметр | Содержание |
|----------|------------|
| **Должность** | Системный аналитик (**Senior**). |
| **Зона** | Функциональные и бизнес-требования (FR/BR), связка UC ↔ FR, приоритеты P0–P2, жизненный цикл обращения в документации, версии артефактов. |
| **Предметная область** | Обработка обращений в банке; SLA и регламенты; роли пользователей кабинетов; CRM на уровне требований. |
| **Опыт** | **Senior**: как правило **5+ лет** системного / продуктового анализа; опыт BR/FR, acceptance criteria; желательно **банк, финтех, CRM/BPM, Service Desk**; понимание регуляторных сроков (`core-source-context.md`). |
| **Артефакты** | `functional-requirements.md`, `business-requirements.md`, `use-case.md`, `state-diagram.md`, `glossary.md`, реестры в `docs/`. |
| **Не входит** | Реализация кода и вёрстки; выбор npm-библиотек без согласования с FR/ADR. |

## 2. Архитектор ПО

| Параметр | Содержание |
|----------|------------|
| **Должность** | Архитектор программного обеспечения / архитектор решения (**Senior**). |
| **Зона** | Границы систем и контейнеров, согласованность с ландшафтом банка; **ADR**; верхний уровень **C4**; принципы интеграций (mock vs реальный API по ADR). |
| **Предметная область** | Фронтенд-SPA в контуре МОО; сопоставление с импортами (`incoming-artifacts/`), IcePanel/C4-черновики. |
| **Опыт** | **Senior**: **5+ лет** в проектировании информационных систем; опыт **корпоративной архитектуры**, документирования решений; понимание ограничений React-SPA и mock-слоя. |
| **Артефакты** | `docs/c4-architecture-overview.md`, `docs/adr/`, при необходимости обновление `artifacts-catalog.md`. **Учебный демо-контур** (курс «Системный анализ»): [ADR-004](adr/ADR-004-education-demo-backend.md), [план бэкенда](backend-development-plan.md). |
| **Не входит** | Детальная проработка полей экранов (это системный аналитик + FR); непосредственная верстка. |

## 3. Senior UX/UI-дизайнер

| Параметр | Содержание |
|----------|------------|
| **Должность** | Senior UX/UI-дизайнер (кабинеты, дизайн-система). |
| **Зона** | Интерфейсные принципы, токены, согласованность экранов; `ui-ux-brief`, `design-system-plan` **в рамках утверждённого ТЗ**. |
| **Предметная область** | Корпоративные web-кабинеты, плотные таблицы, карточки, фильтры; тёмная тема; базовая доступность (подписи, фокус, контраст по правилам проекта). |
| **Опыт** | **Senior**: **5+ лет** UX/UI в B2B или банковских интерфейсах; зрелые **дизайн-системы**; спецификации в Markdown не хуже Figma. |
| **Артефакты** | `ui-ux-brief.md`, `design-system-plan.md`; опора на `docs/ui-artifacts/` как на исторические заметки. |
| **Не входит** | Реализация React-компонентов; изменение FR без согласования. |

## 4. Старший frontend-разработчик

| Параметр | Содержание |
|----------|------------|
| **Должность** | Старший frontend-разработчик (React, TypeScript). |
| **Зона** | Реализация UI и логики по **FR**: компоненты, сервисы, mock-данные, состояния загрузки / ошибка / пусто по ТЗ. |
| **Предметная область** | SPA **Vite**, **React 18**, **TypeScript**, **Tailwind**, **shadcn/ui**, Radix, MUI — по ADR; **localStorage** через сервисный слой. |
| **Опыт** | **Senior**: **5+ лет** коммерческого React и TypeScript; корпоративные админки / финтех; строгая типизация; без расширения функций сверх FR. |
| **Артефакты** | Код в `src/`; изменение стека только через `artifacts-catalog.md` и ADR. |
| **Не входит** | Бэкенд Spring/Java; правка ТЗ без системного аналитика; новые библиотеки без ADR. |

## 5. Старший QA-инженер

| Параметр | Содержание |
|----------|------------|
| **Должность** | Старший QA-инженер (контроль по ТЗ и UC). |
| **Зона** | Статическая сверка реализации с документами; отчёт о расхождениях; приоритет дефектов с отсылкой к FR. |
| **Предметная область** | Тест-анализ по текстовым требованиям; UC и state diagram; граничные состояния SPA. |
| **Опыт** | **Senior**: **5+ лет** QA в продукте; приёмочные критерии, регрессионная дисциплина; понимание React-SPA; желателен опыт **регулируемой** среды. Автотесты E2E для текущего mock-контура не обязательны. |
| **Артефакты** | Запросы с явной ссылкой на FR; задачи / PR. |
| **Не входит** | Изменение ТЗ; правка продуктового кода (описание дефекта — для frontend). |

| Параметр | Содержание |
|----------|------------|
| **Должность** | Старший инженер платформы фронтенда (зависимости, сборка, конфигурация). |

| **Не входит** | Разработка бизнес-фич без задачи; добавление UI-библиотек без процесса. |

## 7. Старший технический писатель

| Параметр | Содержание |
|----------|------------|
| **Должность** | Старший технический писатель. |
| **Зона** | Целостность документации репозитория: структура `docs/`, **VERSIONS**, **artifacts-catalog**, связность README, входящие импорты и каталоги внешних источников; редактура формулировок без изменения смысла без согласования. |
| **Предметная область** | Техническая документация на русском; работа с Markdown и перекрёстными ссылками; единообразие терминов с `glossary.md`. |
| **Опыт** | **Senior**: **5+ лет** техписьма в ИТ; опыт регулируемых или крупных организаций приветствуется. |
| **Артефакты** | `docs/VERSIONS.md`, `docs/artifacts-catalog.md`, индексные `README.md`, при необходимости — правки оглавлений в `incoming-artifacts/`. |
| **Не входит** | Утверждение бизнес-целей и приоритетов (стейкхолдер); установка функциональных требований (системный аналитик). |

## 8. Стейкхолдер — руководитель отдела

| Параметр | Содержание |
|----------|------------|
| **Должность** | Стейкхолдер: **руководитель отдела** (заказчик по процессу: например, линия ДУКО / смежное подразделение — уточняется приказом). |
| **Зона** | Приоритеты и **acceptance** на уровне «что должна дать система бизнесу»; контроль KPI (в т.ч. **99%** своевременности); эскалация разногласий между ролями. |
| **Предметная область** | Организация работы с обращениями; распределение ответственности между подразделениями банка. |
| **Опыт** | Руководящая позиция в операционном или клиентском блоке **Senior**-уровня по ответственности (не ИТ-грейд). |
| **Не входит** | Написание кода и детальных FR (делегирует системному аналитику); не заменяет архитектора по техническим решениям. |

## Минимальный состав потока «ТЗ → код → проверка»

Учитывая минимум **Senior** по ключевым должностям:

| Должность | Роль в потоке |
|-----------|----------------|
| Стейкхолдер | Утверждает цели и приёмку по KPI |
| Системный аналитик | Ведёт FR/BR и трассировку UC |
| Архитектор | Фиксирует архитектурные решения и ADR при изменениях контура |
| Senior UX/UI | Держит UI-бриф и дизайн-систему |
| Старший frontend-разработчик | Реализует SPA |
| Старший QA-инженер | Ревью по ТЗ перед демо/релизом |
| Старший инженер платформы фронтенда | По мере необходимости — сборка и зависимости (часто совмещается с senior frontend) |
| Старший технический писатель | По мере необходимости — порядок в репозитории документов |
