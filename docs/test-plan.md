**Version:** 1.0.0 | **Date:** 2026-05-04 | **Status:** Active

## 1. Цели и объём тестирования

### Цели
- Проверить соответствие REST API функциональным требованиям FR-01..FR-05, FR-17
- Проверить жизненный цикл обращения (FR-02, state-diagram v1.2.0)
- Проверить корректность обработки граничных случаев и ошибок
- Подтвердить работоспособность фронтенда (ручная проверка)

### Что тестируется
| Объект | Тип тестирования |
|---|---|
| `POST /api/appeals` | Функциональное, негативное |
| `GET /api/appeals` | Функциональное, фильтрация, поиск |
| `GET /api/appeals/{id}` | Функциональное, негативное |
| `PATCH /api/appeals/{id}` | Жизненный цикл, негативное |
| Фронтенд (кабинеты) | Ручное smoke-тестирование |

### Что не тестируется
- Безопасность / аутентификация (не реализована, FR-02.2 — учебный контур)
- Внешние интеграции МОКИ / DaData (FR-16.1, FR-16.2 — вне MVP)
- Нагрузочное тестирование

1. Отправить `POST /api/appeals` с телом:
```json
{
  "appealType": "Письменное",
  "applicantCategory": "Физ лицо",
  "applicantName": "Тестов Иван Петрович",
  "phone": "+7 (999) 000-00-01",
```

- Тело ответа содержит `id` (UUID), `number` (автогенерация), `status` = `"Зарегистрировано"`

### TS-02: Жизненный цикл / статусные переходы (FR-02, state-diagram)

---

#### TC-02.1 — Happy Path: полный цикл Зарегистрировано → Закрыто

| Поле | Значение |
|---|---|
| **Тип** | Позитивный / Happy Path |
| **FR** | FR-02.1, FR-02.2 |

**Шаги:**
1. Создать обращение (TC-01.1), сохранить `id`
2. `PATCH /api/appeals/{id}` → `{"status": "Назначено"}`
3. `PATCH /api/appeals/{id}` → `{"status": "На ответственном, взято"}`
4. `PATCH /api/appeals/{id}` → `{"status": "Решено"}`
5. `PATCH /api/appeals/{id}` → `{"status": "Закрыто"}`

**Ожидаемый результат:** Каждый шаг — HTTP 200, `status` обновляется последовательно

---

#### TC-02.2 — Alternative Path: с ответвлением через Запрос в БП

| Поле | Значение |
|---|---|
| **Тип** | Позитивный / Alternative Path |
| **FR** | FR-02.1, FR-12.3 |

**Шаги:**
1. Создать обращение, перевести в `"На ответственном, взято"`
2. `PATCH` → `{"status": "Запрос в БП"}`
3. `PATCH` → `{"status": "На ответственном, взято"}` (возврат из БП)
4. `PATCH` → `{"status": "Решено"}`

**Ожидаемый результат:** Все переходы проходят с HTTP 200

---

#### TC-02.3 — Alternative Path: с ответвлением через Аудит

| Поле | Значение |
|---|---|
| **Тип** | Позитивный / Alternative Path |
| **FR** | FR-02.1, FR-13.3 |

**Шаги:**
1. Перевести обращение в `"Решено"`
2. `PATCH` → `{"status": "Аудит"}`
3. `PATCH` → `{"status": "На аудите"}`
4. `PATCH` → `{"status": "Пройден аудит"}`
5. `PATCH` → `{"status": "Закрыто"}`

**Ожидаемый результат:** Все переходы HTTP 200, финальный `status` = `"Закрыто"`

---

#### TC-02.4 — Негативный: недопустимый переход статуса

| Поле | Значение |
|---|---|
| **Тип** | Негативный |
| **FR** | FR-02.1, FR-02.2 |

**Шаги:**
1. Взять обращение в статусе `"Зарегистрировано"`
2. `PATCH` → `{"status": "Закрыто"}` (прямой переход запрещён)

**Ожидаемый результат:**
- HTTP 409 Conflict или 422 Unprocessable Entity
- Сообщение о недопустимом переходе

---

#### TC-02.5 — Негативный: переход из финального статуса

| Поле | Значение |
|---|---|
| **Тип** | Негативный |
| **FR** | FR-02.1 |

**Шаги:**
1. Обращение в статусе `"Отказано"`
2. `PATCH` → `{"status": "Зарегистрировано"}`

**Ожидаемый результат:** HTTP 4xx, переход запрещён
