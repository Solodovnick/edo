# План разработки учебного демо-бэкенда (REST + PostgreSQL)

**Version:** 1.0.0 | **Date:** 2026-05-03 | **Status:** Active  

**Назначение:** итоговый проект по курсу **«Системный анализ»** — показать связку **фронтенд → API → БД** для демонстрации в школе.  
**Архитектурное решение:** [ADR-004: Учебный демо-бэкенд](adr/ADR-004-education-demo-backend.md).  
**Ограничения стека (импорт):** [ограничения b0b7abff](incoming-artifacts/constraints-import-b0b7abff/README.md) — Spring Boot 2.7+, PostgreSQL 15.2+, Java (Liberica JDK).

---

## 1. Цели демонстрации

| Цель | Как показать |
|------|----------------|
| Единая модель обращения | Таблица(ы) в PostgreSQL, согласованные с [ER-диаграммой](er-diagram.md) (минимум сущность *Обращение*) |
| Сохранение между сессиями | Данные в БД, не в `localStorage` |
| Контракт с UI | REST JSON, CORS с dev-ориджина фронта |
| Разумный объём для курса | Один сервис, одна БД, без микросервисов |

---

## 2. Состав контура (минимум)

1. **СУБД:** PostgreSQL 15.2+ (одна база, одна схема `public` или `edo_demo`).
2. **Приложение:** Spring Boot 2.7+ (или 3.x при согласовании с преподавателем и обновлении этой спецификации) — REST.
3. **API (первый инкремент):**
   - `GET /api/appeals` — список обращений (пагинация опциональна);
   - `GET /api/appeals/{id}` — карточка;
   - `POST /api/appeals` — создание (упрощённая форма);
   - `PATCH /api/appeals/{id}` — обновление статуса / ключевых полей по [state-diagram](state-diagram.md) (допустим сокращённый набор переходов).
4. **Справочники:** `GET /api/statuses` (или вложенно в ответ) — для согласования с жизненным циклом в ТЗ.

Идентификаторы: UUID или bigint — зафиксировать в OpenAPI/контракте при появлении репозитория бэкенда.

## 3. Этапы работ (ориентир 4–6 недель)

| Этап | Содержание | Артефакт / результат |
|------|------------|----------------------|
| **E1** | Установка JDK, PostgreSQL, создание БД, миграции (Flyway/Liquibase или SQL-скрипты) | Пустая схема + таблицы |
| **E2** | Сущность обращения + статусы; начальные данные (seed) для демо | Заполненные тестовые обращения |
| **E3** | REST-слой, валидация, обработка ошибок (400/404/500) | Postman/HTTP клиент |
| **E4** | CORS, профиль `application-demo.yml`, инструкция запуска | Локальный запуск API |
| **E5** | Подключение фронта: переменная `VITE_API_BASE_URL` (или аналог), замена вызовов в `src/services/` на HTTP с fallback на mock по флагу | Демо «список → карточка» из браузера |
| **E6** | Сценарий защиты: диаграмма последовательности (список запросов), скрин БД, скрин UI | Презентационный пакет |

---

## 4. Связь с фронтендом

- Слой **`src/services/`** остаётся единой точкой доступа к данным; учебная ветка добавляет **HTTP-клиент** (fetch/axios) при включённом флаге окружения.
- Полное покрытие FR не требуется: достаточно сценариев **FR-04** (список/карточка) и части **FR-02/FR-03** для защиты.

---

## 5. Вне объёма учебного минимума

- Прод-авторизация (OIDC/LDAP), интеграция CRM, МОКИ, DaData — по [FR-16](functional-requirements.md) только после отдельных ADR.
- Kubernetes, очереди, репликация БД — не требуются для зачёта демо.
