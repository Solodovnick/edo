# Ядро контекста проекта (core source) — ЭДО для ДУКО

**Version:** 1.0.0 | **Date:** 2026-05-03 | **Status:** Active  

Этот файл — **первичное прочтение** для агентов и участников: зачем делается продукт, какие боли и численные цели. Он **не** заменяет функциональное ТЗ: экраны, поля и поведение по-прежнему только из `docs/functional-requirements.md` и связанных утверждённых артефактов.

---

## Организационный контур

- **Заказчик / зона ответственности:** департамент **ДУКО** (управление качеством обслуживания — формулировка рабочая).
- **Продукт:** **собственный модуль ЭДО** (электронного документооборота / учёта обращений) банка — не внешний SaaS, а внутренняя разработка под процессы ДУКО.
- **Задача в одном предложении:** навести порядок в **обработке обращений** клиентов и регулятора.

---

## Главная «боль» и регулятор

- Регулятор (**ЦБ**) штрафует за **нарушение сроков** ответов по обращениям («Змей-Регулятор» — рабочее прозвище процесса давления со стороны нормативных требований).
- Именно **сроки и прослеживаемость** — доминирующий драйвер приоритетов, не только «удобство экрана».

---

## Показатели своевременности

| Показатель | Значение |
|------------|----------|
| Сейчас (факт) | В срок обрабатывают **91%** обращений |
| Требование | **99%** своевременных ответов |
| Цель после стабилизации | Через **3 месяца после запуска** системы выйти на **99%** своевременных ответов |

Любая функция учёта статусов, очередей, эскалаций и отчётности должна **поддерживать** этот KPI, если это явно не противоречит утверждённому ТЗ.

---

## Где теряется качество сегодня

- **Устные обращения** (телефон, офис) страдают больше всего: **теряются**, уходят **без должного контроля** и учёта по сравнению с письменным потоком.
- Следствие для продукта: приоритет **регистрации, статусов и контроля SLA** по **устному** каналу не ниже, чем по письменному (в рамках FR).
