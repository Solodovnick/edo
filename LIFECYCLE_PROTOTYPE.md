# 🏦 ЭДО Банк - Интерактивный прототип жизненного цикла обращений

## 🎯 Обзор

Полнофункциональный интерактивный desktop-прототип (1440px) системы электронного документооборота банка с пошаговыми инструкциями "что нажимать" на каждом этапе обработки обращений.

## 🚀 Быстрый старт

1. На главном Dashboard нажмите на плавающую кнопку **"Интерактивный прототип"** (правый нижний угол)
2. Откроется полноэкранный прототип с 5 вкладками жизненного цикла
3. Переключайтесь между вкладками для изучения всех этапов

## 📋 Структура прототипа

### 5 экранов жизненного цикла:

#### 1️⃣ **Обработать обращение** (Processing)
**Что делать:**
- ✅ "Взять в работу" - назначить себя исполнителем
- ✅ Ответить на 4 gateway-вопроса (оценка обращения)
- ✅ Выполнить действия по выбранному маршруту:
  - Связаться с заявителем (если недостаточно информации)
  - Создать запрос в бизнес-подразделение (если не может обработать сам)
  - Передать в финансовый департамент (если юр/фин характер)
  - Создать инцидент HelpDesk (если технический характер)
- ✅ "Принять решение" (выбор решения + обоснование)
- ✅ "Подготовить ответ" (текст ответного письма)
- ✅ "Отправить на подпись" → симуляция подписи → "Отправить в административный отдел"
- ✅ "Перейти к доведению решения"

**Ветки:**
- Недостаточно информации → Запрос данных → Ожидание → Продолжение
- Не может сам → Запрос в подразделение → Ожидание ответа
- Юр/фин → Передача в комиссию → Ожидание решения
- Технический → HelpDesk инцидент → Ссылка на инцидент

#### 2️⃣ **Довести решение** (Delivery)
**Что делать:**
- ✅ Выбрать способ доведения (Звонок/Письмо/Сообщение)
- ✅ "Зафиксировать попытку" → Результат: Уведомлён (Да/Нет)
- ✅ Если НЕТ: указать причину → Запланировать следующую попытку (до 3 попыток)
- ✅ Если ДА: "Зафиксировать дату доведения"
- ✅ "Передать в аудит"

**Ветки:**
- Успешно уведомлён с 1 попытки → Фиксация → Аудит
- Не уведомлён → Попытка 2/3 → Планирование следующей
- Исчерпаны 3 попытки → Эскалация руководителю / Закрытие без подтверждения

#### 3️⃣ **Аудит** (Audit)
**Что делать:**
- ✅ "Взять в аудит" (доступно через 2 дня после доведения)
- ✅ "Связаться с заявителем"
- ✅ Указать отзыв: Положительный/Отрицательный/Нет ответа
- ✅ Если ОТРИЦАТЕЛЬНЫЙ:
  - "Передать информацию об отрицательном отзыве"
  - "Проанализировать причины"
  - Нужно новое решение? (Да → "Создать новое обращение" / Нет)
- ✅ "Провести аудит качества" (чек-лист из 4 пунктов)
- ✅ "Обновить данные в CRM"
- ✅ "Передать в архивацию"

**Ветки:**
- Положительный отзыв → Аудит → CRM → Архив
- Отрицательный → Передача руководству → Анализ → Новое обращение (опционально) → Аудит → Архив
- Нет ответа → Аудит → CRM → Архив

#### 4️⃣ **Архивация** (Archive)
**Что делать:**
- ✅ Выбрать основание (Решение доведено / 31 декабря)
- ✅ "Перевести статус в Архив" (с подтверждением: необратимо!)
- ✅ "Извлечь метаданные" (просмотр таблицы метаданных)
- ✅ Выбрать хранение: "Есть DWH?" (Да/Нет)
  - Да → "Сохранить в DWH"
  - Нет → "Оставить данные в CRM"
- ✅ "Завершить архивацию (закрыть обращение)"
- ✅ "Перейти к просмотру закрытого обращения"

**Мастер из 3 шагов:**
1. Архивация (триггер + подтверждение)
2. Метаданные (извлечение)
3. Хранение (DWH или CRM)

#### 5️⃣ **Закрыто** (Closed) - Read-only
**Что можно:**
- 📄 Просмотр полной карточки обращения
- 📊 Ключевые даты (создано/решение/доведено/закрыто)
- ✅ SLA статус
- 📑 Сводка всех этапов
- 📥 "Скачать карточку обращения (PDF)"
- 📥 "Скачать протокол (CSV)"
- ➕ "Создать новое обращение на основе этого"

## 🎨 Дизайн-система

### Цветовая палитра банка:
- **Primary Blue:** `#0051BA` - основные действия, активные элементы
- **Success Green:** `#00AA44` - успешные операции, подтверждения
- **Warning Yellow:** `#FFAA00` - предупреждения, ожидания
- **Critical Red:** `#DD0000` - критичные ситуации, ошибки

### UI компоненты:
- **ComplaintHeader** - Шапка с номером, статусом, метаданными
- **TimelineProtocol** - Протокол событий с фильтрацией и поиском
- **ComplaintData** - Сайдбар с данными обращения
- **NextAction** - Блок "Следующее действие" с валидацией

### Состояния кнопок:
- ✅ **Default** - белый фон, серая обводка
- 🔵 **Primary** - синий фон #0051BA
- ⚪ **Secondary** - белый фон, серая обводка потолще
- ❌ **Disabled** - серый фон, серый текст, курсор not-allowed
- ✓ **Success** - зелёный фон или бордер

## 🧩 Ключевые возможности

### ✅ Сквозные элементы на всех вкладках:
1. **Видимый статус** - цветной бейдж в шапке
2. **Текущий шаг** - активная вкладка
3. **Следующее действие** - блок с валидацией и CTA
4. **Протокол событий** - timeline с фильтрами
5. **Данные обращения** - appealId, категория, SLA, контакты

### 🎯 Понятная навигация:
- Крупные кнопки с иконками и понятными названиями
- Decision-точки в виде radio/toggle с подсказками "что произойдёт дальше"
- Disabled состояния с текстом "Станет доступно после..."
- Валидационные сообщения со списком недостающих действий

### 🔄 Интерактивность:
- Все действия логируются в протокол в реальном времени
- Условная логика: показ/скрытие блоков по ответам gateway
- Симуляция асинхронных операций (ожидание подписи, ответа подразделения)
- Confirm-диалоги для необратимых действий (архивация)

### 📊 Протокол событий:
- Автоматическое добавление записей при каждом действии
- Фильтрация по этапам (Регистрация/Обработка/Доведение/Аудит/Архив)
- Поиск по тексту событий
- Экспорт протокола

## 🎓 Инструкции для пользователя

### Как пользоваться прототипом:

1. **Начните с "Обработать обращение"**
   - Нажмите "Взять в работу"
   - Ответьте на вопросы оценки
   - Следуйте подсказкам в блоке "Следующее действие"

2. **Изучите ветки процесса**
   - Попробуйте разные варианты ответов на gateway-вопросы
   - Посмотрите, как меняется UI в зависимости от выбора

3. **Проверьте валидацию**
   - Попробуйте нажать "Перейти к доведению" без заполнения полей
   - Прочитайте список требований в блоке валидации

4. **Следите за протоколом**
   - Каждое действие добавляет событие в timeline
   - Используйте фильтры для навигации по событиям

5. **Пройдите весь цикл**
   - Последовательно переключайтесь между вкладками
   - На странице "Закрыто" увидите итоговую сводку

## 💡 Best Practices

### При расширении функционала:
1. Добавляйте события в протокол при каждом значимом действии
2. Используйте валидацию NextAction для предотвращения пропуска шагов
3. Показывайте понятные подсказки "что произойдёт" рядом с decision-точками
4. Сохраняйте единый стиль кнопок и карточек
5. Тестируйте все ветки процесса

## 🐛 Known Limitations

- Данные не сохраняются между перезагрузками (state только в памяти)
- Нет реальной интеграции с backend
- Симуляция асинхронных операций (подпись, ответы подразделений)
- Фиксированные mock-данные (AP-10234)
